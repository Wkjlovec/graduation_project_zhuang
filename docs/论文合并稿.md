## 摘要

高校师生在课程答疑、学习资料共享和校园信息传播中对在线交流平台有持续需求。现有校园论坛普遍采用单体架构，扩展性不足、前后端耦合、缺乏移动端适配，制约了系统的持续发展。
针对上述问题，本文设计并实现了一个基于微服务架构的高校论坛系统。后端采用 Spring Boot 3 与 Spring Cloud Alibaba 构建，通过 Nacos 实现服务注册与配置管理，经由 Spring Cloud Gateway 统一路由与鉴权，将系统拆分为认证、论坛、通知、搜索和媒体五个独立服务。MySQL 负责数据持久化，Redis 承担缓存与会话管理，用户认证基于 JWT 结合 Spring Security 实现登录、令牌刷新与登出的闭环。前端基于 Vue 3，PC 端采用 Element Plus、移动端采用 Vant，实现帖子发布与浏览、分区讨论、二级评论、点赞和通知管理等功能。系统配套 Docker Compose 基础设施编排和 Python 自动化测试脚本。功能测试覆盖 11 个端到端场景和全部 22 个接口，通过率 100%；并发负载测试在 10 至 100 并发用户下验证了系统的承载能力。
**关键词：** 微服务架构；Spring Boot；Vue 3；JWT；Redis；高校论坛

## Abstract

University teachers and students have a continuous demand for online communication platforms in scenarios such as course Q&A, study material sharing, and campus information dissemination. Existing campus forums generally adopt monolithic architectures with limited scalability, front-back coupling, and lack of mobile adaptation.
To address these issues, this thesis designs and implements a microservice-based forum system. The backend is built with Spring Boot 3 and Spring Cloud Alibaba, using Nacos for service registration and configuration management, and Spring Cloud Gateway for unified routing and authentication. The system is decomposed into five independently deployable services: authentication, forum, notification, search, and media. MySQL handles data persistence and Redis manages caching and sessions. User authentication is based on JWT with Spring Security, covering login, token refresh, and logout. The frontend is built with Vue 3, using Element Plus for PC and Vant for mobile, implementing post publishing, section-based browsing, two-level comments, likes, and notification management. The system includes Docker Compose orchestration and Python automated testing scripts. Functional tests cover 11 end-to-end scenarios and all 22 API endpoints with a 100% pass rate; concurrent load tests under 10 to 100 users verify the system's capacity.
**Keywords:** microservice architecture; Spring Boot; Vue 3; JWT; Redis; university forum

第一章 绪论

1.1 研究背景与意义

互联网的普及深刻改变了人们获取信息和交流沟通的方式。高等院校中，教学通知发布、课程问题讨论、学习经验分享等活动已逐步从线下面对面交流和纸质公告转向线上平台。即时通讯工具传递消息的速度很快，对话内容却随时间迅速沉没——一条课程答疑的截图在群聊中可能几小时后就被新消息覆盖，几天之后更难以检索。论坛系统的价值恰在于此：它将碎片化的问答与讨论按主题沉淀为连续的、可回查的知识内容，课程答疑、学习资料共享、经验复盘和校园通知发布都更适合以论坛帖子的形式长期留存和复用[1]。
国内高校论坛系统的建设可以追溯到上世纪末的校园 BBS 时代。那一批系统以纯文本界面为主，功能集中在发帖和回帖，多采用单体架构部署。用户规模有限时，单体架构尚可运转；在线人数增长、交互需求变得丰富之后，并发处理能力不足、功能扩展困难、版本迭代缓慢等问题开始暴露。水木清华 BBS 早在 2005 年便限制了校外访问，2012 年正式关站；北大未名 BBS 至今仍在运行，注册范围已收窄至 70 余所高校联盟成员。这两个标志性的高校论坛积累了大量用户，底层技术建设却多年未做大规模升级，界面风格与当下的审美习惯差距明显，源码冗余度高，日常维护的成本也在持续增加[2]。
当前许多高校论坛系统在工程实践中依然面临不少短板。部分系统只完成了发帖和回帖的基本流程，用户权限管理与内容审核机制不够完善，社区氛围难以保证[3]。与此同时，多数校园论坛仅提供 PC 端页面，没有针对移动端做专门适配，而根据 CNNIC 第 57 次统计报告，截至 2025 年 12 月我国网民使用手机上网的比例已达 99.6%[4]，相当一部分用户因此无法获得良好的使用体验。从系统内部结构来看，不少论坛仍将前端页面与后端逻辑耦合在同一工程中，任何局部修改都需要全局重新部署，开发效率和可维护性因此受到制约。这种耦合还进一步影响了质量保障工作：功能验证往往依赖手工操作，回归测试和性能测试无法自动重跑，交付质量缺少可量化的判定依据。
面向上述问题，设计和实现一个功能完整、架构合理、可持续维护的高校论坛系统，具有明确的工程价值。本文围绕架构解耦、跨平台适配、安全认证和质量保障四个层面展开工作，将微服务拆分后的独立部署能力、PC 端与移动端的一致使用体验、基于令牌的认证闭环，以及可复现的自动化测试整合到同一套工程方案中。只有将需求分析、架构设计、功能实现与测试验证纳入同一工程闭环，论坛系统才能真正成为高校教学交流中可长期依赖的基础设施。

1.2 研究现状

Web 技术的迭代推动了论坛系统从最初的纯文本 BBS 向现代化应用的持续演变。早期校园论坛以 JSP 或 PHP 搭建，业务逻辑与页面渲染耦合在单体工程中，局部修改往往牵动全局的重新编译与部署。用户规模增长之后，并发瓶颈、扩展困难和迭代缓慢等问题随之浮现。前后端分离架构的普及改变了这一局面：后端退居为 RESTful API 数据层，前端以独立应用承担界面渲染，两端各自迭代、互不牵制。国内近年已有多项校园论坛采用 Spring Boot 与 Vue 的分离方案，功能完整性和开发效率均有明显改善[5]。
微服务架构将单体应用拆解为多个可独立部署的服务单元，服务之间经由轻量级协议通信，业务模块得以解耦并按需伸缩。电商、政务、工业等领域的大规模系统已广泛采纳这一架构。国内近期实践表明，Spring Cloud 体系在服务注册与发现、API 网关路由、流量控制与熔断降级等环节已形成较为完整的技术栈，辅以 Docker 容器化部署和 Kubernetes 编排，可实现服务的标准化交付与弹性伸缩[6]。高校信息化建设同样受益于此：有学者将 Spring Cloud 微服务架构引入协同人员管理平台，借助服务拆分和消息队列实现了业务解耦与数据准实时同步[8]；科研管理领域亦出现了覆盖需求调研到成果转化全流程的微服务工程案例[9]。Agaev 则从单体向微服务迁移的策略角度切入，指出迁移的成功取决于方法组合与数据、交互和运维流程的分阶段重构[20]。
面向高并发场景，已有论坛系统引入 Redis 缓存与消息队列，并结合 Nginx 负载均衡和容器化部署来提升大规模访问下的稳定性[21]。质量保障同样受到关注——有学者指出，合理的测试策略与持续集成流程能显著降低线上故障率并加快需求变更的响应[10]。安全层面的研究也在推进：Aldea 等人系统梳理了微服务环境下认证机制面临的漏洞与防护策略，强调 JWT 令牌的短生命周期设计和最小权限原则是缩小攻击面的关键手段[7]。总体而言，论坛系统的技术选型已从单体框架演进到前后端分离与微服务架构相结合的阶段，但在高校场景中，将微服务架构完整应用于论坛系统并配套可量化、可复现的测试与运维体系的工程实践仍然较少，这正是本文的研究切入点。

1.3 主要研究内容

1.1 节梳理了现有校园论坛在架构扩展性、双端适配、安全管控和质量验证等方面的不足，1.2 节进一步指出高校场景下将微服务架构完整应用于论坛系统的工程实践仍然较少。基于这两方面的分析，本文设计并实现一个基于微服务架构的高校论坛系统。与多数现有工作停留在单体框架或仅完成前后端分离不同，本文在同一工程中完成了从微服务拆分到双端前端适配、从认证闭环到自动化测试的完整链路。具体研究内容包括以下六项。
（1）后端采用 Spring Boot 3 与 Spring Cloud Alibaba，将系统拆分为网关、认证、论坛、通知、搜索和媒体六个独立服务。Nacos 承担服务注册与配置管理，Spring Cloud Gateway 统一路由与鉴权，各服务可独立开发、部署和扩缩。
（2）认证与会话管理基于 JWT 与 Spring Security 设计无状态方案，配合 Redis 存储会话状态，覆盖登录、令牌刷新与登出的完整流程，网关层统一完成令牌校验与用户身份透传。
（3）论坛核心业务的数据建模与功能实现。MySQL 中设计帖子、评论、分区、点赞等实体及其关联关系，Redis 缓存热点数据以降低数据库查询压力，系统支持帖子发布、分区浏览、二级评论、点赞去重等操作，并配套种子数据初始化机制。
（4）前端基于 Vue 3，PC 端采用 Element Plus、移动端采用 Vant 分别构建。两端共享统一的 API 调用层和状态管理逻辑，界面交互各自适配对应终端的使用习惯，网关 CORS 配置保障跨域访问。
（5）通知、搜索和媒体三个辅助服务分别提供系统公告与用户消息管理、基于关键词的帖子过滤检索，以及论坛首页的补充内容展示，用于验证微服务架构下多服务协同工作的可行性。
（6）运维与测试体系包括 Docker Compose 基础设施编排、一键启停脚本、Python 全链路回归测试和缓存基准测试脚本。回归测试覆盖从注册、登录到发帖、评论、搜索的完整业务流程，以及权限校验、并发去重等异常场景，执行后自动生成测试报告；缓存基准测试对比冷启动与热缓存下的接口响应时间，为缓存策略的有效性提供可量化的数据。

第二章 相关技术及方法

2.1 后端技术

系统的后端采用 Java 17 语言开发。Java 通过 JVM 虚拟机使编译产物能够在不同操作系统上一致运行，类型系统成熟、垃圾回收机制完善，长期以来是大型 Web 应用后端开发的主流选择。JDK 目前有 8、11、17 和 21 四个长期支持版本，本文选用 JDK 17，相较于 JDK 8 增加了密封类、模式匹配等语言特性，社区支持和依赖库兼容性也已趋于稳定。
Spring 框架的两项核心机制——控制反转（IoC）和面向切面编程（AOP）——奠定了 Java 企业级开发的基础。控制反转由容器接管对象的创建和依赖关系，调用方从容器获取所需对象而非自行实例化，模块之间的耦合度由此降低。面向切面编程将日志记录、权限校验等横切关注点抽取为独立模块统一维护，避免同一段逻辑在多处重复。Spring Boot 在此基础上进一步简化了项目搭建：自动装配机制根据引入的依赖完成配置，内嵌 Tomcat 容器使应用无须外部 Web 服务器即可独立运行[11]。本文全部后端服务均基于 Spring Boot 3 构建。
微服务架构的核心思路是将一个进程内的业务逻辑拆分为多个独立服务，每个服务拥有自己的数据存储和部署周期，服务之间通过网络接口通信[12]。拆分带来了独立扩缩和版本迭代的能力，代价则是需要额外的基础设施处理服务寻址、配置同步和流量管控。通信方式的选择对性能有直接影响：同步调用在低延迟场景中响应更快，异步消息在高负载下的可扩展性更优[18]。灵活性提升的同时容错问题也随之出现，断路器和故障转移机制是防止单个服务故障蔓延为全局瘫痪的关键手段[17]。API 网关则作为系统面向外部的唯一入口，集中承担请求路由、协议适配和安全控制[16]。本文使用的 Spring Cloud Alibaba 将上述治理能力整合为一套组件：Nacos 同时扮演服务注册中心和配置中心，各服务启动时向 Nacos 注册自身地址，调用方按服务名寻址，实例的增减对上游透明；Spring Cloud Gateway 基于路由规则分发请求并在过滤器链中嵌入令牌校验逻辑；OpenFeign 将跨服务的 HTTP 调用封装为接口声明，降低远程通信的编码成本。
用户认证采用 JWT（JSON Web Token）方案。JWT 将用户身份声明和过期时间编码为经数字签名的令牌字符串，服务端只需校验签名即可确认令牌真伪，不必在服务端集中保存会话状态，天然适配多实例部署的分布式环境。Spring Security 通过过滤器链对每次请求拦截校验，与网关层的令牌验证形成两道屏障。JWT 一经签发便无法主动撤销，本文额外使用 Redis 存储会话状态来支持登出和令牌失效。Aldea 等人在对微服务认证安全的研究中指出，基于令牌的无状态认证在多服务环境中工程优势明显，短生命周期令牌和最小权限原则能有效缩小攻击面[7]。
数据持久化选用 MySQL 8。关系型数据库在标准 SQL 查询、事务处理和索引优化上具有成熟支撑，适合存储用户账户、帖子内容、评论等结构明确且关联关系清晰的业务数据[15]。微服务架构的理想模型要求每个服务管理自身专属的数据域，以避免服务间产生隐式耦合[13]。系统当前的业务规模尚不需要为每个服务单独建库，认证服务、论坛服务和通知服务共享同一个 MySQL 实例，各服务通过不同数据表实现逻辑隔离，搜索服务和媒体服务不直接访问数据库。对象与数据表之间的映射通过 JPA 规范完成，底层由 Hibernate 引擎处理实体的生命周期管理和级联操作。缓存层采用 Redis 7，数据驻留在内存中，读写延迟通常在亚毫秒级别。帖子列表和详情等高频访问数据在 Redis 中设置了带过期时间的缓存，命中时直接返回结果，不再穿透到 MySQL；认证会话和登录失败计数等临时状态同样存储在 Redis 中，利用其原子操作特性保证并发场景下数据的一致性。
MySQL、Redis 和 Nacos 三个基础设施组件通过 Docker 容器化封装。Docker Compose 编排文件集中定义了各容器的镜像版本、端口映射、数据卷和启动依赖关系，一条命令即可在全新的环境中拉起全部服务，开发者本地环境差异带来的配置不一致问题由此消除[14]。容器化与微服务的结合也在安全中台等领域得到验证，通过分布式服务治理和容器编排实现业务模块的解耦与弹性部署[19]。运维脚本以 Bash 编写，按照 common-lib → auth → forum → notification → media → search → gateway 的顺序依次启动各服务并逐一探测健康端点，停止时按反向顺序有序关闭，脚本还支持在有 Nacos 的完整模式和无 Nacos 的直连模式之间自动切换。测试脚本以 Python 编写，负责全链路回归测试和缓存基准测试，执行结束后自动生成 Markdown 格式的结果报告。全部脚本遵循幂等原则，确保在不同时间点执行能够产生可比较的结果。

2.2 前端技术

Web 前端的三项基础技术各司其职：HTML 定义页面结构，CSS 描述样式与布局，JavaScript 为页面注入动态交互能力，使浏览器端能够响应用户操作并与后端通信。早期的前端开发直接操作浏览器 DOM 节点，代码量增长后维护难度迅速上升。现代前端框架以组件化和数据驱动为核心理念，将页面拆分为可复用的独立单元，开发者只需关注数据状态的变化，框架自动完成 DOM 的差量更新。
Vue、React 和 Angular 是当前使用最广泛的三个前端框架。Angular 内置依赖注入、模板引擎和路由系统，功能齐全，适合大型团队的企业级项目，代价是 TypeScript 语言门槛和庞大的概念体系带来的学习曲线。React 和 Vue 走的是更轻量的路线，都通过虚拟 DOM 减少不必要的页面重绘。两者在数据响应机制上有明显差异：React 要求开发者显式调用 setState 或 Hook 函数触发视图更新，数据何时变化、视图何时刷新完全由代码控制；Vue 采用依赖追踪，数据一旦被修改，框架自动识别受影响的视图区域并触发重新渲染，开发者不必手动介入。Vue 的双向数据绑定在表单场景中尤为方便，输入框的值与数据模型实时同步，省去了大量的事件监听代码。Vue 的中文生态丰富、上手门槛较低，官方配套了 Vite 构建工具、Vue Router 路由库和 Pinia 状态管理库，工具链开箱即用，本文据此选用 Vue 3 作为前端框架[5]。
PC 端的 UI 组件库选用 Element Plus，它基于 Vue 3 开发，提供表格、表单、对话框、分页器等桌面端常用控件。移动端选用 Vant，该库针对触屏操作和小屏幕布局做了专门优化，导航栏、单元格、浮动气泡等交互模式均已封装就绪。两端的 UI 层各自独立，底层的数据通信和状态管理则是共享的：Axios 封装的 HTTP 请求层统一处理与后端的数据交互，Pinia 集中管理用户认证等全局状态，Vue Router 负责页面路由控制和未登录访问拦截。打包构建使用 Vite，它利用浏览器原生的 ES 模块机制按需编译，开发环境下的模块热替换速度比传统 Webpack 方案快出一个数量级，调试反馈周期大幅缩短。

第三章 系统需求分析

3.1 系统概述

高校的日常学习生活中，学生之间、师生之间的信息交流从未间断。学生课后向同学请教课堂上没听懂的知识点，在群组中传递实验环境的配置步骤，把整理好的期末复习资料分享给其他人，将社团活动的通知及时送达每一位参与者——这些场景每天都在发生。如果仅依赖即时通讯群组，信息很快就会被新消息淹没，几天之后便难以检索和回溯。论坛系统能够弥补这一不足：它将碎片化的问答和讨论按主题组织成持续沉淀的内容，任何人在任何时间都可以回头查阅，一次性的沟通由此转变为可长期复用的知识资源。
本文设计的微服务论坛系统面向上述高校场景，为在校师生提供帖子发布与浏览、分区讨论、评论互动、信息检索和通知触达等在线交流功能。未登录的游客可以浏览帖子列表、公告摘要、搜索结果和媒体推荐等公开内容，完成注册并登录后则进一步解锁发帖、评论、点赞和查看个人通知等操作。系统没有区分学生、教师等角色，所有注册用户享有相同的操作权限，帖子和评论的编辑删除权限仅取决于当前用户是否为内容作者。这一简化的权限模型放弃了基于角色的访问控制（RBAC）所提供的细粒度灵活性，换来的是更低的系统复杂度，对于高校论坛这类用户角色差异不大的场景是合理的取舍。分区和公告等基础数据在系统启动时通过种子数据初始化机制完成配置。
系统采用前后端分离的 B/S 架构，用户通过浏览器访问 PC 端或移动端页面，所有请求经由统一网关路由至后端各微服务。后端按业务职责拆分为认证服务、论坛服务、通知服务、搜索服务和媒体服务五个独立模块，各模块通过 Nacos 注册中心完成服务发现，以 RESTful 接口相互通信，MySQL 负责数据持久化，Redis 承担热点数据缓存和会话状态存储。用户注册并登录后获取访问令牌，后续请求携带该令牌即可使用论坛的全部功能。网关层根据请求路径和方法判断是否需要认证：GET 类的公开浏览接口允许匿名访问，发帖、评论、点赞等写操作接口则要求携带有效令牌。系统总体架构如图 3-1 所示。

3.2 功能性需求分析

（1）用户认证
论坛中每一条帖子和评论都应当能够追溯到真实的发布者，用户的账户也不能被他人冒用，用户认证模块承担的就是这一职责。游客访问注册接口提交用户名、昵称和密码后成为注册用户。注册用户通过登录接口提交用户名和密码，系统验证凭据后签发访问令牌和刷新令牌，二者均以 JWT 格式返回。访问令牌用于后续请求的身份校验，有效期较短；刷新令牌用于在访问令牌过期后获取新的令牌对，有效期较长。用户主动登出时，系统在 Redis 中删除对应的会话记录，使已签发的令牌立即失效。登录操作包含（include）凭据校验环节，校验失败时扩展（extend）触发失败计数逻辑，连续失败达到阈值后临时锁定账户。用户认证模块的用例如图 3-2 所示。

（8）网关与访问控制
所有来自 PC 端和移动端的 HTTP 请求都先到达网关，再由网关根据预设的路由规则转发至对应的后端服务，网关因此成为系统面向外部的唯一入口。网关同时维护了一份公开接口白名单，包括认证相关接口、帖子列表和详情的 GET 请求、分区列表、通知首页摘要、媒体首页和推荐内容以及搜索接口，这些接口允许匿名访问。白名单之外的所有接口均为受保护接口，要求请求头中携带有效的访问令牌。网关在转发受保护请求前依次执行两项校验：首先验证 JWT 令牌的数字签名和声明字段，确认令牌未被篡改且类型正确；随后查询 Redis 中的会话记录，确认该会话未被主动注销。两项校验均通过后，网关将用户 ID 和用户名写入请求头透传至下游服务，使下游服务无须重复解析令牌。网关同时配置了跨域资源共享规则，允许 PC 端（端口 5173）和移动端（端口 5174）从各自的开发地址访问后端接口。网关与访问控制模块的用例如图 3-8 所示。

3.3 非功能性需求分析

（1）性能需求
帖子列表和帖子详情是用户访问最频繁的两个页面。每次请求都穿透到数据库查询的话，用户并发增长时数据库压力会迅速上升，页面加载速度随之下降。为此，系统对帖子列表设置了 2 分钟过期的 Redis 缓存，对帖子详情设置了 5 分钟过期的缓存，其余缓存项统一采用 10 分钟的默认过期时间。缓存命中时接口应在毫秒级完成响应，缓存未命中时响应时间应控制在 500 毫秒以内。系统配套了缓存基准测试脚本，能够在冷启动和热缓存两种条件下分别采集接口响应时间并计算改善幅度，作为性能达标与否的量化判据。
（2）安全需求
用户会话管理采用 JWT 令牌方案取代传统的 Cookie 机制，跨站请求伪造攻击劫持会话的风险由此降低。访问令牌有效期为 86400 秒（24 小时），刷新令牌有效期为 604800 秒（7 天），二者均通过不低于 32 字符的密钥以 HMAC-SHA 算法签名，防止令牌被伪造或篡改。为防范暴力破解，同一账户连续登录失败 5 次后将被锁定 600 秒（10 分钟），失败计数和锁定状态存储在 Redis 中，利用原子操作保证并发条件下计数的准确性。网关在转发受保护请求前依次校验令牌签名和 Redis 会话状态，双重验证通过后方将请求放行至下游服务。网关的跨域资源共享配置仅允许 PC 端（localhost:5173）和移动端（localhost:5174）两个已注册来源发起请求，其余来源将被拒绝。
（3）可扩展性需求
业务规模增长和新功能模块的接入是系统上线后必须面对的常态。后端五个服务连同网关各自运行在独立进程中，通过 Nacos 注册中心完成服务发现，新增业务模块只需开发独立的 Spring Boot 应用并注册至 Nacos，再由网关配置一条新的路由规则即可接入，无须修改已有服务的任何代码。各服务的数据库地址、Redis 地址、Nacos 地址、JWT 密钥和监听端口等参数全部通过环境变量注入，在开发、测试和部署三个阶段可以使用不同的配置值而不改动源码。运维脚本还支持在有 Nacos 的完整模式和无 Nacos 的直连模式之间自动切换，使系统在网络条件受限的环境中也能正常运行。
（4）可维护性需求
系统上线之后，运营成本和故障定位效率在很大程度上取决于可维护性的高低。MySQL 8.4、Redis 7.4 和 Nacos 2.3.2 三个基础设施组件通过 Docker Compose 编排文件统一管理，镜像版本、端口映射、数据卷和健康检查规则在文件中集中声明，一条命令即可在全新环境中拉起全部依赖服务，避免了手工安装配置带来的人为差错。后端服务的 Bash 启动脚本按照 common-lib → auth → forum → notification → media → search → gateway 的顺序依次启动各服务并逐一探测健康端点，停止时按照反向顺序有序关闭，确保网关最先关闭以阻止新请求进入。Python 回归测试脚本覆盖从注册、登录、发帖、评论、搜索到通知的完整业务链路，以及权限校验、并发点赞去重和令牌过期拦截等异常路径，执行结束后自动输出 Markdown 格式的测试报告，方便快速判断本次变更是否引入了回归问题。
（5）兼容性需求
不同设备上的用户界面应保持一致的风格和清晰的布局，让用户切换终端时不需要重新适应操作逻辑。PC 端基于 Element Plus 组件库构建，适配 Chrome、Firefox 和 Edge 等主流桌面浏览器，开发端口为 5173。移动端基于 Vant 组件库构建，针对触屏操作和小屏幕布局进行了专门适配，开发端口为 5174。两端通过同一网关的 8080 端口访问后端服务，在用户操作出现异常时应提供明确的错误提示信息而非空白页面或未捕获的异常堆栈。后端服务基于 Java 17 和 Spring Boot 3 构建，可在 Linux 和 macOS 环境下运行；基础设施通过 Docker 容器化封装，消除了对宿主机操作系统版本和软件安装路径的依赖。

第四章 系统详细设计与实现

4.1 系统架构设计

第三章的需求分析表明，系统需要支持各业务模块的独立开发与部署、PC 端和移动端的双端适配，以及可复现的自动化测试。单体架构下所有模块打包在同一进程中，任何局部改动都要全局重新部署，难以满足上述要求。微服务架构将业务逻辑拆分为多个独立服务，每个服务拥有自己的部署周期和技术栈选择空间，与本系统的需求契合度更高。基于这一判断，系统整体采用微服务架构，按部署职责划分为四个层次。最外层是用户接入层，PC 端和移动端两个前端应用分别运行在 5173 和 5174 端口，用户通过浏览器访问。所有前端请求汇入网关层，由 Spring Cloud Gateway 在 8080 端口统一接收。网关之后是业务服务层，包含认证服务（8081）、论坛服务（8082）、通知服务（8083）、媒体服务（8084）和搜索服务（8085）五个独立进程，以及被各服务共同依赖的 common-lib 公共库。最底层是基础设施层，MySQL 8.4（3306）、Redis 7.4（6379）和 Nacos 2.3.2（8848）三个组件通过 Docker Compose 编排部署。系统部署图如图 4-1 所示。

图 4-1 系统部署图

网关收到请求后，按配置文件中定义的六条路由规则进行分发。`/api/auth/**` 和 `/api/users/**` 转发至认证服务，`/api/forum/**` 转发至论坛服务，`/api/notifications/**` 转发至通知服务，`/api/media/**` 转发至媒体服务，`/api/search/**` 转发至搜索服务。每条路由均配置了 StripPrefix 过滤器，在转发前去除 URL 中的网关前缀层级，保证下游服务接收到的请求路径与其 Controller 映射一致。路由目标地址写作 `lb://服务名`，由 Nacos 注册中心将服务名解析为实际的 IP 和端口，实现基于服务发现的负载均衡。网关路由转发流程如图 4-2 所示。

图 4-2 网关路由转发流程图

五个业务服务和网关在启动时均向 Nacos 注册自身地址和元信息，Nacos 同时充当配置中心，为各服务提供运行时参数。服务间的调用关系比较简洁：只有搜索服务通过 OpenFeign 声明式客户端调用论坛服务的帖子列表接口获取数据，其余服务之间不存在直接调用。如果运行环境中没有 Nacos，网关的路由目标和搜索服务的 Feign URL 可通过环境变量指向固定的 localhost 地址，系统在直连模式下依然能够正常启动。
微服务架构的理想模型要求每个服务管理自己专属的数据库，以避免服务间产生隐式耦合。本系统当前的业务模块数量和数据量尚未达到必须拆库的程度，认证服务、论坛服务和通知服务共享同一个 MySQL 实例（数据库名 forum_system），各服务通过不同数据表实现逻辑隔离。这一折中方案在当前的业务体量下不会引发数据冲突，未来如果服务规模增长，可以按表拆分为独立数据库而不改动业务代码。Redis 被三方使用：认证服务存储会话状态和登录失败计数，论坛服务缓存帖子列表和详情数据，网关在鉴权过程中查询会话是否有效。搜索服务和媒体服务不直接访问任何数据库或缓存，前者的数据全部来源于对论坛服务的 Feign 调用，后者的推荐数据预置在内存中。
上述架构在松耦合、独立部署和环境适配三个维度上回应了第三章提出的非功能性需求：各服务通过 Nacos 按服务名寻址，实例增减对上游透明，满足可扩展性要求；Docker Compose 和环境变量注入使同一套代码可在开发、测试和部署三个阶段使用不同配置，满足可维护性要求；网关的 CORS 配置和双端前端共用同一组 API，满足兼容性要求。

4.2 功能模块设计

后端代码按 Maven 多模块组织。五个服务对应 auth-user-service、forum-service、notification-service、search-service 和 media-service 五个模块，加上承担路由与鉴权的 gateway-service 和被各模块共同依赖的 common-lib 公共库，共计七个子模块。common-lib 不独立运行，其余六个模块各自作为独立的 Spring Boot 应用部署。各模块的职责边界、对外接口以及相互依赖关系如图 4-3 所示。

图 4-3 系统功能模块结构图

common-lib 被其余六个模块共同依赖，其中定义了统一的接口响应封装类 ApiResponse 和全局常量类 ServiceConstants。后者集中管理 JWT 令牌类型标识、HTTP 请求头名称（Authorization、X-User-Id、X-Username）、Bearer 前缀和 Redis 键前缀等跨服务共享的常量值。将这些常量收敛到一个模块而非各服务各自硬编码，可以从源头上消除因拼写差异或值不一致而导致的联调故障。
gateway-service 是系统的统一入口，对外暴露 8080 端口。除了 4.1 节所述的路由转发职责外，该模块还承载了全局的 JWT 鉴权逻辑，通过 JwtRelayFilter 拦截所有请求，区分公开接口和受保护接口并完成令牌校验与用户身份透传。网关本身不包含任何业务逻辑。
auth-user-service 负责用户身份管理，对外提供六个接口：注册（POST /auth/register）、登录（POST /auth/login）、令牌刷新（POST /auth/refresh）、登出（POST /auth/logout）、查看个人信息（GET /users/me）以及健康探测（GET /users/ping）。该服务内部包含 JwtService 负责令牌签发与解析、AuthSessionService 负责 Redis 会话管理、LoginAttemptService 负责登录失败计数与锁定，以及 Spring Security 配置的内部过滤器链。
forum-service 是系统的核心业务模块，按资源类型可细分为三组接口。帖子接口覆盖列表查询（GET /posts，支持 page、size、sectionId 参数）、发布（POST /posts）、编辑（PUT /posts/{postId}）、删除（DELETE /posts/{postId}）、详情查询（GET /posts/{postId}）和点赞（POST /posts/{postId}/like）共六项操作。评论接口包括发表评论（POST /posts/{postId}/comments）、编辑评论（PUT /posts/{postId}/comments/{commentId}）和删除评论（DELETE /posts/{postId}/comments/{commentId}）三项操作，评论作为帖子的子资源挂在帖子路径下。分区列表由独立的 SectionController 提供（GET /sections），数据来源于启动时的种子初始化。该服务内部配置了 Redis 缓存策略，帖子列表和详情的缓存分别设置了不同的过期时间。
notification-service 提供三个接口：通知首页摘要（GET /notifications/home，返回最新公告和未读计数）、个人通知列表（GET /notifications/my）和标记已读（POST /notifications/{notificationId}/read）。首页摘要接口支持匿名访问，当请求头中未携带用户 ID 时返回未读数为零。该服务在启动时通过 NotificationDataInitializer 初始化若干条系统公告作为演示数据。
search-service 提供一个搜索接口（GET /search/posts，支持 keyword、sectionId、page、size 参数）。该服务内部通过 OpenFeign 客户端 ForumClient 调用论坛服务的帖子列表接口获取原始数据，然后在内存中按关键词对标题、内容摘要和作者名称进行过滤匹配后返回结果。搜索服务不拥有自己的数据库，所有数据均来源于论坛服务。
media-service 提供两个接口：媒体首页（GET /media/home）和推荐列表（GET /media/recommendations，支持 type 参数筛选音乐或书籍）。推荐数据在 MediaRecommendationService 中以静态列表方式预置于内存，不依赖外部数据库。该服务是系统中最轻量的模块，不访问 MySQL 和 Redis。
上述全部接口遵循 RESTful 风格设计：GET 方法用于资源的读取和查询，POST 方法用于资源的创建，PUT 方法用于资源的更新，DELETE 方法用于资源的删除。路径以资源名称为中心组织，子资源嵌套在父资源路径下（如评论接口挂在 /posts/{postId}/comments 下），路径中的动态参数用花括号标识。表 4-1 汇总了系统全部 22 个 API 接口的方法、路径和认证要求。

索引策略直接影响查询性能，表 4-2 列出了系统当前实际使用的全部索引。各表的 id 列为自增主键，MySQL 自动为其创建聚簇索引。JPA 的 ddl-auto:update 机制根据实体类中 @Column(unique=true) 注解，自动为 user_account.username 和 forum_section.name 生成唯一索引。post_like 表的联合唯一约束 uk_post_like_post_user 由 @UniqueConstraint 注解显式定义，约束列为 (post_id, user_id)，MySQL 据此创建联合唯一索引。值得注意的是 forum_post.section_id 和 post_comment.post_id 两个字段：前者在按分区筛选帖子列表时作为查询条件，后者在加载帖子评论列表时作为查询条件，当前数据量下全表扫描尚可接受，数据量增长后应手动添加普通索引以优化查询。

4.4 安全设计

安全设计需要解决两个问题：一是确认"请求来自谁"，二是判断"这个人能否执行该操作"。前者由认证服务内部的令牌签发与校验完成，后者由网关层的统一鉴权完成，登录失败防护则为整个认证流程加上一道额外的保险。
JwtService 承担令牌的生成和解析。生成时，该服务将用户 ID、用户名、会话 ID 和令牌类型（access 或 refresh）四项信息写入 JWT 的声明字段，用 HMAC-SHA 算法和不低于 32 字符的密钥对令牌整体做数字签名，再设置过期时间后输出 JWT 字符串。解析时的步骤与之相反：先验证签名是否完整，再检查令牌类型是否与调用场景匹配，最后从声明中取出用户身份信息。访问令牌和刷新令牌共用同一套签发逻辑，区别仅在于类型标识和过期时间。
用户注册时密码经 BCrypt 算法加密后存入数据库，原始密码不在任何环节以明文形式保留。登录时，AuthService 首先通过 LoginAttemptService 检查当前账户是否处于锁定状态。如果未锁定，则从数据库查询用户记录并使用 PasswordEncoder 比对密码。校验成功后清除失败计数，生成一个随机的 UUID 作为会话 ID，调用 JwtService 签发访问令牌和刷新令牌，同时通过 AuthSessionService 将会话 ID 与用户 ID 的映射关系写入 Redis 并设置与刷新令牌相同的过期时间。校验失败时，失败计数在 Redis 中原子递增，达到 5 次阈值后账户被锁定 600 秒。登录认证的完整流程如图 4-5 所示。

图 4-5 登录认证流程图

网关层的 JwtRelayFilter 对每个到达的请求执行访问控制判断，决策流程如图 4-6 所示。过滤器按优先级依次做三轮检查，任一轮命中即跳过后续步骤。第一轮检查请求方法是否为 OPTIONS（浏览器跨域预检），是则直接放行。第二轮检查请求是否为 GET 且路径命中公开读白名单——白名单包含帖子列表、帖子详情、分区列表、通知首页、媒体首页、媒体推荐和搜索共七个路径，命中则放行。第三轮检查路径是否以 /api/auth/、/api/users/ping 或 /actuator 开头，这些属于认证相关或健康探测接口，同样放行。
三轮检查均未命中的请求进入鉴权流程。过滤器从 Authorization 请求头中提取 Bearer 令牌，用 HMAC-SHA 密钥验证签名，检查声明中 uid、username、sid 和 typ 四个字段是否完整且 typ 为 access，最后查询 Redis 中以 auth:session: 为前缀拼接 sessionId 的键，确认会话记录存在且存储的 userId 与令牌声明一致。任一环节失败返回 401 和 JSON 格式的错误信息。全部通过后，过滤器将用户 ID 和用户名写入 X-User-Id、X-Username 请求头，转发至下游服务，下游服务无须再次解析令牌。

网关还在 application.yml 中配置了全局的跨域资源共享（CORS）规则，仅允许 `http://localhost:5173`（PC 端）和 `http://localhost:5174`（移动端）两个来源发起跨域请求，请求方法限定为 GET、POST、PUT 和 DELETE。不在白名单中的来源发起的请求将被浏览器直接拦截，不会到达后端服务。

4.5 开发与运行环境

后端使用 Java 17 语言和 Maven 构建工具，Spring Boot 版本为 3.2.5，微服务治理组件来自 Spring Cloud 2023.0.1 和 Spring Cloud Alibaba 2023.0.1.2。前端两端均运行在 Node.js 环境下，核心依赖为 Vue 3.5、Vue Router 4.5、Pinia 3 和 Axios 1.9，打包工具 Vite 5.4，TypeScript 编译器 5.9。PC 端的 UI 组件库为 Element Plus 2.13，移动端为 Vant 4.9。

4.6 认证与会话管理实现

AuthService 的 register 方法处理注册请求：先检查用户名是否已存在，已存在则抛出业务异常；通过后创建 UserAccount 实体，用 BCrypt 加密密码并持久化到数据库，最后调用 issueTokens 签发令牌对返回给前端。
登录逻辑在同一个类的 login 方法中实现，完整流程已由 4.4 节的图 4-5 描述。令牌刷新由 refresh 方法完成：解析刷新令牌取出用户 ID 和会话 ID，查询 Redis 确认会话仍然有效，有效则签发新的令牌对并延长会话的过期时间。登出由 logout 方法完成，逻辑很短——直接删除 Redis 中的会话记录，该会话下已签发的所有令牌随即失效。
前端的令牌管理集中在 api/http.ts 文件中，分为请求拦截器和响应拦截器两部分。请求拦截器在每次发送请求前从 localStorage 读取访问令牌写入 Authorization 头。响应拦截器负责处理 401 状态码，其静默刷新流程如图 4-7 所示。收到 401 后拦截器先排除认证接口本身（login/register/refresh/logout），这些接口的 401 直接报错不做刷新。随后检查 localStorage 中是否还有刷新令牌，没有则清除存储跳转登录页。如果刷新令牌存在且当前没有正在进行的刷新请求，拦截器发起 POST /api/auth/refresh 获取新令牌对，成功后更新 localStorage 并通过 resolveQueue 释放等待队列中的请求使其携带新令牌重试。如果此时已有一个刷新请求在途，当前请求会被加入 pendingQueue 排队等候，避免多个 401 响应同时触发刷新造成竞争。令牌更新完成后，http.ts 通过 window.dispatchEvent 发出 auth-session-changed 事件，main.ts 中的监听器收到事件后调用 Pinia auth store 的 syncFromStorage 方法，将最新令牌同步到全局状态。

4.7 论坛核心业务实现

ForumService 的 createPost 方法负责帖子发布，流程如图 4-8 所示。方法执行时先由 validateUser 检查请求头中的 userId 和 username 是否非空，缺失则返回 401；再根据请求体中的 sectionId 调用 loadSection 加载分区实体，分区不存在则返回 404。两项校验都通过后，方法构建 ForumPost 实体写入标题、正文、作者信息和分区信息，调用 forumPostRepository.save 持久化到 MySQL。方法上的 @CacheEvict(cacheNames = CACHE_POST_LIST, allEntries = true) 注解使持久化完成后帖子列表的全部缓存条目被自动淘汰。编辑（updatePost）和删除（deletePost）在 validateUser 之后还要经过 assertPostOwner 校验——比较请求头中的用户 ID 与帖子的 authorId，不一致则返回 4033 权限异常。删除帖子时会级联清除该帖子下的全部评论和点赞记录。

图 4-8 帖子发布流程图

帖子列表查询使用 Spring Data JPA 的分页机制，通过 PageRequest.of(page, size, Sort.by(DESC, "id")) 构建按 ID 降序排列的分页参数。当请求携带 sectionId 参数时，调用 forumPostRepository.findBySectionId 按分区筛选；否则调用 findAll 返回全部帖子。查询方法上标注了 @Cacheable 注解，缓存键由 sectionId、page 和 size 三个参数拼接生成（格式为 "sectionId:page:size" 或 "all:page:size"），缓存名称为 forum:post:list，过期时间在 RedisCacheConfig 中配置为 2 分钟。帖子详情查询的缓存名称为 forum:post:detail，缓存键为帖子 ID，过期时间为 5 分钟，查询结果包含帖子正文和按创建时间升序排列的全部评论。读操作优先命中缓存，写操作（发布、编辑、删除、评论、点赞）执行后通过 @CacheEvict 或 @Caching 注解淘汰相关缓存，其中发布帖子仅淘汰列表缓存，而编辑、删除、评论和点赞操作同时淘汰列表缓存和对应帖子的详情缓存。缓存的读写策略如图 4-9 所示。

评论的二级结构依赖 post_comment 表的 parent_comment_id 字段，发表流程如图 4-10 所示。addComment 方法在校验用户身份和帖子存在性后，调用 resolveParentCommentId 处理客户端传入的 parentCommentId：值为 null 时视为一级评论，不为 null 时去数据库查找父评论并确认它属于同一帖子，查不到则返回 404。评论实体持久化到 MySQL 后，帖子列表和对应帖子的详情缓存同步淘汰。编辑评论（updateComment）的权限校验与帖子相同，比较 comment.getUserId() 与请求头中的用户 ID。删除评论（deleteComment）的逻辑略有差异：canDeleteComment 方法允许帖子作者删除自己帖子下的任意评论，不限于评论作者本人。

点赞功能依赖 post_like 表的联合唯一约束 uk_post_like_post_user。likePost 方法首先通过 postLikeRepository.existsByPostIdAndUserId 查询是否已存在点赞记录，如果存在则直接抛出 4091 异常。随后构建 PostLike 实体并尝试插入数据库，由于存在联合唯一约束，即使在高并发场景下两个请求同时通过了 exists 检查，数据库也会在第二次插入时抛出 DataIntegrityViolationException 异常，业务代码在 catch 块中捕获该异常并返回重复点赞的提示信息，从而实现应用层和数据库层的双重去重保障。插入成功后，帖子的 like_count 字段通过 post.setLikeCount(post.getLikeCount() + 1) 递增并持久化，同时 @Caching 注解淘汰帖子列表和对应帖子的详情缓存。
论坛服务启动时，DemoDataInitializer 会检查数据库中是否已存在分区数据，如果没有则写入若干预设分区和 8 篇示例帖子。这一机制使得开发者在全新环境中拉起服务后即可看到有内容的论坛页面，回归测试脚本也依赖这些种子数据作为测试前提。

如 4.2 节所述，媒体服务不访问 MySQL 和 Redis，数据全部驻留在内存中。MediaRecommendationService 在构造函数中以硬编码方式初始化音乐和书籍两个推荐列表，每条记录包含标题、作者/艺术家和描述三个字段。home 接口从两个列表中各取前几条作为首页摘要，recommendations 接口根据 type 参数（music、book 或 all）返回完整列表。

deploy/docker-compose.yml 集中定义了 MySQL、Redis 和 Nacos 三个容器的镜像版本、端口映射、数据卷和健康检查规则，端口和密码全部通过环境变量注入，方便在不同环境中切换配置。后端服务的生命周期由两个 Bash 脚本管理：start-backend-all.sh 按 common-lib 构建 → auth → forum → notification → media → search → gateway 的顺序依次启动各服务，每启动一个就轮询其 /actuator/health 端点直至返回 200；stop-backend-all.sh 按反向顺序有序关闭，网关最先停止以阻止新请求进入。start-infra-auto.sh 负责基础设施的拉起，在有 Docker 的环境中执行 docker compose up -d，在无 Docker 的环境中回退到本机安装的 MySQL 和 Redis。
测试脚本同样以 Python 编写，共有三个。run_system_regression.py 通过标准 HTTP 请求驱动完整的业务流程测试，覆盖注册、登录、发帖、评论、搜索、通知、媒体、未授权访问、越权操作、并发重复点赞和令牌过期拦截等场景，执行结束后自动生成 Markdown 格式的功能测试报告。benchmark_forum_cache.py 对帖子列表接口执行一次冷启动请求和多次热缓存请求，记录最小值、最大值、平均值和 P95 延迟，输出缓存命中前后的性能改善幅度。load_test_concurrent.py 基于 Python 的 ThreadPoolExecutor，分别模拟不同规模的并发用户同时向帖子列表接口发送请求，统计各并发级别下的平均响应时间、P95 延迟、成功率和吞吐量（QPS），用于评估系统在不同压力下的承载能力。三个脚本的具体测试过程和输出结果将在第五章中详述。

5.4 测试结论

功能测试覆盖了系统全部 22 个 API 接口和 11 个端到端业务场景，通过率为 100%。正常业务流程（分区查询、注册登录、发帖评论搜索、通知和媒体浏览）运行结果与预期一致，异常路径下的四种防护机制（无令牌拦截、越权操作拦截、并发点赞去重、登出后会话失效）也按照第三章需求分析和第四章安全设计中定义的行为正确响应。
性能测试从两个角度评估了系统的运行效率。缓存基准测试表明 Redis 缓存对帖子列表接口的响应时间有明显的改善作用，并发负载测试在 10 至 100 并发用户的规模下验证了系统能够维持稳定的响应和较高的成功率。
需要指出的是，上述测试全部在本地开发环境的单机上执行，与生产环境的网络延迟和硬件配置存在差异。系统尚未进行云服务器部署下的大规模压力测试，测试覆盖的广度和深度仍有提升空间，相关局限将在第六章中进一步讨论。

第六章 总结与展望

6.1 总结

高校师生在课程答疑、学习资料共享、经验复盘和校园活动通知等场景中产生了大量在线交流需求。论坛系统能够将碎片化的问答按主题沉淀为可回查的知识内容，在上述场景中具有不可替代的作用。传统单体架构下前后端耦合、独立部署困难、缺乏移动端支持等问题使得既有校园论坛难以适应日益增长的用户规模和交互需求。本文正是基于此背景，设计并实现了一个基于微服务架构的高校论坛系统。
本文从研究背景和国内外现状出发，分析了现有校园论坛在架构、适配和测试方面存在的不足，将用户划分为游客和注册用户两种角色，梳理了用户认证、帖子管理、评论互动、点赞、通知、搜索、媒体推荐和网关访问控制八个功能模块的用例关系，并从性能、安全、可扩展性、可维护性和兼容性五个维度给出了非功能性需求的量化指标。
系统按部署职责划分为用户接入层、网关层、业务服务层和基础设施层四个层次。后端基于 Java 17 和 Spring Boot 3 构建，借助 Spring Cloud Alibaba 的 Nacos 注册中心和 Spring Cloud Gateway 网关组件将业务逻辑拆分为五个可独立部署的微服务，MySQL 8 负责数据持久化，Redis 7 承担缓存与会话管理。数据库设计包含 6 张业务表，安全设计围绕 JWT 令牌签发与校验、网关双重鉴权和登录失败锁定三条线路展开。
认证模块实现了注册、登录、令牌刷新与登出的完整闭环。论坛核心业务覆盖帖子的发布、编辑、删除和按分区分页查询，评论支持二级回复结构，点赞通过联合唯一约束和应用层双重校验实现去重，Redis 缓存策略对帖子列表和详情分别设置了不同的过期时间并由写操作自动淘汰。前端基于 Vue 3 开发，PC 端使用 Element Plus、移动端使用 Vant，两端共享统一的 Axios 请求层和 Pinia 状态管理逻辑，响应拦截器实现了令牌的静默刷新。
MySQL、Redis 和 Nacos 通过 Docker Compose 编排部署，Bash 脚本管理后端服务的顺序启动与有序停止。Python 功能测试脚本覆盖了 11 个端到端业务场景和全部 22 个 API 接口，缓存基准测试和并发负载测试分别验证了缓存策略的有效性和系统在 10 至 100 并发用户下的承载能力，测试通过率为 100%。

6.2 存在的问题与展望

系统在开发和测试过程中暴露了一些值得改进的地方：
（1）搜索服务的检索能力是最明显的短板。当前实现采用内存中逐条遍历和字符串 contains 匹配的方式过滤帖子，数据量较小时尚可工作，帖子数量增长后匹配效率会明显下降，分词、模糊匹配和相关度排序等高级检索特性也无法支持。已有研究在微服务架构下通过缓存机制与索引优化将端到端处理延迟控制在 2 秒以内[15]，后续可借鉴类似思路引入 Elasticsearch 全文搜索引擎，利用倒排索引和分词分析器提升检索精度和响应速度。
（2）日志采集与监控机制尚未建立。各微服务目前仅依赖 Spring Boot 默认的控制台日志输出，缺少统一的日志收集、存储和查询平台。分布式系统的故障定位需要跨多个服务节点关联日志信息，仅靠控制台输出难以胜任。已有研究通过集成 Hystrix 等监控组件实现了服务状态的实时追踪和自动熔断[17]，后续可引入 ELK（Elasticsearch + Logstash + Kibana）日志采集栈，结合链路追踪工具实现请求全链路的可观测性。
（3）部署架构尚未在云环境下进行生产级验证。全部测试在本地单机上完成，无法体现微服务架构在水平扩展和负载均衡上的优势。已有研究在智慧校园系统中将容器化微服务部署至 Kubernetes 集群，并讨论了性能瓶颈、服务间通信和数据一致性等实际挑战[14]，后续可参考这一路径将系统部署至云服务器，实现弹性伸缩和滚动更新，并接入 CDN 加速前端静态资源分发。
（4）测试体系有继续完善的空间。安全防护目前依赖 Spring Security 过滤器链和网关层的 JWT 双重校验，框架组件已能覆盖认证与访问控制的基本需求，后续若面向更复杂的生产环境，可借助 OWASP ZAP 等工具对 SQL 注入、XSS 等攻击向量做系统性扫描。单元测试层面，当前质量保障以端到端功能测试为主，粒度集中在接口级别；业务逻辑复杂度增长后，引入 JUnit 对核心服务方法编写单元测试并统计覆盖率，能够在更早的阶段发现问题，降低集成测试的排错成本。
