# 答辩 Q&A 准备

## 一、高频技术问题

### Q1：为什么选择微服务架构而不是单体架构？

本系统包含认证、论坛、通知、搜索、媒体五个业务模块和一个网关，功能边界清晰。单体架构下所有模块打包在同一进程中，任何局部改动都需要全局重新部署，开发效率和可维护性受限。微服务架构允许各服务独立开发、部署和扩缩，通过 Nacos 注册中心实现服务发现，新增模块只需注册一个新服务并在网关配置路由规则即可接入，不影响已有服务。当然，微服务也引入了额外的运维复杂度，本系统通过 Docker Compose 编排和 Bash 启停脚本来降低这部分成本。

### Q2：为什么选择 Spring Boot + Vue 而不是 Django + React 或其他组合？

后端选 Spring Boot 3 主要基于三点：Java 17 的跨平台能力和成熟的类型系统、Spring Cloud Alibaba 提供的微服务治理组件（Nacos + Gateway + OpenFeign）开箱即用、以及国内 Java 后端的社区生态和文档资源最为丰富。Django 虽然开发效率高，但缺少与 Nacos 等微服务组件的原生集成。前端选 Vue 3 是因为其依赖追踪的响应式机制降低了开发心智负担，双向数据绑定在表单场景中效率更高，中文生态和文档也优于 React。

### Q3：JWT 令牌一经签发无法撤销，你是怎么解决登出问题的？

纯 JWT 确实无法主动撤销，这是无状态认证的固有限制。本系统采用 JWT + Redis 混合方案：登录时在 Redis 中写入 sessionId → userId 的映射，网关的 JwtRelayFilter 在校验 JWT 签名之后还会查询 Redis 确认会话是否仍然有效。用户登出时删除 Redis 中的会话记录，已签发的令牌即使未过期也会在网关层被拒绝。这样既保留了 JWT 无状态校验的性能优势，又通过 Redis 补充了主动撤销能力。

### Q4：并发点赞去重是怎么实现的？

采用应用层 + 数据库层双重保障。应用层先通过 `existsByPostIdAndUserId` 查询是否已存在点赞记录，存在则直接返回"已点赞"。如果两个请求同时通过了应用层检查，数据库层的联合唯一约束 `uk_post_like_post_user(post_id, user_id)` 会拒绝第二次插入并抛出 `DataIntegrityViolationException`，业务代码在 catch 块中捕获并返回提示。这种乐观检查 + 悲观约束的组合以最小的锁开销解决了并发去重。

### Q5：Redis 缓存策略是怎么设计的？缓存一致性怎么保证？

帖子列表缓存 2 分钟过期，帖子详情缓存 5 分钟过期。读操作优先命中缓存，写操作（发布、编辑、删除、评论、点赞）通过 Spring 的 @CacheEvict 注解自动淘汰相关缓存。发布帖子仅淘汰列表缓存，编辑和删除同时淘汰列表和详情缓存。这是一种"写时淘汰"策略，不保证强一致性（存在短暂的缓存与数据库不一致窗口），但对论坛这种读多写少的场景足够——用户发帖后最多 2 分钟列表就会刷新。

### Q6：前端静默刷新令牌是怎么避免竞争条件的？

响应拦截器收到 401 后，检查是否已有刷新请求在途。如果没有，发起 POST /api/auth/refresh 获取新令牌对；如果已有一个刷新请求在途，当前请求加入 pendingQueue 排队等候。刷新成功后通过 resolveQueue 释放队列中所有等待的请求，让它们携带新令牌重试。这样无论同时有多少个 401 响应，只会触发一次刷新请求，避免了令牌被多次刷新导致的竞争问题。

### Q7：搜索服务为什么用内存过滤而不是 Elasticsearch？

当前系统定位为本科毕设，帖子数据量较小（种子数据 8 篇 + 测试新增），内存中逐条 contains 匹配完全能满足需求。引入 Elasticsearch 会增加一个独立的基础设施组件，运维复杂度和资源占用都会上升，对当前规模而言收益不明显。但在 6.2 展望中已经指出，帖子数量增长后应引入 Elasticsearch 利用倒排索引和分词分析器提升检索精度和响应速度。

### Q8：数据库为什么没有按微服务拆分成独立的库？

微服务架构的理想模型确实要求每个服务管理自己专属的数据库，但本系统当前只有 6 张业务表、3 个服务访问数据库，业务规模不需要为每个服务单独建库。认证、论坛和通知服务共享同一个 MySQL 实例（forum_system 库），各服务通过不同数据表实现逻辑隔离。这是一个务实的折中——避免了过早引入分布式事务的复杂性，同时保留了未来按表拆库的能力（不需要改动业务代码）。

## 二、测试数据相关问题

### Q9：测试数据可靠吗？在本地单机上测的并发有参考价值吗？

功能测试的 11 个场景全部通过，这部分不受运行环境影响——接口返回值要么正确要么不正确。性能测试确实在本地单机上执行，与生产环境存在差异，论文 5.4 节已明确指出这一局限。不过本地测试的价值在于：（1）验证了缓存策略的有效性（冷启动 59ms vs 热缓存 15ms，提升 74.66%），这个比例在任何环境下都是成立的；（2）并发测试在 1000 用户下成功率仍达 99.99%，QPS 保持在 1450 以上，说明系统在单机条件下已具备一定的承载能力。

### Q10：功能测试覆盖率够吗？为什么没有单元测试？

功能测试覆盖了全部 22 个 API 接口和 11 个端到端场景，包括正常流程和 4 种异常路径（无令牌、越权、重复点赞、登出失效），覆盖面在本科毕设中已算完整。没有引入 JUnit 单元测试主要是因为当前业务逻辑复杂度有限，端到端测试已经能够有效验证各接口的正确性。论文 6.2 展望中提到，业务逻辑复杂度增长后引入单元测试并统计覆盖率是下一步工作方向。

## 三、架构与设计问题

### Q11：Nacos 挂了系统还能用吗？

能。运维脚本支持在有 Nacos 和无 Nacos 两种模式之间自动切换。如果 Nacos 不可用，网关的路由目标和搜索服务的 Feign URL 通过环境变量指向固定的 localhost 地址，系统在直连模式下依然能够正常启动和运行，只是失去了动态服务发现和配置推送的能力。

### Q12：forum_post 表的 author_name 和 section_name 冗余字段违反范式，为什么保留？

这是性能和规范性之间的取舍。帖子列表是系统中访问频率最高的接口，冗余存储使得列表查询在单表上即可完成，省去了与 user_account 和 forum_section 的三表 JOIN。配合 Redis 缓存后数据库压力进一步降低。代价是用户修改昵称或分区重命名时需要额外的更新逻辑来同步冗余字段，但在当前系统中这两种操作频率很低，这一取舍是合理的。

### Q13：网关的 CORS 配置只允许 localhost，部署到服务器后怎么办？

CORS 配置中的允许来源地址通过 application.yml 配置，部署时修改为服务器的实际域名即可，不需要改代码。当前配置 localhost:5173 和 localhost:5174 是开发环境的设置。

## 四、创新点与价值问题

### Q14：你认为这个系统最大的创新点是什么？

三个层面：（1）架构层面，将微服务体系完整应用于高校论坛场景，包含服务拆分、网关鉴权、服务发现的完整链路，而非仅停留在前后端分离；（2）安全层面，JWT + Redis 混合会话管理配合前端静默刷新的并发队列机制，在本科论文中较少见；（3）质量保障层面，配套了三个 Python 自动化测试脚本（功能回归 + 缓存基准 + 并发负载），测试过程可复现、结果可量化，不依赖手工操作。

### Q15：这个系统和市面上的 Discuz、Discourse 比有什么优势？

不适合直接比较。Discuz 和 Discourse 是成熟的商业/开源产品，经过多年迭代，功能远比本系统丰富。本系统的价值不在于功能的完备性，而在于作为一个工程实践案例，完整地演示了微服务架构在中等规模 Web 应用中的落地路径——从需求分析到架构设计到编码实现到自动化测试的全流程。

## 五、答辩演示建议

### 演示重点（如有现场演示环节）
1. 启动 Docker Compose → 启动后端服务 → 打开 PC 端和移动端
2. 演示注册 → 登录 → 发帖 → 评论 → 点赞 → 搜索的完整流程
3. 演示游客无法发帖（401 拦截）→ 登录后解锁
4. 运行一个测试脚本，展示自动化测试报告

### 容易被追问的点
- "Docker Compose 的 depends_on 能保证服务启动顺序吗？" → 只保证容器启动顺序，不保证服务就绪，所以启动脚本用了健康端点轮询。
- "为什么用 Python 写测试脚本而不是 JMeter？" → Python 标准库即可完成 HTTP 请求和并发测试，不依赖额外工具，脚本可直接版本管理。
- "你的系统能支撑多少用户？" → 本地单机测试下 1000 并发用户时 QPS 约 1450，成功率 99.99%。生产环境需要云服务器部署和负载均衡才能给出准确数字。
