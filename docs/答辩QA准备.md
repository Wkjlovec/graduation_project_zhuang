# 答辩 Q&A 准备

## 一、选题与研究方法（通用问题）

### Q1：你为什么选择这个课题？

两个原因。一是实际需求——高校师生在课程答疑和资料共享中确实需要一个能长期保存内容的讨论平台，即时通讯群聊的信息几天后就难以检索。二是技术兴趣——微服务架构在工业界已经广泛应用，但在高校论坛这类中等规模的 Web 系统中完整落地的工程案例并不多，做这个课题可以同时锻炼架构设计和工程实践能力。

### Q2：研究这个课题的意义和目的是什么？

目的是解决现有高校论坛的三个问题：单体架构扩展困难、缺乏移动端适配、测试依赖手工操作。意义在于提供一个可参考的工程实践路径——从微服务拆分、双端前端适配、JWT 认证闭环到自动化测试，在同一个项目中打通完整链路。

### Q3：你的论文采用了哪些与软件工程相关的研究方法？

主要是工程实践方法。需求分析阶段用了用例建模（UML 用例图和活动图），设计阶段用了分层架构设计和 ER 建模，实现阶段是迭代开发（先搭骨架再逐服务填充），测试阶段用了黑盒端到端测试和基准性能测试。没有用问卷调查或统计分析这类社科方法，因为这是一个系统开发类论文，核心是设计、实现和验证。

### Q4：论文中的核心概念是什么？用你自己的话概括。

核心概念是"微服务架构在高校论坛中的工程落地"。具体来说，把一个论坛系统按业务边界拆成多个独立部署的服务，用 Nacos 做服务发现、Gateway 做统一网关、JWT+Redis 做认证，前端 PC 和移动端共享同一套后端 API，再用自动化脚本验证整个系统的正确性和性能。

### Q5：全文的基本框架是怎么安排的？各部分之间的逻辑关系如何？

六章形成闭环。第一章提出问题（背景和现状），第二章介绍解决问题所需的技术工具，第三章分析要解决什么（需求），第四章说明怎么解决的（设计和实现），第五章验证解决效果（测试），第六章回顾和展望。每一章的输出是下一章的输入：需求指标驱动架构选择，架构设计指导编码实现，测试结果回应需求指标。

### Q6：论文有何创新之处？

三个层面。架构上，将 Spring Cloud 微服务体系（Nacos + Gateway + OpenFeign）完整应用于高校论坛，而不是只做前后端分离。安全上，JWT + Redis 混合认证方案配合前端静默刷新的并发队列机制。质量保障上，三套 Python 自动化测试脚本覆盖功能回归、缓存基准和并发负载，执行后自动生成报告，不依赖手工操作。

### Q7：论文和系统有哪些不足之处？

两个主要不足。一是搜索服务采用内存逐条过滤，帖子量增长后效率会下降，应引入 Elasticsearch。二是全部测试在本地单机完成，没有在云环境下做过生产级验证。此外，没有引入单元测试和安全扫描工具，测试覆盖的深度还有提升空间。

### Q8：如果不按照你说的微服务方式去做，结果会怎样？

单体架构在用户量小的时候完全够用，开发和部署都更简单。问题会在系统规模增长后暴露——模块间耦合导致改一个功能牵动全局，新功能接入需要重新编译整个工程，不同模块无法独立扩容。多数老旧高校论坛正是因为这些原因逐渐停止维护。微服务不是唯一选择，但对于需要持续迭代和多模块协作的系统，它提供了更好的工程基础。

### Q9：论文虽未论及，但与其较密切相关的问题还有哪些？

消息队列（如 RabbitMQ）可以进一步解耦服务间通信，比如发帖后异步触发通知；全文搜索引擎（Elasticsearch）可以替代当前的内存过滤；链路追踪（如 SkyWalking）可以帮助定位分布式系统中的性能瓶颈；CI/CD 持续集成流程可以将测试脚本接入代码提交后的自动验证。这些都是微服务工程实践中常见的配套组件，本文因篇幅和时间限制未涉及。

## 二、技术选型问题

### Q10：为什么选择 Spring Boot + Vue 而不是 Django + React？

后端选 Spring Boot 3 是因为 Spring Cloud Alibaba 提供 Nacos、Gateway、OpenFeign 等微服务组件的开箱支持，Django 在这方面缺少原生集成。前端选 Vue 3 是因为依赖追踪的响应式机制降低开发心智负担，双向数据绑定在表单场景中效率更高，中文社区生态也最为成熟。

### Q11：为什么用 Python 写测试脚本而不是 JMeter 或 Postman？

Python 的 urllib 和 threading 是标准库，不需要安装任何第三方依赖，脚本可以直接纳入版本管理。JMeter 虽然功能更强，但需要额外安装 GUI 工具，不适合在 CI 环境中自动执行。Postman 适合手动调试接口但不擅长并发测试。Python 脚本兼顾了功能测试、缓存基准和并发负载三种场景，且输出格式可自定义。

### Q12：MySQL 和 Redis 的分工是什么？为什么两个都要用？

MySQL 负责数据持久化——用户账户、帖子、评论、点赞、通知等结构化数据存在 MySQL 中，断电不丢失。Redis 负责两件事：一是缓存高频读取的帖子列表和详情（命中时响应从 59ms 降到 15ms），二是存储认证会话和登录失败计数这类临时状态。两者职责不同，缺一不可。

## 三、设计层面的具体问题

### Q13：为什么选微服务架构而不是单体？

本系统有五个功能边界清晰的业务模块，微服务让它们可以独立开发和部署。单体架构改一个模块就要全局重新部署。当然微服务也增加了运维复杂度，本系统用 Docker Compose 和 Bash 启停脚本来降低这部分成本。

### Q14：数据库为什么没按微服务拆成独立的库？

当前只有 6 张表、3 个服务访问数据库，业务规模不需要拆库。共享一个 MySQL 实例通过不同数据表做逻辑隔离，避免了过早引入分布式事务的复杂性，同时保留了未来按表拆库的能力。

### Q15：forum_post 表的冗余字段违反范式，为什么保留？

author_name 和 section_name 冗余存储了 user_account.username 和 forum_section.name。帖子列表是访问最频繁的接口，冗余让查询在单表上完成，省去三表 JOIN。代价是用户改昵称时需要同步更新帖子中的冗余字段，但这种操作频率很低。

### Q16：ER 图中的实体和关系能详细说一下吗？

五个实体：user_account、forum_section、forum_post、post_comment、notification_message。七条关系中六条是一对多（用户发布帖子、用户撰写评论、用户接收通知、分区包含帖子、帖子包含评论、评论自关联回复），一条是多对多（用户点赞帖子，通过 post_like 关联表拆解，联合唯一约束保证每人每帖最多一次点赞）。

### Q17：网关的 JwtRelayFilter 具体做了什么？

七级检查。第一轮放行 OPTIONS 预检请求，第二轮放行 GET 白名单（帖子列表、详情、分区、通知首页、媒体、搜索），第三轮放行认证接口和健康探测。不满足这三轮的进入鉴权：提取 Bearer 令牌 → 验证 JWT 签名 → 检查声明字段（uid/username/sid/typ=access）→ 查询 Redis 确认会话存在。全部通过后将用户 ID 和用户名写入请求头透传给下游服务。

### Q18：CORS 只配了 localhost，部署后怎么办？

CORS 允许来源在 application.yml 中配置，部署时改成服务器域名即可，不需要改代码。当前 localhost:5173 和 5174 是开发环境设置。

## 四、实现层面的具体问题

### Q19：JWT 不能撤销，登出怎么实现的？

JWT + Redis 混合方案。登录时在 Redis 写入 sessionId → userId 映射，网关校验 JWT 签名后还查 Redis 确认会话有效。登出时删除 Redis 记录，令牌虽未过期但会在网关层被拒绝。兼顾了无状态校验的性能和主动撤销的能力。

### Q20：并发点赞去重怎么做的？

双重保障。应用层先查 existsByPostIdAndUserId，存在则直接返回"已点赞"。如果两个请求同时通过应用层检查，数据库的联合唯一约束 uk_post_like_post_user 会拒绝第二次插入，业务代码捕获 DataIntegrityViolationException 后返回提示。乐观检查 + 悲观约束，零锁开销。

### Q21：缓存策略怎么保证一致性？

写时淘汰策略。发布帖子时淘汰列表缓存，编辑和删除同时淘汰列表和详情缓存，通过 @CacheEvict 注解实现。不保证强一致性——存在短暂的缓存与数据库不一致窗口（最多 2 分钟列表 / 5 分钟详情），但对论坛这种读多写少的场景完全够用。

### Q22：前端静默刷新令牌怎么避免竞争？

响应拦截器收到 401 后，检查是否已有刷新请求在途。没有就发起 POST /api/auth/refresh；已有则当前请求加入 pendingQueue 排队。刷新成功后通过 resolveQueue 释放所有等待请求，让它们携带新令牌重试。无论多少个 401 同时触发，只会发一次刷新请求。

### Q23：搜索服务为什么不用 Elasticsearch？

当前帖子量很小（种子数据 8 篇 + 测试新增），内存 contains 匹配完全够用。引入 Elasticsearch 会增加一个独立组件，运维复杂度和资源占用上升，收益不明显。论文 6.2 展望中已指出后续应引入 Elasticsearch。

### Q24：服务间通信是怎么实现的？

只有搜索服务通过 OpenFeign 声明式客户端调用论坛服务的 GET /posts 接口获取数据，其余服务之间没有直接调用。OpenFeign 把 HTTP 调用封装成接口声明，调用方只需写接口方法，框架自动处理序列化、负载均衡和错误重试。

### Q25：Nacos 挂了系统还能用吗？

能。运维脚本支持有 Nacos 和无 Nacos 两种模式自动切换。无 Nacos 时网关路由和搜索服务的 Feign URL 通过环境变量指向 localhost 固定地址，系统在直连模式下正常运行，只是失去了动态服务发现能力。

### Q26：Docker Compose 的 depends_on 能保证服务就绪吗？

不能。depends_on 只保证容器启动顺序，不保证容器内的服务已完成初始化。所以启动脚本在每个服务启动后会轮询 /actuator/health 端点，直到返回 200 才启动下一个服务。

## 五、测试层面的具体问题

### Q27：功能测试覆盖率够吗？

覆盖了全部 22 个 API 接口和 11 个端到端场景，包括 5 个正常业务流程和 4 种异常路径（无令牌拦截、越权操作、并发重复点赞、登出后失效）以及 2 个令牌生命周期场景。在本科毕设中这个覆盖面已经比较完整。

### Q28：为什么没有单元测试？

当前业务逻辑复杂度有限，端到端测试已经能有效验证各接口的正确性。JUnit 单元测试的价值在业务逻辑复杂、分支条件多的场景下更明显。论文 6.2 展望中提到，后续引入单元测试是改进方向。

### Q29：本地测试数据有参考价值吗？

功能测试不受环境影响——接口返回值对就是对、错就是错。性能测试的绝对数值在本地和生产环境会不同，但缓存命中前后的相对改善幅度（74.66%）在任何环境下都成立。并发测试在 1000 用户下成功率 99.99%，说明代码层面没有并发缺陷。论文 5.4 已明确指出本地测试的局限性。

### Q30：你的系统能支撑多少用户？

本地单机测试下 1000 并发用户时 QPS 约 1450，成功率 99.99%。但这是本地环境数据，生产环境的承载能力取决于服务器配置、网络延迟和负载均衡策略。要给出准确数字需要在云服务器上部署并进行压力测试。

## 六、宏观与延伸问题

### Q31：立论的主要依据是什么？

两方面。一是现实问题——对照 CNNIC 报告和国内外论坛发展现状，现有高校论坛在架构、移动端和测试方面确实存在短板。二是技术可行性——Spring Cloud 微服务体系和 Vue 前后端分离技术栈在工业界已有大量成熟案例，将其迁移到高校论坛场景在技术上是可行的。

### Q32：这个系统和市面上的 Discuz、Discourse 比有什么优势？

不适合直接比较。Discuz 和 Discourse 是经过多年迭代的成熟产品，功能远比本系统丰富。本系统的价值不在功能完备性，而在于作为工程实践案例，完整演示了微服务架构在中等规模 Web 应用中的落地路径——从需求分析到自动化测试的全流程。

### Q33：如果给你更多时间，你会怎么改进？

三个优先级最高的改进。一是引入 Elasticsearch 替换内存搜索。二是部署到云服务器接入 Kubernetes 做生产级验证。三是引入 JUnit 单元测试并接入 CI/CD 流水线，每次代码提交后自动运行全套测试。

---

### Q34：系统有管理员账户吗？管理员和普通用户的权限有什么不同？

当前系统没有管理员角色。user_account 表中没有 role 字段，所有注册用户享有完全相同的操作权限。权限控制只有一条规则：帖子和评论的编辑、删除仅限内容作者本人——代码中通过 assertPostOwner 比对请求头中的 userId 与帖子的 authorId 来判断。帖子作者还可以删除自己帖子下的任意评论（canDeleteComment 方法额外判断了帖子 authorId）。

这是一个有意的简化设计。高校论坛的用户角色差异不大，引入 RBAC（基于角色的访问控制）会增加数据库字段、权限判断逻辑和管理界面的开发量，对当前系统规模来说收益有限。如果后续需要区分管理员（比如内容审核、用户封禁），可以在 user_account 表增加 role 字段，并在网关或各服务的拦截器中添加角色校验逻辑，现有架构完全支持这一扩展。

## 七、答辩现场建议

### 回答原则
- 先给结论再展开，不要铺垫太长
- 不知道的坦诚说"这部分我还没有深入研究"，不要编
- 被指出不足时认可并说明改进方向，不要辩解

### 容易被追问的点
- "你说微服务好，那微服务的缺点是什么？" → 运维复杂度增加（服务数量多、网络调用多、分布式事务难处理），本系统用 Docker Compose 和 Bash 脚本来降低运维成本。
- "Spring Security 在你系统里做了什么？" → 认证服务内部用 Spring Security 的 SecurityFilterChain 配置了接口的访问规则，比如哪些路径需要认证、哪些放行。但实际的 JWT 校验逻辑主要在网关的 JwtRelayFilter 中完成。
- "你的系统有做日志记录吗？" → 各服务使用 Spring Boot 默认的 Logback 控制台日志输出，没有搭建统一的日志收集平台。这是论文中提到的后续改进方向之一。
- "为什么前端用两套代码而不是一套响应式？" → 两套代码各自适配终端特性（PC 用 Element Plus 的分栏布局，移动端用 Vant 的触屏组件），交互体验优于单纯的 CSS 响应式。底层 API 层和状态管理是共享的，改一处接口两端同步生效。

| OpenFeign | 声明式 HTTP 客户端，将跨服务的远程调用封装为 Java 接口方法，本系统中搜索服务通过它调用论坛服务 |
| Vue 3 | 渐进式前端框架，采用依赖追踪的响应式机制，数据变化时自动更新视图，本系统 PC 端和移动端均基于 Vue 3 |

| RESTful | 一种 API 设计风格，用 HTTP 方法（GET/POST/PUT/DELETE）对应资源的读取/创建/更新/删除 |

| JPA | Java Persistence API，Java 的 ORM 规范，底层由 Hibernate 引擎实现对象与数据表的映射 |
| BCrypt | 密码哈希算法，本系统用它加密存储用户密码，原始密码不以明文形式保留 |
| HMAC-SHA | 基于哈希的消息认证码算法，本系统用它对 JWT 令牌进行数字签名，防止令牌被篡改 |
| QPS | Queries Per Second，每秒查询数，衡量系统吞吐量的指标 |
| P95 | 第 95 百分位响应时间，表示 95% 的请求在该时间内完成 |
| Elasticsearch | 开源全文搜索引擎，基于倒排索引，本系统展望中提到可用于替代当前的内存过滤搜索 |
| Kubernetes (K8s) | 容器编排平台，支持自动扩缩容和滚动更新，本系统展望中提到可用于云端生产部署 |
| CI/CD | Continuous Integration / Continuous Delivery，持续集成/持续交付，自动化的代码构建、测试和部署流程 |
| ER 图 | Entity-Relationship Diagram，实体关系图，用矩形（实体）、菱形（联系）和椭圆（属性）描述数据库结构 |
| UML | Unified Modeling Language，统一建模语言，本系统用其中的用例图和活动图描述需求 |
