第二章 相关技术及方法

2.1 后端技术

本文系统的后端采用 Java 17 语言开发。Java 是一门面向对象的跨平台编程语言，通过 JVM 虚拟机实现了编译产物在不同操作系统上的一致运行，其成熟的类型系统和垃圾回收机制使之在大型 Web 应用开发中被长期广泛使用。目前 JDK 存在 8、11、17 和 21 四个长期支持版本，本文选用 JDK 17 作为开发环境，该版本在密封类、模式匹配等语言特性上相较于 JDK 8 有明显增强，同时具备稳定的社区支持和依赖库兼容性。

Spring 框架是 Java 生态中应用最为广泛的企业级开发框架，其核心设计思想包括控制反转（IoC）和面向切面编程（AOP）两个方面。控制反转将对象的创建和依赖关系交由容器管理，调用方不再直接实例化被调用对象，而是从容器中获取，从而降低了模块之间的耦合度。面向切面编程则允许将日志记录、权限校验等横切关注点从业务逻辑中分离出来，以独立模块的形式统一维护，避免相同代码在多处重复出现。在 Spring 框架的基础上，Spring Boot 进一步简化了应用的初始搭建过程，其自动装配机制根据引入的依赖自动完成配置，内嵌的 Tomcat 容器使应用无须额外部署即可独立运行[10]。本文全部后端服务均基于 Spring Boot 3 构建。

微服务架构将原本集中在一个进程内的业务逻辑拆分为多个独立运行的服务，每个服务拥有自己的数据存储和部署周期，服务之间通过网络接口通信。这种拆分方式使得各服务能够独立扩缩和版本迭代，但也要求额外的基础设施来处理服务寻址、配置同步和流量管控等问题[11]。在分布式环境下，服务间的通信方式对系统整体性能有显著影响，同步调用在低延迟场景中效率较高，而异步消息在高负载下具备更好的可扩展性和弹性[17]。微服务架构在带来灵活性的同时也引入了容错需求，断路器和故障转移等机制的设计对于防止级联故障至关重要[16]。在 API 网关层面，网关作为微服务系统的统一入口，承担请求路由、协议适配和安全控制等职责[15]。Spring Cloud Alibaba 为 Spring Boot 应用提供了一套面向微服务场景的治理组件：Nacos 承担服务注册中心和配置中心的双重角色，各服务启动时向 Nacos 注册自身地址，调用方按服务名而非固定 IP 寻址，使实例的增减对上游透明；Spring Cloud Gateway 作为系统的统一入口，基于路由规则将请求分发至目标服务，并在过滤器链中嵌入令牌校验等横切逻辑；OpenFeign 则将跨服务的 HTTP 调用封装为接口声明，降低了远程通信的编码成本。

用户认证采用 JWT（JSON Web Token）方案。JWT 将用户身份声明和过期时间编码为经数字签名的令牌字符串，服务端凭签名即可验证令牌真伪，无须在服务端集中保存会话状态，因此天然适配多实例部署的分布式环境。Spring Security 框架通过过滤器链机制对每次请求进行拦截校验，与网关层的令牌验证形成两级安全屏障。由于 JWT 一经签发便无法主动撤销，本文额外使用 Redis 存储会话状态来支持登出和令牌失效操作。有研究系统梳理了微服务架构下认证机制面临的安全挑战与应对方案，指出基于令牌的无状态认证在多服务环境中具有明显的工程优势[6]。

数据持久化采用 MySQL 8，该数据库支持标准 SQL 查询、事务处理和索引优化，适合存储用户账户、帖子内容、评论等具有明确结构和关联关系的业务数据[14]。在微服务架构的理想模型中，每个服务应管理自身专属的数据域以避免隐式耦合[12]。本文系统出于毕业设计规模的考量，认证服务、论坛服务和通知服务共享同一个 MySQL 数据库实例，各服务通过不同的数据表实现逻辑隔离，搜索服务和媒体服务则不直接访问数据库。本文通过 JPA 规范完成 Java 对象与数据表之间的映射，将数据库访问操作抽象为接口方法调用，借助 Hibernate 引擎处理实体的生命周期管理和级联操作。缓存层采用 Redis 7，数据驻留在内存中，读写延迟通常在亚毫秒级别。本文将 Redis 用于两个方面：一是对帖子列表和详情等高频访问数据设置带过期时间的缓存，缓存命中时直接返回结果以降低数据库查询压力；二是存储认证会话和登录失败计数等临时状态，利用其原子操作特性保证并发场景下数据的一致性。

系统使用 Docker 对 MySQL、Redis 和 Nacos 进行容器化封装，通过 Docker Compose 编排文件定义各容器的镜像版本、端口映射、数据卷和启动依赖关系，执行单条命令即可在任意安装了 Docker 的环境中拉起全部基础服务，消除了因开发者本地环境差异导致的配置不一致问题。容器化部署已成为微服务系统实现平台无关交付的常用手段[13]。在实际工程中，容器化与微服务的结合不仅用于 Web 应用，也被广泛应用于安全中台的建设，通过容器化部署和分布式服务治理实现业务模块解耦[18]。运维脚本以 Bash 编写，负责后端服务的顺序启动、健康探测和有序停止，并支持根据 Nacos 是否可用在完整模式与直连模式之间自动切换。测试脚本以 Python 编写，承担全链路回归测试和缓存基准测试任务，执行完成后自动生成 Markdown 格式的结果报告。全部脚本遵循幂等原则设计，确保在不同时间点执行能够产生可比较的结果。

2.2 前端技术

Web 前端技术建立在 HTML、CSS 和 JavaScript 三项基础之上。HTML 负责页面的结构骨架，CSS 控制样式与布局，JavaScript 则赋予页面动态交互的能力，使开发者能够响应用户操作、处理数据并与后端服务器通信。随着前端工程复杂度的持续增长，直接操作浏览器 DOM 节点的开发方式越来越难以维护，由此催生了以组件化和数据驱动为核心理念的现代前端框架。

目前主流的前端框架主要有 Vue、React 和 Angular 三个。Angular 提供了一体化的解决方案，内置依赖注入、模板引擎和路由系统，适合大型团队的企业级项目开发，但由于采用 TypeScript 语言且概念体系庞大，学习曲线相对陡峭。React 和 Vue 都采用了组件化的开发思想，将完整页面拆分为若干功能独立的逻辑单元，同时通过虚拟 DOM 技术减少不必要的页面重绘以提升渲染效率。二者的主要区别在于数据响应机制：React 要求开发者通过显式调用 setState 或 Hook 函数触发视图更新，数据变化的时机完全由开发者控制；Vue 则采用依赖追踪的方式，在数据被修改时自动识别受影响的视图区域并触发重新渲染，无须开发者手动介入。Vue 还提供了双向数据绑定特性，能够在表单场景中实现数据与视图的实时同步，简化了交互逻辑的编写。考虑到 Vue 中文生态丰富、上手门槛较低，且官方配套了 Vite 构建工具、Vue Router 路由库和 Pinia 状态管理库等开箱即用的工具链，本文选用 Vue 3 作为前端框架[4]。

PC 端选用 Element Plus 作为组件库，该库基于 Vue 3 开发，提供了表格、表单、对话框、分页器等常用桌面端交互组件。移动端选用 Vant 组件库，针对触屏操作和小屏幕布局进行了专门优化，提供导航栏、单元格、浮动气泡等移动端常见交互模式。两端虽然选择了不同的 UI 组件库，但共享同一套 Axios 封装的 HTTP 请求层与后端进行数据通信，使用 Pinia 管理全局的用户认证状态，通过 Vue Router 实现页面路由控制和未登录访问拦截。前端项目使用 Vite 进行打包构建，Vite 利用浏览器原生的 ES 模块机制实现按需编译，在开发环境下的模块热替换速度显著快于传统的 Webpack 方案，有效缩短了前端开发的调试反馈周期。
