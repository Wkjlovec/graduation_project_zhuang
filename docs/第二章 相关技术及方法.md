第二章 相关技术及方法

## 2.1 后端技术

本文系统的后端采用 Java 17 语言开发。Java 是一门面向对象的跨平台编程语言，通过 JVM 虚拟机实现了编译产物在不同操作系统上的一致运行，其成熟的类型系统和垃圾回收机制使之在大型 Web 应用开发中被长期广泛使用。目前 JDK 存在 8、11、17 和 21 四个长期支持版本，本文选用 JDK 17 作为开发环境，该版本在密封类、模式匹配等语言特性上相较于 JDK 8 有明显增强，同时具备稳定的社区支持和依赖库兼容性。

Spring 框架是 Java 生态中应用最为广泛的企业级开发框架，其核心设计思想包括控制反转（IoC）和面向切面编程（AOP）两个方面。控制反转将对象的创建和依赖关系交由容器管理，调用方不再直接实例化被调用对象，而是从容器中获取，从而降低了模块之间的耦合度。面向切面编程则允许将日志记录、权限校验等横切关注点从业务逻辑中分离出来，以独立模块的形式统一维护，避免相同代码在多处重复出现。在 Spring 框架的基础上，Spring Boot 进一步简化了应用的初始搭建过程，其自动装配机制根据引入的依赖自动完成配置，内嵌的 Tomcat 容器使应用无须额外部署即可独立运行[9]。本文全部后端服务均基于 Spring Boot 3 构建。

微服务架构将原本集中在一个进程内的业务逻辑拆分为多个独立运行的服务，每个服务拥有自己的数据存储和部署周期，服务之间通过网络接口通信。这种拆分方式使得各服务能够独立扩缩和版本迭代，但也要求额外的基础设施来处理服务寻址、配置同步和流量管控等问题[10]。Spring Cloud Alibaba 为 Spring Boot 应用提供了一套面向微服务场景的治理组件：Nacos 承担服务注册中心和配置中心的双重角色，各服务启动时向 Nacos 注册自身地址，调用方按服务名而非固定 IP 寻址，使实例的增减对上游透明；Spring Cloud Gateway 作为系统的统一入口，基于路由规则将请求分发至目标服务，并在过滤器链中嵌入令牌校验等横切逻辑；OpenFeign 则将跨服务的 HTTP 调用封装为接口声明，降低了远程通信的编码成本。

用户认证采用 JWT（JSON Web Token）方案。JWT 将用户身份声明和过期时间编码为经数字签名的令牌字符串，服务端凭签名即可验证令牌真伪，无须在服务端集中保存会话状态，因此天然适配多实例部署的分布式环境。Spring Security 框架通过过滤器链机制对每次请求进行拦截校验，与网关层的令牌验证形成两级安全屏障。由于 JWT 一经签发便无法主动撤销，本文额外使用 Redis 存储会话状态来支持登出和令牌失效操作。

## 2.2 数据库技术

数据库是 Web 应用系统进行数据组织和持久化存储的基础，不同类型的数据库适用于不同的应用场景。关系型数据库以行列表格的形式组织数据，通过主键和外键建立表与表之间的关联关系，适合存储结构明确、需要事务保障的业务数据。非关系型数据库则根据数据模型的不同可进一步分为键值型、文档型、列式和图数据库等类别，在高吞吐、低延迟或数据结构灵活的场景中具有优势。

MySQL 是目前使用最为广泛的开源关系型数据库管理系统之一，支持标准 SQL 查询、事务处理、索引优化和存储过程等功能，在中小规模 Web 应用中被作为首选的持久化方案[11]。本文使用 MySQL 8 存储用户账户、帖子内容、评论、分区和点赞等核心业务数据，通过 JPA（Java Persistence API）规范完成 Java 对象与数据表之间的映射，将数据库访问操作抽象为接口方法调用，减少手写 SQL 的工作量，同时借助 Hibernate 作为 JPA 的实现引擎来处理实体的生命周期管理和级联操作。

Redis 是基于内存的键值存储系统，数据驻留在 RAM 中，读写延迟通常在亚毫秒级别，远低于磁盘数据库的访问速度。Redis 提供字符串、哈希、列表、集合和有序集合等多种数据结构，并支持对键设置过期时间以实现自动淘汰。本文将 Redis 用于两个方面：一是对帖子列表和帖子详情等高频访问数据设置带过期时间的缓存，当缓存命中时直接返回结果，避免每次请求都穿透到 MySQL；二是存储用户认证会话和登录失败计数等具有时效性的临时状态，利用 Redis 的原子递增操作保证并发场景下计数的准确性。

## 2.3 前端技术

Web 前端技术建立在 HTML、CSS 和 JavaScript 三项基础之上。HTML 负责页面的结构骨架，CSS 控制样式与布局，JavaScript 则赋予页面动态交互的能力，使开发者能够响应用户操作、处理数据并与后端服务器通信。随着前端工程复杂度的持续增长，直接操作浏览器 DOM 节点的开发方式越来越难以维护，由此催生了以组件化和数据驱动为核心理念的现代前端框架。

目前主流的前端框架主要有 Vue、React 和 Angular 三个。Angular 提供了一体化的解决方案，内置依赖注入、模板引擎和路由系统，适合大型团队的企业级项目开发，但由于采用 TypeScript 语言且概念体系庞大，学习曲线相对陡峭。React 和 Vue 都采用了组件化的开发思想，将完整页面拆分为若干功能独立的逻辑单元，同时通过虚拟 DOM 技术减少不必要的页面重绘以提升渲染效率。二者的主要区别在于数据响应机制：React 要求开发者通过显式调用 setState 或 Hook 函数触发视图更新，数据变化的时机完全由开发者控制；Vue 则采用依赖追踪的方式，在数据被修改时自动识别受影响的视图区域并触发重新渲染，无须开发者手动介入。Vue 还提供了双向数据绑定特性，能够在表单场景中实现数据与视图的实时同步，简化了交互逻辑的编写[12]。考虑到 Vue 中文生态丰富、上手门槛较低，且官方配套了 Vite 构建工具、Vue Router 路由库和 Pinia 状态管理库等开箱即用的工具链，本文选用 Vue 3 作为前端框架。

PC 端选用 Element Plus 作为组件库，该库基于 Vue 3 开发，提供了表格、表单、对话框、分页器等常用桌面端交互组件。移动端选用 Vant 组件库，针对触屏操作和小屏幕布局进行了专门优化，提供导航栏、单元格、浮动气泡等移动端常见交互模式。两端虽然选择了不同的 UI 组件库，但共享同一套 Axios 封装的 HTTP 请求层与后端进行数据通信，使用 Pinia 管理全局的用户认证状态，通过 Vue Router 实现页面路由控制和未登录访问拦截。前端项目使用 Vite 进行打包构建，Vite 利用浏览器原生的 ES 模块机制实现按需编译，在开发环境下的模块热替换速度显著快于传统的 Webpack 方案，有效缩短了前端开发的调试反馈周期。

## 2.4 基础设施与运维技术

Docker 是一种操作系统级别的虚拟化技术，它将应用程序及其运行所需的全部依赖打包为标准化的容器镜像，使得同一镜像可以在任何安装了 Docker 引擎的环境中以一致的方式运行。与传统虚拟机相比，Docker 容器共享宿主机的操作系统内核，启动速度快、资源占用低，更适合微服务场景下大量轻量级服务实例的部署需求。Docker Compose 则在此基础上提供了多容器编排能力，通过一个 YAML 格式的配置文件即可定义多个容器的镜像来源、端口映射、数据卷挂载和启动顺序等信息，执行单条命令便可完成全部基础服务的创建与启动[13]。本文使用 Docker Compose 统一编排 MySQL、Redis 和 Nacos 三个基础设施组件，消除了因开发者本地环境差异导致的配置不一致问题。

运维脚本和测试脚本分别以 Bash 和 Python 编写。Bash 脚本负责后端服务的顺序启动、健康探测和有序停止，支持根据 Nacos 是否可用在完整模式与直连模式之间自动切换。Python 脚本承担全链路回归测试和缓存基准测试任务，测试过程中通过标准 HTTP 请求与系统接口交互，执行完成后自动生成 Markdown 格式的结果报告。全部脚本遵循幂等原则设计，确保在不同时间点执行能够产生可比较、可复现的结果，为系统的持续验证提供稳定的基础条件。

## 2.5 本章小结

本章从后端技术、数据库技术、前端技术和基础设施与运维技术四个方面，阐述了本文系统所涉及的关键技术及其工作原理。这些技术的选型均围绕微服务架构下高校论坛系统的实际需求展开，具体的实现细节和设计决策将在后续章节中结合系统架构和功能模块进一步展开。
