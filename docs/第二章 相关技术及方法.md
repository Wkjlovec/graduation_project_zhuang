第二章 相关技术及方法

2.1 后端技术

本文系统的后端采用 Java 17 语言开发。Java 通过 JVM 虚拟机将编译产物在不同操作系统上一致运行，类型系统成熟、垃圾回收机制完善，长期以来是大型 Web 应用后端开发的主流选择。JDK 目前有 8、11、17 和 21 四个长期支持版本，本文选用 JDK 17，相较于 JDK 8 增加了密封类、模式匹配等语言特性，社区支持和依赖库兼容性也已趋于稳定。

Spring 框架的两项核心机制——控制反转（IoC）和面向切面编程（AOP）——奠定了 Java 企业级开发的基础。控制反转由容器接管对象的创建和依赖关系，调用方从容器获取所需对象而非自行实例化，模块之间的耦合度由此降低。面向切面编程将日志记录、权限校验等横切关注点抽取为独立模块统一维护，避免同一段逻辑在多处重复。Spring Boot 在此基础上进一步简化了项目搭建：自动装配机制根据引入的依赖完成配置，内嵌 Tomcat 容器使应用无须外部 Web 服务器即可独立运行[11]。本文全部后端服务均基于 Spring Boot 3 构建。

微服务架构的核心思路是将一个进程内的业务逻辑拆分为多个独立服务，每个服务拥有自己的数据存储和部署周期，服务之间通过网络接口通信[12]。拆分带来了独立扩缩和版本迭代的能力，代价则是需要额外的基础设施处理服务寻址、配置同步和流量管控。通信方式的选择对性能有直接影响：同步调用在低延迟场景中响应更快，异步消息在高负载下的可扩展性更优[18]。灵活性提升的同时容错问题也随之出现，断路器和故障转移机制是防止单个服务故障蔓延为全局瘫痪的关键手段[17]。API 网关则作为系统面向外部的唯一入口，集中承担请求路由、协议适配和安全控制[16]。本文使用的 Spring Cloud Alibaba 将上述治理能力整合为一套组件：Nacos 同时扮演服务注册中心和配置中心，各服务启动时向 Nacos 注册自身地址，调用方按服务名寻址，实例的增减对上游透明；Spring Cloud Gateway 基于路由规则分发请求并在过滤器链中嵌入令牌校验逻辑；OpenFeign 将跨服务的 HTTP 调用封装为接口声明，降低远程通信的编码成本。

用户认证采用 JWT（JSON Web Token）方案。JWT 将用户身份声明和过期时间编码为经数字签名的令牌字符串，服务端只需校验签名即可确认令牌真伪，不必在服务端集中保存会话状态，天然适配多实例部署的分布式环境。Spring Security 通过过滤器链对每次请求拦截校验，与网关层的令牌验证形成两道屏障。JWT 一经签发便无法主动撤销，本文额外使用 Redis 存储会话状态来支持登出和令牌失效。Aldea 等人在对微服务认证安全的研究中指出，基于令牌的无状态认证在多服务环境中工程优势明显，短生命周期令牌和最小权限原则能有效缩小攻击面[7]。

数据持久化选用 MySQL 8。关系型数据库在标准 SQL 查询、事务处理和索引优化上具有成熟支撑，适合存储用户账户、帖子内容、评论等结构明确且关联关系清晰的业务数据[15]。微服务架构的理想模型要求每个服务管理自身专属的数据域，以避免服务间产生隐式耦合[13]。本文出于毕业设计的规模考量，让认证服务、论坛服务和通知服务共享同一个 MySQL 实例，各服务通过不同数据表实现逻辑隔离，搜索服务和媒体服务不直接访问数据库。对象与数据表之间的映射通过 JPA 规范完成，底层由 Hibernate 引擎处理实体的生命周期管理和级联操作。缓存层采用 Redis 7，数据驻留在内存中，读写延迟通常在亚毫秒级别。帖子列表和详情等高频访问数据在 Redis 中设置了带过期时间的缓存，命中时直接返回结果，不再穿透到 MySQL；认证会话和登录失败计数等临时状态同样存储在 Redis 中，利用其原子操作特性保证并发场景下数据的一致性。

MySQL、Redis 和 Nacos 三个基础设施组件通过 Docker 容器化封装。Docker Compose 编排文件集中定义了各容器的镜像版本、端口映射、数据卷和启动依赖关系，一条命令即可在全新的环境中拉起全部服务，开发者本地环境差异带来的配置不一致问题由此消除[14]。容器化与微服务的结合也在安全中台等领域得到验证，通过分布式服务治理和容器编排实现业务模块的解耦与弹性部署[19]。运维脚本以 Bash 编写，按照 common-lib → auth → forum → notification → media → search → gateway 的顺序依次启动各服务并逐一探测健康端点，停止时按反向顺序有序关闭，脚本还支持在有 Nacos 的完整模式和无 Nacos 的直连模式之间自动切换。测试脚本以 Python 编写，负责全链路回归测试和缓存基准测试，执行结束后自动生成 Markdown 格式的结果报告。全部脚本遵循幂等原则，确保在不同时间点执行能够产生可比较的结果。

2.2 前端技术

Web 前端技术建立在 HTML、CSS 和 JavaScript 三项基础之上。HTML 负责页面的结构骨架，CSS 控制样式与布局，JavaScript 则赋予页面动态交互的能力，使开发者能够响应用户操作、处理数据并与后端服务器通信。随着前端工程复杂度的持续增长，直接操作浏览器 DOM 节点的开发方式越来越难以维护，由此催生了以组件化和数据驱动为核心理念的现代前端框架。

目前主流的前端框架主要有 Vue、React 和 Angular 三个。Angular 提供了一体化的解决方案，内置依赖注入、模板引擎和路由系统，适合大型团队的企业级项目开发，但由于采用 TypeScript 语言且概念体系庞大，学习曲线相对陡峭。React 和 Vue 都采用了组件化的开发思想，将完整页面拆分为若干功能独立的逻辑单元，同时通过虚拟 DOM 技术减少不必要的页面重绘以提升渲染效率。二者的主要区别在于数据响应机制：React 要求开发者通过显式调用 setState 或 Hook 函数触发视图更新，数据变化的时机完全由开发者控制；Vue 则采用依赖追踪的方式，在数据被修改时自动识别受影响的视图区域并触发重新渲染，无须开发者手动介入。Vue 还提供了双向数据绑定特性，能够在表单场景中实现数据与视图的实时同步，简化了交互逻辑的编写。考虑到 Vue 中文生态丰富、上手门槛较低，且官方配套了 Vite 构建工具、Vue Router 路由库和 Pinia 状态管理库等开箱即用的工具链，本文选用 Vue 3 作为前端框架[5]。

PC 端选用 Element Plus 作为组件库，该库基于 Vue 3 开发，提供了表格、表单、对话框、分页器等常用桌面端交互组件。移动端选用 Vant 组件库，针对触屏操作和小屏幕布局进行了专门优化，提供导航栏、单元格、浮动气泡等移动端常见交互模式。两端选择了不同的 UI 组件库，在数据通信层面则共享同一套 Axios 封装的 HTTP 请求层与后端交互，使用 Pinia 管理全局的用户认证状态，通过 Vue Router 实现页面路由控制和未登录访问拦截。前端项目使用 Vite 进行打包构建，Vite 利用浏览器原生的 ES 模块机制实现按需编译，在开发环境下的模块热替换速度显著快于传统的 Webpack 方案，有效缩短了前端开发的调试反馈周期。
