## 【P3 研究背景与意义】（1分30秒）

高校师生在课程答疑、资料分享中需要一个能持续沉淀信息的平台。即时通讯群里的内容几小时后就被覆盖，论坛的价值在于按主题把碎片化问答组织为可回查的知识。

但现有校园论坛有三个不足。架构上，水木清华、北大未名等早期 BBS 都是单体架构，前后端耦合，局部修改需全局重部署。终端上，CNNIC 报告显示移动端用户比例已达百分之九十九点六，但多数论坛只有 PC 端。测试上，功能验证依赖手工操作，缺少量化判据。

从研究现状看，Spring Cloud 微服务技术栈已经成熟，但在高校论坛场景中完整应用微服务并配套可复现测试的工程实践仍较少，这是本文的切入点。

本课题有五个创新点：微服务拆分实现松耦合、双端统一 API、JWT 加 Redis 认证闭环、缓存提升百分之八十以上、以及自动化测试覆盖全部接口。

→ 接下来介绍需求分析。

## 【P4 需求分析】（50秒）

功能需求分为八个模块：用户认证、帖子管理、评论互动、点赞去重、通知、搜索、媒体推荐和网关访问控制。

非功能需求五个维度：性能上缓存命中后毫秒级响应；安全上 JWT 双令牌加失败锁定；可扩展性上新模块注册 Nacos 即可接入；可维护性上 Docker 一键编排加自动回归；兼容性上支持 PC 和移动双端。

权限模型分为游客和注册用户。游客可浏览和搜索，注册用户可发帖、评论和点赞，编辑删除权限仅限内容作者。

→ 需求明确后，进入架构设计。

---

## 【P5 整体架构】（30秒）

系统分四层。用户接入层是 PC 端和移动端两个 Vue 3 应用。网关层由 Gateway 在 8080 端口统一接收请求，负责路由、JWT 验证和会话校验。业务服务层包含认证、论坛、通知、媒体和搜索五个独立服务。基础设施层由 MySQL、Redis 和 Nacos 通过 Docker Compose 编排。各服务通过 Nacos 按服务名寻址，实例增减对上游透明。

---

## 【P6 模块划分】（30秒）

后端共七个 Maven 子模块。common-lib 提供公共常量和统一响应封装。Gateway 负责路由和鉴权。auth 提供 6 个接口处理认证和会话。forum 是核心模块，10 个接口覆盖帖子、评论、点赞和缓存。notification 管理通知，search 通过 OpenFeign 跨服务调用实现搜索，media 提供静态推荐。系统共 22 个 RESTful API。

---

## 【P7 数据库设计】（30秒）

数据库 6 张表，满足第三范式。这里有个设计取舍：forum_post 表冗余存储了作者名和分区名，牺牲了范式严格性，换来帖子列表查询免 JOIN，配合 Redis 缓存降低数据库压力。post_like 表通过联合唯一约束从数据库层保障点赞去重。Redis 方面，会话和失败计数用于认证安全，帖子列表和详情分别设置了两分钟和五分钟的缓存过期时间。

## 【P8 认证与论坛核心实现】（1分钟）

这是本系统最核心的两段逻辑。

登录流程：先检查账户是否锁定，然后 BCrypt 比对密码，失败时 Redis 原子递增计数，达到五次锁定十分钟。成功后签发双令牌，Redis 写入会话。网关的 JwtRelayFilter 对每个请求执行签名验证、类型检查和 Redis 会话校验，通过后将用户身份透传给下游服务。

论坛业务：发帖时通过 CacheEvict 注解自动淘汰列表缓存。点赞采用双重去重——业务层先查再插，数据库联合唯一约束兜底，高并发下也能保证正确性。

其他难点包括前端 401 并发刷新用 pendingQueue 解决、缓存一致性用主动淘汰加短 TTL 兜底。

---

## 【P9 双端前端】（30秒）

两端各有六个页面，PC 端用 Element Plus 左右分栏布局，移动端用 Vant 纵向堆叠适配小屏幕。底层共享 API 调用层、Pinia 状态管理和路由守卫，UI 层各自独立。不同的 localStorage 前缀使双端可同域运行不冲突。

---

## 【P10 辅助服务与运维】（20秒）

通知、搜索和媒体三个辅助服务验证了微服务架构下的多服务协同。运维方面，Docker Compose 编排基础设施，Bash 脚本按顺序启停六个服务并逐一验活，支持有无 Nacos 的三种模式自动切换。

## 【P11 系统测试】（30秒）

功能测试覆盖 11 个场景和全部 22 个接口，通过率百分之百。正常流程五个场景，安全防护四个场景验证了无令牌拦截、越权拦截、并发点赞去重和登出失效，令牌生命周期两个场景。缓存测试显示响应从三十毫秒降至五毫秒，提升超百分之八十。并发负载在十到一百用户下保持稳定。

## 【P13 总结与展望】（1分钟）

三项成果：完成了微服务架构拆分和四层部署设计；实现了论坛核心业务和 PC 加移动双端前端；建立了覆盖全部接口的可复现测试体系。

四个不足：搜索基于内存遍历不支持分词；日志监控机制未建立；未做云环境生产级验证；单元测试覆盖度待提升。

未来方向：引入 Elasticsearch 全文搜索、ELK 日志栈加链路追踪、Kubernetes 集群部署、以及完善安全扫描和单元测试。
