## 第一部分：开场与题目（约 30 秒）

[PPT-1 封面]

各位老师好，我是 ____（姓名），学号 ____，指导老师是汪红松老师。我的毕业论文题目是《基于微服务架构的高校论坛系统设计与实现》。

一句话概括：针对高校论坛单体架构扩展性不足、缺乏移动端适配和自动化测试的问题，本文设计并实现了一套基于 Spring Cloud 的微服务论坛系统。

## 第二部分：研究背景与创新点（约 1 分钟）

[PPT-2 过渡页：01 研究背景与创新点]

[PPT-3 背景与创新点]

先说背景。高校师生在课程答疑、资料共享和校园通知中对在线交流有持续需求，论坛系统的价值在于将碎片化讨论按主题沉淀为可回查的内容。

但现有高校论坛存在三个痛点。第一，单体架构耦合——改一个模块就要全局重新部署，无法独立扩展。第二，缺乏移动端适配——CNNIC 报告显示手机上网比例已达 99.6%，只有 PC 端意味着大量用户体验很差。第三，手工测试无法量化——功能验证靠人工点击，缺少可复现的判定依据。

针对这三个问题，本文有三个创新点。一是将微服务架构完整落地，Nacos 注册发现、Gateway 网关鉴权、OpenFeign 服务间调用一整套用上了。二是设计了 JWT 加 Redis 的混合认证方案，令牌可以主动撤销，前端有并发队列的静默刷新机制。三是配套了三套 Python 自动化测试脚本——功能回归、缓存基准和并发负载，执行后自动生成报告。

## 第三部分：需求分析与目标（约 1 分钟）

[PPT-4 过渡页：02 需求分析与目标]

[PPT-5 需求与技术路线]

系统面向两种用户。游客可以浏览帖子列表、搜索内容和查看通知摘要，注册用户在此基础上可以发帖、评论、点赞和管理个人通知。编辑和删除权限仅限作者本人——系统没有区分学生和教师角色，这个简化的权限模型降低了复杂度，对高校论坛来说是合理的取舍。

功能上拆成八个模块：认证、帖子、评论、点赞、通知、搜索、媒体推荐和网关访问控制。

非功能性需求从五个维度给了指标：性能方面缓存命中毫秒级响应，安全方面 JWT 双令牌加登录失败锁定，可扩展方面新模块热接入 Nacos，可维护方面 Docker Compose 一键拉起，兼容方面 PC 和移动端双端适配。

技术路线是：需求分析定指标，架构设计做拆分，分服务编码实现，最后用自动化脚本验证。

## 第四部分：系统设计与架构（约 2 分钟）

[PPT-6 过渡页：03 系统设计与架构]

[PPT-7 四层架构]

系统按部署职责分四层。最外层用户接入层，PC 端跑在 5173 端口、移动端跑在 5174 端口。所有请求汇入网关层，Spring Cloud Gateway 在 8080 端口做路由转发和 JWT 鉴权。网关之后是业务服务层，五个微服务——认证、论坛、通知、媒体、搜索，全部注册到 Nacos。最底层基础设施层，MySQL、Redis 和 Nacos 三个组件通过 Docker Compose 编排。

为什么选微服务？因为单体架构改一个模块就得全局重新部署，微服务让各服务独立开发和部署。三个服务共享一个 MySQL 实例是务实的折中，当前业务体量不需要拆库，未来可以按表拆分。

[PPT-8 数据库与接口]

数据库有六张业务表，五个实体通过七条关系连接，其中点赞是用户和帖子之间的多对多关系，通过 post_like 关联表拆解。forum_post 表冗余存储了作者名和分区名，严格来说违反第三范式，但省去了三表 JOIN，帖子列表查询在单表上就能完成。

接口方面，全部 22 个 API 遵循 RESTful 风格。认证服务 6 个，论坛服务 10 个，通知、搜索和媒体加起来 6 个。网关维护公开接口白名单，白名单外的写操作接口要求携带有效的 JWT 令牌。

技术选型上，后端选 Spring Boot 3 是因为 Spring Cloud Alibaba 提供 Nacos、Gateway、OpenFeign 的开箱支持，Django 在这方面缺少原生集成。

## 第五部分：实现与关键技术（约 2 分钟）

[PPT-9 过渡页：04 实现与关键技术]

[PPT-10 核心实现]

认证模块的核心是 JWT 加 Redis 的混合方案。为什么不用纯 JWT？因为 JWT 一经签发无法主动撤销，登出时令牌仍然有效。加了 Redis 之后，登录时把 sessionId 和 userId 的映射写入 Redis，网关校验时不仅验 JWT 签名还查 Redis 会话是否存在，登出只需删除 Redis 记录，令牌立即失效。

缓存策略是读写双路径。读操作先查 Redis，命中直接返回；未命中查 MySQL 并回写 Redis。帖子列表缓存 2 分钟、详情缓存 5 分钟。写操作通过 @CacheEvict 注解自动淘汰缓存。实际效果是冷启动 59 毫秒降到热缓存 15 毫秒，提升 74.66%。

并发点赞用了双重去重。应用层先查 exists 做快速判断，数据库层用联合唯一约束兜底。两个请求同时通过应用层检查，第二次插入也会被数据库拒绝，零锁开销。

前端的静默刷新机制处理 401 状态码时，如果已有刷新请求在途，后续请求会加入等待队列而不是重复触发刷新，刷新成功后统一释放队列。

[PPT-11 过渡页：05 测试结果]

[PPT-12 测试结果]

测试方面简要汇报。功能测试覆盖 11 个端到端场景和全部 22 个接口，通过率 100%。这 11 个场景中有 4 个专门验证异常路径——无令牌拦截、越权操作拦截、并发重复点赞和登出后会话失效。

缓存基准测试：冷启动 59 毫秒，热缓存平均 15 毫秒，P95 为 26 毫秒，性能提升 74.66%。

并发负载测试分五档，从 10 到 1000 并发用户。1000 并发下 QPS 仍保持 1450 以上，成功率 99.99%。所有测试在本地单机完成，与生产环境存在差异，这一点论文中已做说明。

## 第六部分：总结与展望（约 1 分钟）

[PPT-13 过渡页：06 总结与展望]

[PPT-14 总结与展望]

最后总结。三点核心结论：第一，微服务架构可以在高校论坛这种中等规模的 Web 应用中实际落地，五个服务独立部署和扩缩是可行的。第二，PC 端和移动端共享同一套后端 API，用户切换设备时操作体验保持一致。第三，三套 Python 测试脚本让功能回归、缓存验证和并发压测都可以自动执行和复现。

两个不足：一是搜索服务用的是内存过滤方案，帖子量增长后效率会下降，后续可引入 Elasticsearch。二是全部测试在本地完成，没有在云环境下做生产级验证，后续可部署到 Kubernetes 集群。
