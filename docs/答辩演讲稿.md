## 第一部分：开场与题目（约 30 秒）

[PPT 第 1 页：封面]

各位老师好，我是 ____（姓名），学号 ____，指导老师是汪红松老师。我的毕业论文题目是《基于微服务架构的高校论坛系统设计与实现》。

简单来说，这个课题要解决的问题是：现有高校论坛大多采用单体架构，功能扩展困难、缺乏移动端适配、测试依赖手工操作。本文的工作是用 Spring Cloud 微服务架构重新设计和实现一套论坛系统，同时配套 PC 端和移动端两套前端以及可自动执行的测试脚本。

---

## 第二部分：研究背景与意义（约 1 分钟）

[PPT 第 2 页：背景与创新点]

先说背景。高校师生在课程答疑、资料共享、校园通知等场景中对在线交流有持续需求。论坛系统的价值在于按主题沉淀内容，比即时通讯群聊更适合长期保存和回查。

但实际情况是，多数校园论坛存在三个明显的短板。第一，单体架构把所有模块打包在一个工程里，改一个地方就要全局重新部署。第二，只有 PC 端页面，CNNIC 报告显示手机上网比例已达 99.6%，没有移动端意味着相当一部分用户体验很差。第三，功能验证基本靠手工点击，没有可重复执行的自动化测试，交付质量缺少量化依据。

多数高校论坛在工程上的应对策略偏向保守——要么用更高配置的服务器硬扛并发，要么只在前端加一套响应式布局，但架构层面的耦合和测试层面的空白始终没有触及。

本文针对这三个问题，在架构解耦、双端适配、安全认证和质量保障四个层面做了改进。具体的创新点有三个：一是将微服务架构完整应用于高校论坛，而不是只做前后端分离；二是设计了 JWT 加 Redis 的混合认证方案，支持令牌的主动撤销和前端的静默刷新；三是配套了三套 Python 自动化测试脚本，覆盖功能回归、缓存基准和并发负载，执行后自动生成报告。

## 第三部分：需求分析与目标（约 1 分 30 秒）

[PPT 第 3 页：需求与技术路线]

系统面向两种用户：游客和注册用户。游客可以浏览帖子列表、搜索内容、查看通知摘要和媒体推荐，这些都是公开操作。注册用户在此基础上可以发帖、评论、点赞、查看个人通知，帖子和评论的编辑删除权限仅限作者本人。

功能上拆分成八个模块：用户认证、帖子管理、评论互动、点赞、通知服务、搜索服务、媒体推荐和网关访问控制。每个模块都建立了用例模型，include 和 extend 关系在论文第三章有详细描述。

非功能性需求从五个维度给出了指标。性能方面，缓存命中时毫秒级响应，未命中时控制在 500 毫秒以内。安全方面，JWT 双令牌加 Redis 会话管理，登录失败 5 次锁定 10 分钟。可扩展性方面，新增模块只需注册到 Nacos 并配置网关路由。可维护性方面，Docker Compose 一条命令拉起基础设施，Bash 脚本按顺序启停服务。兼容性方面，PC 端和移动端分别适配桌面浏览器和触屏设备。

技术路线是：需求分析确定功能和非功能指标，然后做架构设计和数据库设计，接着分服务编码实现，最后用自动化脚本进行测试验证。

## 第四部分：系统设计与架构（约 2 分钟）

[PPT 第 4 页：系统架构]

系统按部署职责分为四个层次。最外层是用户接入层，PC 端跑在 5173 端口、移动端跑在 5174 端口。所有请求汇入网关层，Spring Cloud Gateway 在 8080 端口统一接收，完成路由转发和 JWT 鉴权。网关之后是业务服务层，五个微服务分别是认证 8081、论坛 8082、通知 8083、媒体 8084、搜索 8085，外加一个 common-lib 公共库被所有服务共同依赖。最底层是基础设施层，MySQL、Redis 和 Nacos 三个组件通过 Docker Compose 编排。

服务间的调用关系很简洁，只有搜索服务通过 OpenFeign 调用论坛服务获取帖子数据，其余服务之间没有直接调用。

[PPT 第 5 页：数据库与接口]

数据库设计包含六张业务表，分属认证、论坛和通知三个服务。搜索和媒体服务不直接访问数据库。ER 图采用 Chen 记法绘制，五个实体通过七条关系连接，其中点赞是用户与帖子之间的多对多关系，通过 post_like 关联表实现。

接口方面，全部 22 个 API 遵循 RESTful 风格，GET 和 POST 对应读取与创建，PUT 和 DELETE 对应更新与删除。网关维护了一份公开接口白名单，白名单外的接口要求携带有效的 JWT 令牌。

技术选型上，后端选 Spring Boot 3 是因为 Spring Cloud Alibaba 提供了 Nacos、Gateway、OpenFeign 等微服务组件的开箱即用支持；前端选 Vue 3 是因为依赖追踪的响应式机制和双向数据绑定降低了开发成本，中文社区生态也最为成熟。

## 第五部分：实现与关键技术（约 2 分钟）

[PPT 第 6 页：认证与缓存]

认证模块的核心流程是：用户提交用户名和密码，系统先检查账户是否被锁定，未锁定则校验凭据。校验成功后生成一个随机 sessionId，签发访问令牌和刷新令牌，同时将 sessionId 与 userId 的映射写入 Redis。登出时直接删除 Redis 中的会话记录，令牌随即失效。

网关的 JwtRelayFilter 对每个请求做七级检查：OPTIONS 预检放行、GET 白名单放行、认证接口放行，不满足这三条的进入鉴权流程——提取 Bearer 令牌、验证 JWT 签名、检查声明字段、查询 Redis 会话。全部通过后透传用户身份到下游服务。

前端的响应拦截器处理 401 状态码时，会检查是否有刷新令牌、是否已有刷新请求在途。如果在途就把当前请求加入等待队列，避免多次触发刷新。刷新成功后释放队列中所有请求携带新令牌重试。

缓存策略是读路径优先命中 Redis，未命中再查 MySQL 并回写缓存。帖子列表缓存 2 分钟、详情缓存 5 分钟。写操作通过 @CacheEvict 注解自动淘汰相关缓存。

在并发点赞场景下，应用层先查 exists，数据库层用联合唯一约束兜底。即使两个请求同时通过应用层检查，第二次插入也会被数据库拒绝，业务代码捕获异常后返回提示，实现双重去重。

[PPT 第 7 页：测试结果]

测试方面简要汇报。功能测试覆盖 11 个端到端场景和全部 22 个接口，通过率 100%。其中包括 4 个异常路径：无令牌拦截、越权操作拦截、并发重复点赞和登出后会话失效。

缓存基准测试显示，冷启动响应时间 59 毫秒，热缓存平均降至 15 毫秒，性能提升 74.66%。

并发负载测试分五档：10、50、100、500 和 1000 并发用户。1000 并发下 QPS 仍保持在 1450 以上，成功率 99.99%。测试全部在本地单机完成，与生产环境存在差异，这一点在论文中已做说明。

## 第六部分：总结与展望（约 1 分钟）

[PPT 第 8 页：总结与致谢]

最后做个总结。本文的主要工作有三点：第一，将 Spring Cloud 微服务架构完整应用于高校论坛系统，五个业务服务可独立部署和扩缩，验证了这一架构在中等规模 Web 应用中的可行性。第二，PC 端和移动端共享同一套后端 API，两端各自适配终端特性，用户切换设备时操作体验保持一致。第三，三套 Python 测试脚本覆盖了功能回归、缓存基准和并发负载，测试过程可复现、结果可量化。

系统目前存在两个不足。一是搜索服务采用内存过滤，帖子量增长后检索效率会下降，后续可引入 Elasticsearch。二是全部测试在本地完成，没有在云环境下做过生产级验证，后续可部署至 Kubernetes 集群进行压力测试。
