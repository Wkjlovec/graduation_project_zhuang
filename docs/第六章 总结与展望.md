第六章 总结与展望

6.1 总结

高校师生在日常学习和科研活动中产生了大量的在线交流需求，涵盖课程答疑、学习资料共享、经验复盘和校园活动通知等多个场景。论坛系统凭借其话题聚合与历史沉淀的特性，成为承载上述需求的重要平台。然而，传统的校园论坛大多采用单体架构，前后端代码耦合在同一工程中，在功能迭代和用户增长的压力下，系统在并发处理、独立部署和版本演进方面的局限日渐突出。本文正是针对这些问题，设计并实现了一个基于微服务架构的高校论坛系统。系统采用前后端分离的 B/S 架构进行开发，后端基于 Java 17 语言和 Spring Boot 3 框架构建，借助 Spring Cloud Alibaba 提供的 Nacos 注册中心和 Spring Cloud Gateway 网关组件，将业务逻辑拆分为认证、论坛、通知、搜索和媒体五个可独立部署的微服务，各服务通过 RESTful 接口通信，数据持久化由 MySQL 8 完成，缓存与会话管理由 Redis 7 承担，用户认证采用 JWT 结合 Spring Security 的无状态方案。前端基于 Vue 3 开发，分别使用 Element Plus 和 Vant 两套组件库构建 PC 端与移动端应用，两端共享同一套 Axios 封装的请求层和 Pinia 状态管理逻辑，通过 Vite 进行打包构建。系统配套了 Docker Compose 基础设施编排、Bash 一键启停与健康检查脚本，以及 Python 编写的全链路回归测试和缓存基准测试脚本，形成了从开发到验证的完整工程闭环。论文主要完成以下工作：

（1）对高校论坛的使用场景进行了需求调研与分析，将系统用户划分为游客和注册用户两种角色，梳理了用户认证、帖子管理、评论互动、点赞、通知服务、搜索服务、媒体推荐和网关访问控制八个功能模块的用例关系，绘制了各模块的用例图和系统整体用例图，并从性能、安全、可扩展性、可维护性和兼容性五个维度明确了非功能性需求的量化指标。

（2）完成了系统的架构设计与数据库设计。按照部署职责将系统划分为用户接入层、网关层、业务服务层和基础设施层四个层次，绘制了部署图和网关路由转发流程图。数据库设计包含 user_account、forum_section、forum_post、post_comment、post_like 和 notification_message 共 6 张业务表，表结构满足第三范式，其中 forum_post 表的 author_name 和 section_name 两个冗余字段属于以空间换时间的反范式设计，使帖子列表查询可在单表上完成而无须多表 JOIN。通过 E-R 图展示了各实体之间的关联关系，并列出了系统实际使用的全部索引清单。

（3）实现了认证与会话管理、论坛核心业务和安全访问控制三个关键模块。认证模块基于 JWT 签发访问令牌和刷新令牌，配合 Redis 存储会话状态，形成了注册、登录、令牌刷新与登出的完整闭环，并通过 LoginAttemptService 实现了连续失败锁定机制。网关层的 JwtRelayFilter 对每个请求执行 JWT 签名校验和 Redis 会话检查的双重验证，区分公开接口白名单和受保护接口，校验通过后将用户身份信息透传至下游服务。论坛核心业务实现了帖子的发布、编辑、删除和按分区分页查询，评论通过 parent_comment_id 字段实现二级回复结构，点赞通过联合唯一约束和应用层双重校验实现去重。Redis 缓存策略对帖子列表和详情分别设置了 2 分钟和 5 分钟的过期时间，写操作执行后通过 Spring Cache 注解淘汰相关缓存条目。

（4）完成了前端 PC 端和移动端两个应用的开发。PC 端基于 Element Plus 组件库构建，采用左侧主内容区与右侧边栏的布局；移动端基于 Vant 组件库构建，采用纵向堆叠布局适配触屏操作。两端各包含登录页、帖子列表页、帖子详情页、发帖页、通知页和个人中心页六个页面组件，共享统一的 API 调用层和路由守卫逻辑。前端响应拦截器实现了 401 状态码的静默刷新机制，通过 pendingQueue 等待队列防止多个并发请求同时触发令牌刷新导致的竞争问题。

（5）搭建了容器化基础设施与自动化测试体系。MySQL 8.4、Redis 7.4 和 Nacos 2.3.2 通过 Docker Compose 编排部署，Bash 脚本按照依赖顺序启动各服务并逐一探测健康端点，停止时按反向顺序有序关闭。Python 回归测试脚本覆盖了 11 个端到端业务场景和全部 22 个 API 接口，测试通过率为 100%。缓存基准测试脚本对比了冷启动与热缓存下的接口响应时间，并发负载测试在 10 至 100 并发用户规模下验证了系统的基本承载能力，为缓存策略的有效性提供了可量化的数据支撑。

6.2 存在的问题与展望

受开发周期和个人能力的限制，系统在功能完整性、性能优化和测试覆盖等环节仍存在改进空间。以下列举目前的不足及后续改进方向。

（1）搜索服务的检索能力有待增强。当前的搜索服务采用内存中逐条遍历和字符串 contains 匹配的方式实现关键词过滤，这种方案在数据量较小时能够满足需求，但随着帖子数量增长，匹配效率将显著下降，且不支持分词、模糊匹配和相关度排序等高级检索特性。后续可考虑引入 Elasticsearch 全文搜索引擎，通过倒排索引和分词分析器提升检索精度和响应速度。

（2）系统尚未实现完善的日志采集与监控机制。当前各微服务仅依赖 Spring Boot 默认的控制台日志输出，缺少统一的日志收集、存储和查询平台。在生产环境中，分布式系统的故障定位往往需要跨多个服务节点关联日志信息。后续可引入 ELK（Elasticsearch + Logstash + Kibana）日志采集栈，结合链路追踪工具实现请求全链路的可观测性。

（3）测试覆盖范围有待进一步扩展。当前测试工作以功能回归测试和缓存性能测试为主，尚未进行基于专业工具的安全漏洞扫描（如 SQL 注入、XSS 攻击检测）、多浏览器和多设备的兼容性测试，以及基于 JUnit 的后端单元测试覆盖率统计。后续可引入 OWASP ZAP 进行自动化安全扫描，使用 Selenium 或 Playwright 实现前端跨浏览器自动化测试，并建立持续集成流程以保证每次代码变更的测试覆盖率不低于预设阈值。

（4）系统的部署架构尚未进行云环境下的生产级验证。当前测试和开发均在本地环境完成，单机部署无法体现微服务架构在水平扩展和负载均衡方面的优势。后续可将系统部署至云服务器，借助 Kubernetes 实现各服务的弹性伸缩和滚动更新，并接入 CDN 加速前端静态资源的分发，以评估系统在真实网络环境和更大用户规模下的表现。

（5）前端的交互体验和功能丰富度有进一步提升的空间。当前系统提供了论坛基本的发帖、评论、点赞和搜索功能，但在内容富文本编辑、图片上传、用户头像、个人主页、私信、收藏、热门排行等方面尚未涉及。后续可根据实际用户反馈逐步引入这些功能模块，进一步提升论坛的社区活跃度和用户粘性。
