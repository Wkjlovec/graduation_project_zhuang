第六章 总结与展望

6.1 总结

高校师生在日常学习和科研活动中产生了大量的在线交流需求，涵盖课程答疑、学习资料共享、经验复盘和校园活动通知等多个场景。论坛系统凭借其话题聚合与历史沉淀的特性，成为承载上述需求的重要平台。日益丰富的交互形式和持续增长的用户规模对论坛系统的架构弹性、跨终端适配能力和质量保障手段提出了更高的要求，传统单体架构下前后端耦合、独立部署困难、缺乏移动端支持等问题使得既有校园论坛难以满足这些要求。本文正是基于此背景，设计并实现了一个基于微服务架构的高校论坛系统。本文首先从研究背景和国内外现状出发，分析了现有校园论坛在架构、适配和测试方面的不足，明确了本文的研究切入点。然后对系统进行需求分析，将用户划分为游客和注册用户两种角色，梳理了用户认证、帖子管理、评论互动、点赞、通知、搜索、媒体推荐和网关访问控制八个功能模块的用例关系，并从性能、安全、可扩展性、可维护性和兼容性五个维度给出了非功能性需求的量化指标。在系统设计部分，按照部署职责将系统划分为用户接入层、网关层、业务服务层和基础设施层四个层次，后端基于 Java 17 和 Spring Boot 3 框架构建，借助 Spring Cloud Alibaba 提供的 Nacos 注册中心和 Spring Cloud Gateway 网关组件将业务逻辑拆分为五个可独立部署的微服务，数据持久化由 MySQL 8 完成，缓存与会话管理由 Redis 7 承担；数据库设计包含 6 张业务表，通过 E-R 图展示实体关联，并列出了全部索引清单；安全设计围绕 JWT 令牌签发与校验、网关双重鉴权和登录失败锁定三条线路展开。在系统实现部分，认证模块实现了注册、登录、令牌刷新与登出的完整闭环，论坛核心业务实现了帖子的发布、编辑、删除和按分区分页查询，评论支持二级回复结构，点赞通过联合唯一约束和应用层双重校验实现去重，Redis 缓存策略对帖子列表和详情分别设置了不同的过期时间并由写操作自动淘汰。前端基于 Vue 3 开发，分别使用 Element Plus 和 Vant 两套组件库构建 PC 端与移动端应用，两端共享统一的 Axios 请求层和 Pinia 状态管理逻辑，响应拦截器实现了令牌静默刷新机制。在运维与测试部分，MySQL、Redis 和 Nacos 通过 Docker Compose 编排部署，Bash 脚本负责后端服务的顺序启动与有序停止，Python 回归测试脚本覆盖了 11 个端到端业务场景和全部 22 个 API 接口，缓存基准测试和并发负载测试分别验证了缓存策略的有效性和系统的基本承载能力，测试通过率为 100%。

6.2 存在的问题与展望

受开发周期和个人能力的限制，系统目前仍存在以下不足，有待后续改进。

（1）搜索服务的检索能力有待增强。当前的搜索服务采用内存中逐条遍历和字符串 contains 匹配的方式实现关键词过滤，这种方案在数据量较小时能够满足需求，但随着帖子数量增长，匹配效率将显著下降，且不支持分词、模糊匹配和相关度排序等高级检索特性。已有研究在微服务架构下设计了面向多源异构数据的实时处理平台，通过缓存机制与索引优化将端到端处理延迟控制在 2 秒以内[15]。后续可借鉴类似思路引入 Elasticsearch 全文搜索引擎，通过倒排索引和分词分析器提升检索精度和响应速度。

（2）系统尚未实现完善的日志采集与监控机制。当前各微服务仅依赖 Spring Boot 默认的控制台日志输出，缺少统一的日志收集、存储和查询平台。在生产环境中，分布式系统的故障定位往往需要跨多个服务节点关联日志信息。已有研究指出，通过集成 Hystrix 等监控组件可以实时追踪服务状态并在故障率超过阈值时自动触发熔断，有效防止故障蔓延[17]。后续可在此基础上引入 ELK（Elasticsearch + Logstash + Kibana）日志采集栈，结合链路追踪工具实现请求全链路的可观测性。

（3）系统的部署架构尚未进行云环境下的生产级验证。当前测试和开发均在本地环境完成，单机部署无法体现微服务架构在水平扩展和负载均衡方面的优势。已有研究在智慧校园系统中采用容器化微服务架构并部署至 Kubernetes 集群，讨论了在实际应用中面临的性能瓶颈、服务间通信和数据一致性等挑战[14]。后续可借鉴上述实践将系统部署至云服务器，实现各服务的弹性伸缩和滚动更新，并接入 CDN 加速前端静态资源的分发，以评估系统在真实网络环境和更大用户规模下的表现。
