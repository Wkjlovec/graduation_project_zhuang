第四章 系统详细设计与实现

## 4.1 系统架构设计

系统整体采用微服务架构，按照部署职责可划分为四个层次：用户接入层、网关层、业务服务层和基础设施层。用户接入层包括 PC 端和移动端两个前端应用，分别运行在 5173 和 5174 端口。网关层由 Spring Cloud Gateway 承担，监听 8080 端口，是所有外部请求进入后端的唯一通道。业务服务层包含认证服务（8081）、论坛服务（8082）、通知服务（8083）、媒体服务（8084）和搜索服务（8085）五个独立进程，以及被各服务依赖的 common-lib 公共库。基础设施层由 Docker Compose 编排的三个容器组成：MySQL 8.4（3306）、Redis 7.4（6379）和 Nacos 2.3.2（8848）。系统部署图如图 4-1 所示。

图 4-1 系统部署图

各层之间的通信关系如下。PC 端和移动端通过 HTTP 协议向网关的 8080 端口发送请求。网关根据配置文件中定义的六条路由规则，按请求路径前缀将请求分发至对应的后端服务。具体路由映射关系为：`/api/auth/**` 和 `/api/users/**` 路由至认证服务，`/api/forum/**` 路由至论坛服务，`/api/notifications/**` 路由至通知服务，`/api/media/**` 路由至媒体服务，`/api/search/**` 路由至搜索服务。各路由均配置了 StripPrefix 过滤器，在转发前去除 URL 中的网关前缀层级，使下游服务接收到的请求路径与其自身的 Controller 映射一致。路由目标地址采用 `lb://服务名` 格式，由 Nacos 注册中心解析服务名到实际的 IP 和端口，实现基于服务发现的负载均衡。网关路由转发流程如图 4-2 所示。

图 4-2 网关路由转发流程图

在业务服务层内部，五个服务和网关在启动时均向 Nacos 注册自身地址和元信息，Nacos 同时作为配置中心为各服务提供运行时参数。服务之间的调用关系较为简单：搜索服务通过 OpenFeign 声明式客户端调用论坛服务的帖子列表接口获取数据，其余服务之间不存在直接调用。当系统运行在无 Nacos 的直连模式下，网关的路由目标地址和搜索服务的 Feign URL 均可通过环境变量指定为固定的 localhost 地址，绕过服务发现机制。

在数据访问方面，认证服务、论坛服务和通知服务共享同一个 MySQL 数据库实例（数据库名 forum_system），各服务通过不同的数据表实现逻辑隔离。认证服务和论坛服务同时访问 Redis，前者用于存储会话状态和登录失败计数，后者用于缓存帖子列表和详情数据。网关也连接 Redis 用于在鉴权过程中查询会话有效性。搜索服务和媒体服务不直接访问任何数据库或缓存。
