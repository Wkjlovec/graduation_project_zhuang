第四章 系统详细设计与实现

4.1 系统架构设计

第三章的需求分析表明，系统需要支持各业务模块的独立开发与部署、PC 端和移动端的双端适配，以及可复现的自动化测试。单体架构下所有模块打包在同一进程中，任何局部改动都要全局重新部署，难以满足上述要求。微服务架构将业务逻辑拆分为多个独立服务，每个服务拥有自己的部署周期和技术栈选择空间，与本系统的需求契合度更高。基于这一判断，系统整体采用微服务架构，按部署职责划分为四个层次。最外层是用户接入层，PC 端和移动端两个前端应用分别运行在 5173 和 5174 端口，用户通过浏览器访问。所有前端请求汇入网关层，由 Spring Cloud Gateway 在 8080 端口统一接收。网关之后是业务服务层，包含认证服务（8081）、论坛服务（8082）、通知服务（8083）、媒体服务（8084）和搜索服务（8085）五个独立进程，以及被各服务共同依赖的 common-lib 公共库。最底层是基础设施层，MySQL 8.4（3306）、Redis 7.4（6379）和 Nacos 2.3.2（8848）三个组件通过 Docker Compose 编排部署。系统部署图如图 4-1 所示。

图 4-1 系统部署图

网关收到请求后，按配置文件中定义的六条路由规则进行分发。`/api/auth/**` 和 `/api/users/**` 转发至认证服务，`/api/forum/**` 转发至论坛服务，`/api/notifications/**` 转发至通知服务，`/api/media/**` 转发至媒体服务，`/api/search/**` 转发至搜索服务。每条路由均配置了 StripPrefix 过滤器，在转发前去除 URL 中的网关前缀层级，保证下游服务接收到的请求路径与其 Controller 映射一致。路由目标地址写作 `lb://服务名`，由 Nacos 注册中心将服务名解析为实际的 IP 和端口，实现基于服务发现的负载均衡。网关路由转发流程如图 4-2 所示。

图 4-2 网关路由转发流程图

五个业务服务和网关在启动时均向 Nacos 注册自身地址和元信息，Nacos 同时充当配置中心，为各服务提供运行时参数。服务间的调用关系比较简洁：只有搜索服务通过 OpenFeign 声明式客户端调用论坛服务的帖子列表接口获取数据，其余服务之间不存在直接调用。如果运行环境中没有 Nacos，网关的路由目标和搜索服务的 Feign URL 可通过环境变量指向固定的 localhost 地址，系统在直连模式下依然能够正常启动。

微服务架构的理想模型要求每个服务管理自己专属的数据库，以避免服务间产生隐式耦合。本系统当前的业务模块数量和数据量尚未达到必须拆库的程度，认证服务、论坛服务和通知服务共享同一个 MySQL 实例（数据库名 forum_system），各服务通过不同数据表实现逻辑隔离。这一折中方案在当前的业务体量下不会引发数据冲突，未来如果服务规模增长，可以按表拆分为独立数据库而不改动业务代码。Redis 被三方使用：认证服务存储会话状态和登录失败计数，论坛服务缓存帖子列表和详情数据，网关在鉴权过程中查询会话是否有效。搜索服务和媒体服务不直接访问任何数据库或缓存，前者的数据全部来源于对论坛服务的 Feign 调用，后者的推荐数据预置在内存中。

上述架构在松耦合、独立部署和环境适配三个维度上回应了第三章提出的非功能性需求：各服务通过 Nacos 按服务名寻址，实例增减对上游透明，满足可扩展性要求；Docker Compose 和环境变量注入使同一套代码可在开发、测试和部署三个阶段使用不同配置，满足可维护性要求；网关的 CORS 配置和双端前端共用同一组 API，满足兼容性要求。

4.2 功能模块设计

后端代码按 Maven 多模块组织，共包含七个子模块。common-lib 是公共库，不独立运行；其余六个模块——gateway-service、auth-user-service、forum-service、notification-service、search-service 和 media-service——各自作为独立的 Spring Boot 应用部署。各模块的职责边界、对外接口以及相互依赖关系如图 4-3 所示。

图 4-3 系统功能模块结构图

common-lib 被其余六个模块共同依赖，其中定义了统一的接口响应封装类 ApiResponse 和全局常量类 ServiceConstants。后者集中管理 JWT 令牌类型标识、HTTP 请求头名称（Authorization、X-User-Id、X-Username）、Bearer 前缀和 Redis 键前缀等跨服务共享的常量值。将这些常量收敛到一个模块而非各服务各自硬编码，可以从源头上消除因拼写差异或值不一致而导致的联调故障。

gateway-service 是系统的统一入口，对外暴露 8080 端口。除了 4.1 节所述的路由转发职责外，该模块还承载了全局的 JWT 鉴权逻辑，通过 JwtRelayFilter 拦截所有请求，区分公开接口和受保护接口并完成令牌校验与用户身份透传。网关本身不包含任何业务逻辑。

auth-user-service 负责用户身份管理，对外提供六个接口：注册（POST /auth/register）、登录（POST /auth/login）、令牌刷新（POST /auth/refresh）、登出（POST /auth/logout）、查看个人信息（GET /users/me）以及健康探测（GET /users/ping）。该服务内部包含 JwtService 负责令牌签发与解析、AuthSessionService 负责 Redis 会话管理、LoginAttemptService 负责登录失败计数与锁定，以及 Spring Security 配置的内部过滤器链。

forum-service 是系统的核心业务模块，按资源类型可细分为三组接口。帖子接口覆盖列表查询（GET /posts，支持 page、size、sectionId 参数）、发布（POST /posts）、编辑（PUT /posts/{postId}）、删除（DELETE /posts/{postId}）、详情查询（GET /posts/{postId}）和点赞（POST /posts/{postId}/like）共六项操作。评论接口包括发表评论（POST /posts/{postId}/comments）、编辑评论（PUT /posts/{postId}/comments/{commentId}）和删除评论（DELETE /posts/{postId}/comments/{commentId}）三项操作，评论作为帖子的子资源挂在帖子路径下。分区列表由独立的 SectionController 提供（GET /sections），数据来源于启动时的种子初始化。该服务内部配置了 Redis 缓存策略，帖子列表和详情的缓存分别设置了不同的过期时间。

notification-service 提供三个接口：通知首页摘要（GET /notifications/home，返回最新公告和未读计数）、个人通知列表（GET /notifications/my）和标记已读（POST /notifications/{notificationId}/read）。首页摘要接口支持匿名访问，当请求头中未携带用户 ID 时返回未读数为零。该服务在启动时通过 NotificationDataInitializer 初始化若干条系统公告作为演示数据。

search-service 提供一个搜索接口（GET /search/posts，支持 keyword、sectionId、page、size 参数）。该服务内部通过 OpenFeign 客户端 ForumClient 调用论坛服务的帖子列表接口获取原始数据，然后在内存中按关键词对标题、内容摘要和作者名称进行过滤匹配后返回结果。搜索服务不拥有自己的数据库，所有数据均来源于论坛服务。

media-service 提供两个接口：媒体首页（GET /media/home）和推荐列表（GET /media/recommendations，支持 type 参数筛选音乐或书籍）。推荐数据在 MediaRecommendationService 中以静态列表方式预置于内存，不依赖外部数据库。该服务是系统中最轻量的模块，不访问 MySQL 和 Redis。

上述全部接口遵循 RESTful 风格设计：GET 方法用于资源的读取和查询，POST 方法用于资源的创建，PUT 方法用于资源的更新，DELETE 方法用于资源的删除。路径以资源名称为中心组织，子资源嵌套在父资源路径下（如评论接口挂在 /posts/{postId}/comments 下），路径中的动态参数用花括号标识。表 4-1 汇总了系统全部 22 个 API 接口的方法、路径和认证要求。

4.3 数据库设计

系统的业务数据全部存储在 MySQL 的 forum_system 库中，表结构由 JPA 的 ddl-auto:update 机制在各服务首次启动时自动创建，共包含 6 张业务表，分属认证服务、论坛服务和通知服务三个模块。各实体之间的关联关系如图 4-4 所示。图中用实线标注了四条核心业务关联：user_account 与 forum_post 之间通过 author_id 构成一对多，forum_section 与 forum_post 之间通过 section_id 构成一对多，forum_post 与 post_comment、post_like 之间分别通过 post_id 构成一对多。虚线标注的是 user_id 关联和自引用：post_comment、post_like 和 notification_message 三张表的 user_id 字段均指向 user_account，post_comment 的 parent_comment_id 自关联形成二级回复结构。

6 张表的非主键字段均完全依赖于主键，不存在传递依赖，满足第三范式（3NF）。需要单独说明的是 forum_post 表中的 author_name 和 section_name 两个字段。它们分别冗余存储了 user_account.username 和 forum_section.name 的值，严格按范式应当移除并在查询时通过 JOIN 获取。保留它们的原因很直接：帖子列表是系统中访问频率最高的接口，冗余存储使得列表查询在单表上即可完成，省去了三表 JOIN 的开销，配合 Redis 缓存后数据库压力进一步降低。代价是当用户修改昵称或分区重命名时，已有帖子中的冗余字段不会自动同步，需要额外的更新逻辑来保证一致性。在当前系统中昵称和分区名修改的频率很低，这一取舍是可接受的。

索引策略直接影响查询性能，表 4-2 列出了系统当前实际使用的全部索引。各表的 id 列为自增主键，MySQL 自动为其创建聚簇索引。JPA 的 ddl-auto:update 机制根据实体类中 @Column(unique=true) 注解，自动为 user_account.username 和 forum_section.name 生成唯一索引。post_like 表的联合唯一约束 uk_post_like_post_user 由 @UniqueConstraint 注解显式定义，约束列为 (post_id, user_id)，MySQL 据此创建联合唯一索引。值得注意的是 forum_post.section_id 和 post_comment.post_id 两个字段：前者在按分区筛选帖子列表时作为查询条件，后者在加载帖子评论列表时作为查询条件，当前数据量下全表扫描尚可接受，数据量增长后应手动添加普通索引以优化查询。

4.4 安全设计

系统的安全机制围绕用户身份认证和请求访问控制两个核心环节构建，涉及认证服务内部的令牌签发与校验、网关层的统一鉴权以及登录失败防护三个部分。

认证服务内部的 JwtService 负责令牌的生成与解析。生成令牌时，该服务将用户 ID、用户名、会话 ID 和令牌类型（access 或 refresh）写入 JWT 的声明字段，使用 HMAC-SHA 算法和不低于 32 字符的密钥对整个令牌进行数字签名，最后设置过期时间并输出 JWT 字符串。解析令牌时，该服务首先验证签名是否被篡改，然后检查令牌类型是否与预期一致，最后从声明字段中提取用户身份信息。访问令牌和刷新令牌共用同一套签发逻辑，仅在类型标识和过期时间上存在差异。

用户登录时，AuthService 首先通过 LoginAttemptService 检查当前账户是否处于锁定状态。如果未锁定，则从数据库查询用户记录并使用 PasswordEncoder 比对密码。校验成功后清除失败计数，生成一个随机的 UUID 作为会话 ID，调用 JwtService 签发访问令牌和刷新令牌，同时通过 AuthSessionService 将会话 ID 与用户 ID 的映射关系写入 Redis 并设置与刷新令牌相同的过期时间。校验失败时，失败计数在 Redis 中原子递增，达到 5 次阈值后账户被锁定 600 秒。登录认证的完整流程如图 4-5 所示。

图 4-5 登录认证流程图

网关层的 JwtRelayFilter 在每个请求到达时执行访问控制判断，其决策流程如图 4-6 所示。过滤器首先判断请求方法是否为 OPTIONS（浏览器跨域预检请求），若是则直接放行。接着检查是否为 GET 请求且路径命中公开读白名单，白名单包含帖子列表（/api/forum/posts）、帖子详情（/api/forum/posts/{id}）、分区列表（/api/forum/sections）、通知首页（/api/notifications/home）、媒体首页（/api/media/home）、媒体推荐（/api/media/recommendations）和搜索（/api/search/posts）共七个路径。随后检查路径是否以 /api/auth/、/api/users/ping 或 /actuator 开头，这些路径同样直接放行。以上三项检查均未命中的请求进入鉴权流程：过滤器从 Authorization 请求头中提取 Bearer 令牌，使用 HMAC-SHA 密钥验证签名，检查令牌声明中的 uid、username、sid 和 typ 四个字段是否完整且 typ 为 access，最后查询 Redis 中以 auth:session: 为前缀加 sessionId 组成的键，确认会话记录存在且存储的 userId 与令牌声明一致。以上任一环节失败均返回 401 状态码和 JSON 格式的错误信息。全部校验通过后，过滤器将用户 ID 和用户名写入 X-User-Id 和 X-Username 请求头并转发至下游服务。

4.5 开发与运行环境

系统的后端开发环境基于 Java 17 和 Maven 构建，各微服务使用 Spring Boot 3.2.5 作为基础框架，微服务治理组件来自 Spring Cloud 2023.0.1 和 Spring Cloud Alibaba 2023.0.1.2。前端开发环境基于 Node.js，PC 端和移动端均使用 Vue 3.5、Vue Router 4.5、Pinia 3 和 Axios 1.9，构建工具为 Vite 5.4，TypeScript 编译器版本为 5.9。PC 端使用 Element Plus 2.13 组件库，移动端使用 Vant 4.9 组件库。

基础设施由 Docker Compose 编排的三个容器组成：MySQL 8.4（字符集 utf8mb4，附带健康检查）、Redis 7.4（开启 AOF 持久化，附带健康检查）和 Nacos 2.3.2（standalone 模式）。三个容器运行在名为 forum-net 的 bridge 网络中，数据通过命名卷持久化。

后端项目的目录结构按 Maven 多模块组织，根 pom.xml 声明了 7 个子模块：common-lib、gateway-service、auth-user-service、forum-service、notification-service、search-service 和 media-service。每个服务模块遵循标准的 Spring Boot 项目布局，源码位于 src/main/java 下，配置文件位于 src/main/resources/application.yml。前端项目的源码位于 src/ 目录下，按 api/、views/、stores/、router/、components/ 和 styles/ 组织。

4.6 认证与会话管理实现

注册流程的实现位于 AuthService 的 register 方法中。该方法首先检查用户名是否已存在，若存在则抛出业务异常。随后创建 UserAccount 实体，使用 BCrypt 算法对密码进行加密后持久化到数据库，最后调用 issueTokens 方法签发令牌对并返回给前端。

登录流程的实现位于 AuthService 的 login 方法中，其完整逻辑已在 4.4 节的流程图中描述。令牌刷新流程位于 refresh 方法中，该方法解析刷新令牌获取用户 ID 和会话 ID，验证 Redis 中的会话记录是否有效，有效则签发新的令牌对并更新会话过期时间。登出流程位于 logout 方法中，该方法直接删除 Redis 中的会话记录，使所有基于该会话签发的令牌立即失效。

前端的令牌管理实现位于 api/http.ts 文件中。请求拦截器在每次发送请求前自动从 localStorage 读取访问令牌并写入 Authorization 请求头。响应拦截器在收到 401 状态码时执行静默刷新逻辑，其完整流程如图 4-7 所示。拦截器首先判断当前请求是否为认证接口（login/register/refresh/logout），如果是则不进行刷新直接报错。接着检查 localStorage 中是否存在刷新令牌，不存在则清除存储并跳转至登录页。如果存在刷新令牌且当前没有其他刷新请求正在进行中，拦截器发起 POST /api/auth/refresh 请求获取新的令牌对，成功后更新 localStorage 并通过 resolveQueue 函数释放等待队列中的所有请求使其携带新令牌重试。如果已有刷新请求正在进行中，当前请求被加入 pendingQueue 等待队列，避免多个并发请求同时触发刷新导致的竞争问题。令牌更新后，http.ts 通过 window.dispatchEvent 发送 auth-session-changed 自定义事件，main.ts 中的事件监听器收到事件后调用 Pinia auth store 的 syncFromStorage 方法同步最新的令牌值到全局状态。

图 4-7 前端 401 静默刷新流程图

4.7 论坛核心业务实现

帖子发布的实现位于 ForumService 的 createPost 方法中，其流程如图 4-8 所示。该方法首先通过 validateUser 校验请求头中携带的用户身份信息（userId 和 username）是否非空，校验不通过则返回 401。随后根据请求体中的 sectionId 调用 loadSection 从数据库加载分区实体，如果分区不存在则返回 404。校验通过后，方法构建 ForumPost 实体，将标题、正文、作者 ID、作者用户名、分区 ID 和分区名称写入实体字段，调用 forumPostRepository.save 持久化到 MySQL。方法上标注的 @CacheEvict(cacheNames = CACHE_POST_LIST, allEntries = true) 注解在持久化完成后自动淘汰帖子列表的全部缓存条目。帖子编辑（updatePost）和删除（deletePost）方法在 validateUser 之后额外调用 assertPostOwner 方法，比较请求头中的用户 ID 与帖子的 authorId 是否一致，不一致则抛出 4033 权限异常。删除帖子时还会级联删除该帖子下的全部评论（postCommentRepository.deleteByPostId）和点赞记录（postLikeRepository.deleteByPostId）。

图 4-8 帖子发布流程图

帖子列表查询使用 Spring Data JPA 的分页机制，通过 PageRequest.of(page, size, Sort.by(DESC, "id")) 构建按 ID 降序排列的分页参数。当请求携带 sectionId 参数时，调用 forumPostRepository.findBySectionId 按分区筛选；否则调用 findAll 返回全部帖子。查询方法上标注了 @Cacheable 注解，缓存键由 sectionId、page 和 size 三个参数拼接生成（格式为 "sectionId:page:size" 或 "all:page:size"），缓存名称为 forum:post:list，过期时间在 RedisCacheConfig 中配置为 2 分钟。帖子详情查询的缓存名称为 forum:post:detail，缓存键为帖子 ID，过期时间为 5 分钟，查询结果包含帖子正文和按创建时间升序排列的全部评论。读操作优先命中缓存，写操作（发布、编辑、删除、评论、点赞）执行后通过 @CacheEvict 或 @Caching 注解淘汰相关缓存，其中发布帖子仅淘汰列表缓存，而编辑、删除、评论和点赞操作同时淘汰列表缓存和对应帖子的详情缓存。缓存的读写策略如图 4-9 所示。

图 4-9 缓存读写策略流程图

评论功能通过 post_comment 表的 parent_comment_id 字段实现二级结构，其流程如图 4-10 所示。addComment 方法首先校验用户身份和帖子存在性，然后调用 resolveParentCommentId 方法处理 parentCommentId 参数：如果该参数为 null，则当前评论为一级评论；如果不为 null，方法会验证父评论是否存在且属于同一帖子，验证通过则当前评论为二级回复，否则返回 404。评论实体构建完成后持久化到 MySQL，同时淘汰帖子列表和对应帖子的详情缓存。评论的编辑（updateComment）方法通过比较 comment.getUserId() 与请求头中的用户 ID 来校验作者权限，不一致则抛出 4033 异常。删除评论（deleteComment）的权限判断稍有不同，由 canDeleteComment 方法实现，帖子作者也有权删除其帖子下的任意评论。

图 4-10 评论发表流程图

点赞功能依赖 post_like 表的联合唯一约束 uk_post_like_post_user。likePost 方法首先通过 postLikeRepository.existsByPostIdAndUserId 查询是否已存在点赞记录，如果存在则直接抛出 4091 异常。随后构建 PostLike 实体并尝试插入数据库，由于存在联合唯一约束，即使在高并发场景下两个请求同时通过了 exists 检查，数据库也会在第二次插入时抛出 DataIntegrityViolationException 异常，业务代码在 catch 块中捕获该异常并返回重复点赞的提示信息，从而实现应用层和数据库层的双重去重保障。插入成功后，帖子的 like_count 字段通过 post.setLikeCount(post.getLikeCount() + 1) 递增并持久化，同时 @Caching 注解淘汰帖子列表和对应帖子的详情缓存。

论坛服务在启动时通过 DemoDataInitializer 初始化种子数据，包括若干预设分区和 8 篇示例帖子，为开发和测试环境提供开箱即用的演示内容。

4.9 辅助服务与运维实现

通知服务在启动时通过 NotificationDataInitializer 初始化若干条 ANNOUNCEMENT 类型的系统公告。home 接口返回最新的公告列表以及当前用户的未读通知计数，当请求头中未携带用户 ID 时未读计数返回零。my 接口返回指定用户的全部通知列表。read 接口将指定通知的 is_read 字段更新为 true。

搜索服务的完整数据流如图 4-15 所示。用户在前端输入关键词后，请求经网关转发至搜索服务的 SearchController（GET /search/posts）。SearchService 内部通过 ForumClient（@FeignClient 注解声明的 OpenFeign 客户端）调用论坛服务的 GET /posts 接口，传入 sectionId、page 和 size 参数获取帖子列表。如果用户未输入关键词（keyword 为空或 blank），搜索服务直接将论坛服务返回的全部帖子转换为 SearchPostResponse 格式返回。如果用户输入了关键词，SearchService 将关键词转为小写后，在内存中遍历每篇帖子，依次对 title、contentPreview（正文前 120 字符）、authorName 和 sectionName 四个字段执行大小写不敏感的 contains 匹配，任一字段命中即保留该帖子。匹配完成后将结果集转换为响应格式返回给前端。搜索服务的默认分页大小为 50（DEFAULT_PAGE_SIZE 常量），当请求中的 size 参数为零或负数时自动使用该默认值。

图 4-15 搜索服务数据流图

媒体服务的 MediaRecommendationService 在构造函数中以硬编码方式初始化音乐和书籍两个推荐列表，每个列表包含若干条 MediaRecommendationItem（含标题、作者/艺术家和描述字段）。home 接口从两个列表中各取前几条作为首页摘要返回，recommendations 接口根据 type 参数（music、book 或 all）返回完整列表。

deploy/docker-compose.yml 文件定义了 MySQL、Redis 和 Nacos 三个容器的镜像版本、端口映射、数据卷和健康检查规则，全部通过环境变量支持自定义端口和密码。deploy/scripts/start-backend-all.sh 脚本按照 common-lib 构建 → auth → forum → notification → media → search → gateway 的顺序依次启动各服务，每个服务启动后轮询其 /actuator/health 端点直至返回 200。deploy/scripts/stop-backend-all.sh 按反向顺序有序停止各服务。deploy/scripts/start-infra-auto.sh 在有 Docker 的环境中自动执行 docker compose up -d，在无 Docker 的环境中尝试使用本机安装的 MySQL 和 Redis。

deploy/scripts/run_system_regression.py 通过标准 HTTP 请求驱动完整的业务流程测试，覆盖注册两个用户、登录、发帖、评论、搜索、通知首页、媒体首页、个人通知、标记已读、未授权发帖、禁止删除他人帖子、并发重复点赞和令牌过期拦截等场景，执行结束后自动生成 Markdown 格式的回归测试报告。deploy/scripts/benchmark_forum_cache.py 对指定接口执行一次冷启动请求和多次热缓存请求，计算最小值、最大值、平均值和 P95 延迟，输出性能提升百分比的 Markdown 报告。回归测试报告输出效果如图 4-16 所示，缓存基准测试报告输出效果如图 4-17 所示。
