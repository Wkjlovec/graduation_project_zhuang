第四章 系统详细设计与实现

## 4.1 系统架构设计

系统整体采用微服务架构，按照部署职责可划分为四个层次：用户接入层、网关层、业务服务层和基础设施层。用户接入层包括 PC 端和移动端两个前端应用，分别运行在 5173 和 5174 端口。网关层由 Spring Cloud Gateway 承担，监听 8080 端口，是所有外部请求进入后端的唯一通道。业务服务层包含认证服务（8081）、论坛服务（8082）、通知服务（8083）、媒体服务（8084）和搜索服务（8085）五个独立进程，以及被各服务依赖的 common-lib 公共库。基础设施层由 Docker Compose 编排的三个容器组成：MySQL 8.4（3306）、Redis 7.4（6379）和 Nacos 2.3.2（8848）。系统部署图如图 4-1 所示。

图 4-1 系统部署图

各层之间的通信关系如下。PC 端和移动端通过 HTTP 协议向网关的 8080 端口发送请求。网关根据配置文件中定义的六条路由规则，按请求路径前缀将请求分发至对应的后端服务。具体路由映射关系为：`/api/auth/**` 和 `/api/users/**` 路由至认证服务，`/api/forum/**` 路由至论坛服务，`/api/notifications/**` 路由至通知服务，`/api/media/**` 路由至媒体服务，`/api/search/**` 路由至搜索服务。各路由均配置了 StripPrefix 过滤器，在转发前去除 URL 中的网关前缀层级，使下游服务接收到的请求路径与其自身的 Controller 映射一致。路由目标地址采用 `lb://服务名` 格式，由 Nacos 注册中心解析服务名到实际的 IP 和端口，实现基于服务发现的负载均衡。网关路由转发流程如图 4-2 所示。

图 4-2 网关路由转发流程图

在业务服务层内部，五个服务和网关在启动时均向 Nacos 注册自身地址和元信息，Nacos 同时作为配置中心为各服务提供运行时参数。服务之间的调用关系较为简单：搜索服务通过 OpenFeign 声明式客户端调用论坛服务的帖子列表接口获取数据，其余服务之间不存在直接调用。当系统运行在无 Nacos 的直连模式下，网关的路由目标地址和搜索服务的 Feign URL 均可通过环境变量指定为固定的 localhost 地址，绕过服务发现机制。

在数据访问方面，认证服务、论坛服务和通知服务共享同一个 MySQL 数据库实例（数据库名 forum_system），各服务通过不同的数据表实现逻辑隔离。认证服务和论坛服务同时访问 Redis，前者用于存储会话状态和登录失败计数，后者用于缓存帖子列表和详情数据。网关也连接 Redis 用于在鉴权过程中查询会话有效性。搜索服务和媒体服务不直接访问任何数据库或缓存。

## 4.2 功能模块设计

系统后端共包含七个 Maven 模块，其中 common-lib 为公共库，不独立运行，其余六个模块各自作为独立的 Spring Boot 应用部署。各模块的职责边界、对外暴露的接口以及相互之间的依赖关系构成了系统的功能模块结构，如图 4-3 所示。

图 4-3 系统功能模块结构图

common-lib 是所有业务服务共同依赖的基础模块，定义了统一的接口响应封装类 ApiResponse 和全局常量类 ServiceConstants。ServiceConstants 中集中管理了 JWT 令牌类型标识、HTTP 请求头名称（Authorization、X-User-Id、X-Username）、Bearer 前缀以及 Redis 键前缀等在多个服务间需要保持一致的常量值，避免各服务各自硬编码导致的不一致风险。

gateway-service 是系统的统一入口，对外暴露 8080 端口。除了 4.1 节所述的路由转发职责外，该模块还承载了全局的 JWT 鉴权逻辑，通过 JwtRelayFilter 拦截所有请求，区分公开接口和受保护接口并完成令牌校验与用户身份透传。网关本身不包含任何业务逻辑。

auth-user-service 负责用户身份管理，对外提供六个接口：注册（POST /auth/register）、登录（POST /auth/login）、令牌刷新（POST /auth/refresh）、登出（POST /auth/logout）、查看个人信息（GET /users/me）以及健康探测（GET /users/ping）。该服务内部包含 JwtService 负责令牌签发与解析、AuthSessionService 负责 Redis 会话管理、LoginAttemptService 负责登录失败计数与锁定，以及 Spring Security 配置的内部过滤器链。

forum-service 是系统的核心业务模块，承载了帖子和评论的全部操作。帖子相关接口包括：列表查询（GET /posts，支持 page、size、sectionId 参数）、发布（POST /posts）、编辑（PUT /posts/{postId}）、删除（DELETE /posts/{postId}）、详情查询（GET /posts/{postId}）和点赞（POST /posts/{postId}/like）。评论相关接口包括：发表评论（POST /posts/{postId}/comments）、编辑评论（PUT /posts/{postId}/comments/{commentId}）和删除评论（DELETE /posts/{postId}/comments/{commentId}）。此外，分区列表通过独立的 SectionController 提供（GET /sections）。该服务内部配置了 Redis 缓存策略和种子数据初始化器。

notification-service 提供三个接口：通知首页摘要（GET /notifications/home，返回最新公告和未读计数）、个人通知列表（GET /notifications/my）和标记已读（POST /notifications/{notificationId}/read）。首页摘要接口支持匿名访问，当请求头中未携带用户 ID 时返回未读数为零。该服务在启动时通过 NotificationDataInitializer 初始化若干条系统公告作为演示数据。

search-service 提供一个搜索接口（GET /search/posts，支持 keyword、sectionId、page、size 参数）。该服务内部通过 OpenFeign 客户端 ForumClient 调用论坛服务的帖子列表接口获取原始数据，然后在内存中按关键词对标题、内容摘要和作者名称进行过滤匹配后返回结果。搜索服务不拥有自己的数据库，所有数据均来源于论坛服务。

media-service 提供两个接口：媒体首页（GET /media/home）和推荐列表（GET /media/recommendations，支持 type 参数筛选音乐或书籍）。推荐数据在 MediaRecommendationService 中以静态列表方式预置于内存，不依赖外部数据库。该服务是系统中最轻量的模块，不访问 MySQL 和 Redis。

前端分为 PC 端和移动端两个独立项目，均基于 Vue 3 构建。两端共享相同的页面模块划分，各包含六个页面组件：登录页（LoginView）、帖子列表页（PostListView）、帖子详情页（PostDetailView）、发帖页（CreatePostView）、通知页（NotificationView）和个人中心页（ProfileView）。PC 端使用 Element Plus 组件库渲染桌面端交互控件，移动端使用 Vant 组件库适配触屏操作。两端在 API 调用层（api/ 目录）、状态管理（stores/auth.ts）和路由配置（router/index.ts）方面保持一致的代码结构和业务逻辑。
