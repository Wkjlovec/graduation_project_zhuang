第四章 系统详细设计与实现

## 4.1 系统架构设计

系统整体采用微服务架构，按照部署职责可划分为四个层次：用户接入层、网关层、业务服务层和基础设施层。用户接入层包括 PC 端和移动端两个前端应用，分别运行在 5173 和 5174 端口。网关层由 Spring Cloud Gateway 承担，监听 8080 端口，是所有外部请求进入后端的唯一通道。业务服务层包含认证服务（8081）、论坛服务（8082）、通知服务（8083）、媒体服务（8084）和搜索服务（8085）五个独立进程，以及被各服务依赖的 common-lib 公共库。基础设施层由 Docker Compose 编排的三个容器组成：MySQL 8.4（3306）、Redis 7.4（6379）和 Nacos 2.3.2（8848）。系统部署图如图 4-1 所示。

图 4-1 系统部署图

各层之间的通信关系如下。PC 端和移动端通过 HTTP 协议向网关的 8080 端口发送请求。网关根据配置文件中定义的六条路由规则，按请求路径前缀将请求分发至对应的后端服务。具体路由映射关系为：`/api/auth/**` 和 `/api/users/**` 路由至认证服务，`/api/forum/**` 路由至论坛服务，`/api/notifications/**` 路由至通知服务，`/api/media/**` 路由至媒体服务，`/api/search/**` 路由至搜索服务。各路由均配置了 StripPrefix 过滤器，在转发前去除 URL 中的网关前缀层级，使下游服务接收到的请求路径与其自身的 Controller 映射一致。路由目标地址采用 `lb://服务名` 格式，由 Nacos 注册中心解析服务名到实际的 IP 和端口，实现基于服务发现的负载均衡。网关路由转发流程如图 4-2 所示。

图 4-2 网关路由转发流程图

在业务服务层内部，五个服务和网关在启动时均向 Nacos 注册自身地址和元信息，Nacos 同时作为配置中心为各服务提供运行时参数。服务之间的调用关系较为简单：搜索服务通过 OpenFeign 声明式客户端调用论坛服务的帖子列表接口获取数据，其余服务之间不存在直接调用。当系统运行在无 Nacos 的直连模式下，网关的路由目标地址和搜索服务的 Feign URL 均可通过环境变量指定为固定的 localhost 地址，绕过服务发现机制。

在数据访问方面，认证服务、论坛服务和通知服务共享同一个 MySQL 数据库实例（数据库名 forum_system），各服务通过不同的数据表实现逻辑隔离。认证服务和论坛服务同时访问 Redis，前者用于存储会话状态和登录失败计数，后者用于缓存帖子列表和详情数据。网关也连接 Redis 用于在鉴权过程中查询会话有效性。搜索服务和媒体服务不直接访问任何数据库或缓存。

## 4.2 功能模块设计

系统后端共包含七个 Maven 模块，其中 common-lib 为公共库，不独立运行，其余六个模块各自作为独立的 Spring Boot 应用部署。各模块的职责边界、对外暴露的接口以及相互之间的依赖关系构成了系统的功能模块结构，如图 4-3 所示。

图 4-3 系统功能模块结构图

common-lib 是所有业务服务共同依赖的基础模块，定义了统一的接口响应封装类 ApiResponse 和全局常量类 ServiceConstants。ServiceConstants 中集中管理了 JWT 令牌类型标识、HTTP 请求头名称（Authorization、X-User-Id、X-Username）、Bearer 前缀以及 Redis 键前缀等在多个服务间需要保持一致的常量值，避免各服务各自硬编码导致的不一致风险。

gateway-service 是系统的统一入口，对外暴露 8080 端口。除了 4.1 节所述的路由转发职责外，该模块还承载了全局的 JWT 鉴权逻辑，通过 JwtRelayFilter 拦截所有请求，区分公开接口和受保护接口并完成令牌校验与用户身份透传。网关本身不包含任何业务逻辑。

auth-user-service 负责用户身份管理，对外提供六个接口：注册（POST /auth/register）、登录（POST /auth/login）、令牌刷新（POST /auth/refresh）、登出（POST /auth/logout）、查看个人信息（GET /users/me）以及健康探测（GET /users/ping）。该服务内部包含 JwtService 负责令牌签发与解析、AuthSessionService 负责 Redis 会话管理、LoginAttemptService 负责登录失败计数与锁定，以及 Spring Security 配置的内部过滤器链。

forum-service 是系统的核心业务模块，承载了帖子和评论的全部操作。帖子相关接口包括：列表查询（GET /posts，支持 page、size、sectionId 参数）、发布（POST /posts）、编辑（PUT /posts/{postId}）、删除（DELETE /posts/{postId}）、详情查询（GET /posts/{postId}）和点赞（POST /posts/{postId}/like）。评论相关接口包括：发表评论（POST /posts/{postId}/comments）、编辑评论（PUT /posts/{postId}/comments/{commentId}）和删除评论（DELETE /posts/{postId}/comments/{commentId}）。此外，分区列表通过独立的 SectionController 提供（GET /sections）。该服务内部配置了 Redis 缓存策略和种子数据初始化器。

notification-service 提供三个接口：通知首页摘要（GET /notifications/home，返回最新公告和未读计数）、个人通知列表（GET /notifications/my）和标记已读（POST /notifications/{notificationId}/read）。首页摘要接口支持匿名访问，当请求头中未携带用户 ID 时返回未读数为零。该服务在启动时通过 NotificationDataInitializer 初始化若干条系统公告作为演示数据。

search-service 提供一个搜索接口（GET /search/posts，支持 keyword、sectionId、page、size 参数）。该服务内部通过 OpenFeign 客户端 ForumClient 调用论坛服务的帖子列表接口获取原始数据，然后在内存中按关键词对标题、内容摘要和作者名称进行过滤匹配后返回结果。搜索服务不拥有自己的数据库，所有数据均来源于论坛服务。

media-service 提供两个接口：媒体首页（GET /media/home）和推荐列表（GET /media/recommendations，支持 type 参数筛选音乐或书籍）。推荐数据在 MediaRecommendationService 中以静态列表方式预置于内存，不依赖外部数据库。该服务是系统中最轻量的模块，不访问 MySQL 和 Redis。

前端分为 PC 端和移动端两个独立项目，均基于 Vue 3 构建。两端共享相同的页面模块划分，各包含六个页面组件：登录页（LoginView）、帖子列表页（PostListView）、帖子详情页（PostDetailView）、发帖页（CreatePostView）、通知页（NotificationView）和个人中心页（ProfileView）。PC 端使用 Element Plus 组件库渲染桌面端交互控件，移动端使用 Vant 组件库适配触屏操作。两端在 API 调用层（api/ 目录）、状态管理（stores/auth.ts）和路由配置（router/index.ts）方面保持一致的代码结构和业务逻辑。

## 4.4 安全设计

系统的安全机制围绕用户身份认证和请求访问控制两个核心环节构建，涉及认证服务内部的令牌签发与校验、网关层的统一鉴权以及登录失败防护三个部分。

认证服务内部的 JwtService 负责令牌的生成与解析。生成令牌时，该服务将用户 ID、用户名、会话 ID 和令牌类型（access 或 refresh）写入 JWT 的声明字段，使用 HMAC-SHA 算法和不低于 32 字符的密钥对整个令牌进行数字签名，最后设置过期时间并输出 JWT 字符串。解析令牌时，该服务首先验证签名是否被篡改，然后检查令牌类型是否与预期一致，最后从声明字段中提取用户身份信息。访问令牌和刷新令牌共用同一套签发逻辑，仅在类型标识和过期时间上存在差异。

用户登录时，AuthService 首先通过 LoginAttemptService 检查当前账户是否处于锁定状态。如果未锁定，则从数据库查询用户记录并使用 PasswordEncoder 比对密码。校验成功后清除失败计数，生成一个随机的 UUID 作为会话 ID，调用 JwtService 签发访问令牌和刷新令牌，同时通过 AuthSessionService 将会话 ID 与用户 ID 的映射关系写入 Redis 并设置与刷新令牌相同的过期时间。校验失败时，失败计数在 Redis 中原子递增，达到 5 次阈值后账户被锁定 600 秒。登录认证的完整流程如图 4-5 所示。

图 4-5 登录认证流程图

网关层的 JwtRelayFilter 在每个请求到达时执行访问控制判断。该过滤器首先检查请求是否属于公开接口白名单（OPTIONS 请求、认证接口、帖子列表和详情的 GET 请求、分区列表、通知首页、媒体接口和搜索接口），属于白名单的请求直接放行。对于受保护接口，过滤器从 Authorization 请求头中提取 Bearer 令牌，使用与认证服务相同的密钥验证签名，检查令牌类型是否为 access、用户 ID 和会话 ID 是否有效，随后查询 Redis 确认该会话未被注销。全部校验通过后，过滤器将用户 ID 和用户名写入 X-User-Id 和 X-Username 请求头并转发至下游服务，下游服务从请求头中读取用户身份即可，无须重复解析令牌。

## 4.5 开发与运行环境

系统的后端开发环境基于 Java 17 和 Maven 构建，各微服务使用 Spring Boot 3.2.5 作为基础框架，微服务治理组件来自 Spring Cloud 2023.0.1 和 Spring Cloud Alibaba 2023.0.1.2。前端开发环境基于 Node.js，PC 端和移动端均使用 Vue 3.5、Vue Router 4.5、Pinia 3 和 Axios 1.9，构建工具为 Vite 5.4，TypeScript 编译器版本为 5.9。PC 端使用 Element Plus 2.13 组件库，移动端使用 Vant 4.9 组件库。

基础设施由 Docker Compose 编排的三个容器组成：MySQL 8.4（字符集 utf8mb4，附带健康检查）、Redis 7.4（开启 AOF 持久化，附带健康检查）和 Nacos 2.3.2（standalone 模式）。三个容器运行在名为 forum-net 的 bridge 网络中，数据通过命名卷持久化。

后端项目的目录结构按 Maven 多模块组织，根 pom.xml 声明了 7 个子模块：common-lib、gateway-service、auth-user-service、forum-service、notification-service、search-service 和 media-service。每个服务模块遵循标准的 Spring Boot 项目布局，源码位于 src/main/java 下，配置文件位于 src/main/resources/application.yml。前端项目的源码位于 src/ 目录下，按 api/、views/、stores/、router/、components/ 和 styles/ 组织。

## 4.6 认证与会话管理实现

注册流程的实现位于 AuthService 的 register 方法中。该方法首先检查用户名是否已存在，若存在则抛出业务异常。随后创建 UserAccount 实体，使用 BCrypt 算法对密码进行加密后持久化到数据库，最后调用 issueTokens 方法签发令牌对并返回给前端。

登录流程的实现位于 AuthService 的 login 方法中，其完整逻辑已在 4.4 节的流程图中描述。令牌刷新流程位于 refresh 方法中，该方法解析刷新令牌获取用户 ID 和会话 ID，验证 Redis 中的会话记录是否有效，有效则签发新的令牌对并更新会话过期时间。登出流程位于 logout 方法中，该方法直接删除 Redis 中的会话记录，使所有基于该会话签发的令牌立即失效。

前端的令牌管理实现位于 api/http.ts 文件中。请求拦截器在每次发送请求前自动从 localStorage 读取访问令牌并写入 Authorization 请求头。响应拦截器在收到 401 状态码时，判断当前是否已有刷新请求在进行中：如果没有，则发起刷新请求获取新令牌并重试原始请求；如果已有刷新请求在进行中，则将当前请求加入等待队列，待刷新完成后统一重试。这种单次刷新加队列等待的机制避免了多个并发请求同时触发刷新导致的竞争问题。令牌更新后，http.ts 通过 dispatchEvent 发送 auth-session-changed 事件通知 Pinia 状态管理同步最新的令牌值。

## 4.7 论坛核心业务实现

帖子发布的实现位于 ForumService 的 createPost 方法中。该方法首先校验请求用户的身份信息非空，然后根据 sectionId 从数据库加载分区实体确认分区存在，接着构建 ForumPost 实体并持久化，最后通过 @CacheEvict 注解淘汰帖子列表缓存。帖子编辑和删除方法中增加了作者校验逻辑，只有当请求头中的用户 ID 与帖子的 authorId 一致时才允许操作，否则抛出 403 权限异常。

帖子列表查询使用 Spring Data JPA 的分页机制，通过 PageRequest 构建分页参数，支持按 sectionId 筛选。查询方法上标注了 @Cacheable 注解，缓存键由查询参数组合生成，缓存过期时间在 RedisCacheConfig 中配置为 2 分钟。帖子详情查询同样使用 @Cacheable 注解，过期时间为 5 分钟。读操作优先命中缓存，写操作执行后通过 @CacheEvict 淘汰相关缓存。缓存的读写策略如图 4-6 所示。

图 4-6 缓存读写策略流程图

评论功能通过 post_comment 表的 parentCommentId 字段实现二级结构。发表一级评论时 parentCommentId 为 null，回复已有评论时 parentCommentId 指向被回复的评论 ID。评论的编辑和删除同样包含作者校验。

点赞功能依赖 post_like 表的联合唯一约束 uk_post_like_post_user。当用户点赞时，系统尝试向 post_like 表插入一条记录，如果该用户已对该帖子点赞过，数据库会抛出 DataIntegrityViolationException 异常，业务代码捕获该异常后返回重复点赞的提示信息。插入成功后，帖子的 like_count 字段原子递增，同时淘汰帖子列表和详情缓存。

论坛服务在启动时通过 DemoDataInitializer 初始化种子数据，包括若干预设分区和 8 篇示例帖子，为开发和测试环境提供开箱即用的演示内容。

## 4.9 辅助服务与运维实现

通知服务在启动时通过 NotificationDataInitializer 初始化若干条 ANNOUNCEMENT 类型的系统公告。home 接口返回最新的公告列表以及当前用户的未读通知计数，当请求头中未携带用户 ID 时未读计数返回零。my 接口返回指定用户的全部通知列表。read 接口将指定通知的 is_read 字段更新为 true。

搜索服务的 SearchService 通过 ForumClient（OpenFeign 客户端）调用论坛服务的 GET /posts 接口获取帖子列表，然后在内存中遍历帖子，对标题、内容前 120 字符（previewContent）和作者名称三个字段进行关键词匹配，返回命中的帖子集合。当搜索结果数量较大时，service 层按照分页参数裁剪后返回。

媒体服务的 MediaRecommendationService 在构造函数中以硬编码方式初始化音乐和书籍两个推荐列表，每个列表包含若干条 MediaRecommendationItem（含标题、作者/艺术家和描述字段）。home 接口从两个列表中各取前几条作为首页摘要返回，recommendations 接口根据 type 参数（music、book 或 all）返回完整列表。

基础设施编排方面，deploy/docker-compose.yml 定义了 MySQL、Redis 和 Nacos 三个服务的容器配置，全部通过环境变量支持自定义端口和密码。deploy/scripts/ 目录下的 start-backend-all.sh 脚本按照 common-lib 构建 → auth → forum → notification → media → search → gateway 的顺序依次启动各服务，每个服务启动后轮询其 /actuator/health 端点直至返回 200。stop-backend-all.sh 按反向顺序有序停止各服务。start-infra-auto.sh 在有 Docker 的环境中自动执行 docker compose up -d，在无 Docker 的环境中尝试使用本机安装的 MySQL 和 Redis。

Python 测试脚本方面，deploy/scripts/run_system_regression.py 通过标准 HTTP 请求驱动完整的业务流程测试，覆盖注册两个用户、登录、发帖、评论、搜索、通知首页、媒体首页、个人通知、标记已读、未授权发帖、禁止删除他人帖子、并发重复点赞和令牌过期拦截等场景，执行结束后自动生成 Markdown 格式的回归测试报告。deploy/scripts/benchmark_forum_cache.py 对指定接口执行一次冷启动请求和多次热缓存请求，计算最小值、最大值、平均值和 P95 延迟，输出性能提升百分比的 Markdown 报告。回归测试报告输出效果如图 4-11 所示，缓存基准测试报告输出效果如图 4-12 所示。
