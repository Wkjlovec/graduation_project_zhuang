# 毕业答辩常见问题与解答

> 课题：基于微服务架构的高校论坛系统设计与实现（PC + 移动端）

---

## 一、架构与技术选型类

### Q1：为什么选择微服务架构而不是单体架构？

**回答要点：**

论文1.1节分析了现有校园论坛的问题：早期BBS单体架构（如水木清华BBS、北大未名BBS），用户规模增长后出现并发瓶颈、扩展困难、迭代缓慢。前后端耦合导致任何局部修改都需要全局重新部署。

微服务架构的优势：
- 5个业务服务+网关独立部署（auth、forum、notification、media、search + gateway），职责清晰
- 各服务通过 Nacos 注册发现，实例增减对上游透明
- 论坛服务可独立扩容应对读请求峰值
- 服务之间通过 HTTP/Feign 通信，边界明确

**补充说明**：微服务也带来了额外复杂度。论文2.1节也讨论了这一点——通信方式选择对性能有影响[18]，断路器和故障转移机制是防止故障蔓延的关键[17]。本项目通过 Nacos 统一注册发现、Gateway 统一入口、Docker Compose 一键启停来控制运维复杂度。

---

### Q2：为什么选择 Spring Cloud Alibaba + Nacos，而不是 Eureka 或 Consul？

**回答要点：**

| 维度 | Nacos | Eureka | Consul |
|------|-------|--------|--------|
| 功能 | 注册发现 + 配置管理 | 仅注册发现 | 注册发现 + KV |
| 维护状态 | 阿里活跃维护 | Netflix已停止维护 | HashiCorp维护 |
| 生态 | Spring Cloud Alibaba 原生 | Spring Cloud Netflix（已进入维护模式） | 需额外适配 |
| 部署复杂度 | 单机standalone即可 | 较简单 | 需要额外组件 |

选择理由：
1. Nacos 同时支持**注册发现和配置管理**，一个组件覆盖两类需求
2. Spring Cloud Alibaba 生态与 Spring Cloud Gateway 无缝集成
3. 社区活跃，文档丰富，问题排查资料充足
4. standalone 模式下部署简单，适合毕设开发环境

---

### Q3：为什么选择 Spring Cloud Gateway 而不是 Nginx 做网关？

**回答要点：**

Spring Cloud Gateway 是应用级网关，Nginx 是基础设施级反向代理，两者定位不同：

1. **服务发现集成**：Gateway 原生支持 `lb://服务名` 路由，新增服务无需手动修改配置；Nginx 需手动维护 upstream 列表
2. **过滤器链**：Gateway 支持自定义 Java 过滤器，本项目在 `JwtRelayFilter` 中实现了 JWT 验证、Redis 会话校验、用户身份透传，这些逻辑用 Nginx + Lua 实现成本更高
3. **与 Spring 生态深度集成**：共享 JWT 密钥配置、Redis 连接等 Spring Boot 基础设施
4. **动态路由**：配合 Nacos 可实现动态路由变更

在生产环境中，通常 Nginx 在前（处理静态资源、SSL、限流），Gateway 在后（处理业务路由、鉴权），两者互补。本项目简化了部署层，直接使用 Gateway 统一入口。

---

### Q4：为什么前端选择 Vue 3 而不是 React？

**回答要点：**

1. **组合式 API（Composition API）**：Vue 3 的 `setup` + `ref`/`reactive` 在状态管理上比 React Hooks 更直观，学习曲线更平缓
2. **双端 UI 框架生态**：Element Plus（PC）和 Vant（移动端）都是 Vue 3 原生适配的成熟组件库，且同为国内团队维护，中文文档完善
3. **模板语法**：Vue 的模板语法对初学者更友好，与传统 HTML 结构一致
4. **TypeScript 支持**：Vue 3 对 TypeScript 的支持已经非常成熟

React 也是优秀的选择，但在本项目中 Vue 3 在双端 UI 库生态和中文社区支持上更有优势。

### Q5：为什么使用 JWT 而不是传统 Session？

**回答要点：**

传统 Session 方案将会话信息存储在服务端内存，在微服务架构下存在问题：
- 多服务间 Session 不共享，需引入 Session 同步或集中存储
- 横向扩展时需要 sticky session 或分布式 Session

JWT 方案优势：
1. **无状态**：令牌自包含用户信息，网关可独立验证，不依赖 auth 服务
2. **跨服务透传**：网关验证后通过 `X-User-Id` / `X-Username` 头传递身份

**但本项目并非纯无状态 JWT**：
- 引入了 Redis 会话存储 (`auth:session:{sid}`)，用于支持**登出即失效**——纯 JWT 方案中令牌签发后无法撤销
- 这是 JWT + Session 的混合方案，兼顾了无状态验证的性能和会话管理的安全性
- 登出时删除 Redis 会话，网关校验 session 不存在 → 拒绝请求

## 二、系统设计与实现类

### Q6：系统中的缓存策略是怎么设计的？如何保证缓存一致性？

**回答要点：**

**缓存架构**：使用 Spring Cache + Redis 作为缓存层

| 缓存名 | TTL | 用途 |
|--------|-----|------|
| `forum:post:list` | 2 分钟 | 帖子列表分页缓存 |
| `forum:post:detail` | 5 分钟 | 帖子详情缓存 |
| 默认 | 10 分钟 | 其他缓存 |

**一致性策略**：
1. **主动失效**：写操作（发帖、编辑、删除、评论、点赞）通过 `@CacheEvict` 主动清除相关缓存
2. **TTL 兜底**：即使主动失效遗漏，短 TTL（2~5 分钟）保证数据最终一致
3. **序列化**：使用 `GenericJackson2JsonRedisSerializer`，支持泛型反序列化，避免类型丢失

**性能效果**：
- 冷请求（首次）：约 22~31ms
- 热请求（缓存命中）：约 4~5ms
- 提升比例：81~85%

**未采用的方案及理由**：
- 未使用缓存预热（业务场景不需要）
- 未使用延迟双删（复杂度过高，短 TTL 已足够）
- 未使用 Caffeine 本地缓存（微服务多实例时本地缓存不一致）

### Q7：JWT 双令牌机制具体是怎么工作的？

**回答要点：**

**双令牌设计**：
- **Access Token**：短期有效（默认 86400 秒 / 24 小时），用于接口鉴权
- **Refresh Token**：长期有效（默认 604800 秒 / 7 天），用于刷新 Access Token

**完整流程**：
1. 登录成功 → auth-service 签发 Access + Refresh Token，Redis 写入 session
2. 前端携带 Access Token → Gateway 验证签名、类型、Redis session
3. Access Token 过期 → 前端收到 401 → 自动调用 `/api/auth/refresh`
4. Refresh Token 有效 → 签发新 Access Token → 前端重试原请求
5. Refresh Token 也过期 → 清除本地存储 → 跳转登录页
6. 登出 → 调用 `/api/auth/logout` → Redis 删除 session → 旧 Token 全部失效

**前端并发刷新处理**：
- 当多个请求同时收到 401 时，只触发一次 refresh 调用
- 其他请求进入 pending queue，refresh 成功后统一重试
- 通过 `isRefreshing` 标志位 + Promise 队列实现

---

### Q8：网关的 JWT 过滤器具体做了什么？

**回答要点：**

`JwtRelayFilter` 是一个全局过滤器（order = -100，最先执行），核心逻辑：

1. **路径白名单判断**：OPTIONS 请求、`/api/auth/**`、`/api/users/ping`、`/actuator/**` 直接放行
2. **匿名可读路径**：GET 请求的帖子列表/详情、分区、首页通知、推荐、搜索等只读接口允许匿名访问
3. **JWT 验证**：
   - 提取 `Authorization: Bearer <token>`
   - 使用 JJWT 验证签名（HMAC-SHA256）
   - 检查 token 类型为 `access`（不接受 refresh token）
4. **Redis 会话校验**：
   - 从 JWT claims 中提取 `sid`（session ID）
   - 查询 Redis `auth:session:{sid}` 是否存在
   - 不存在 → 说明已登出或会话过期 → 401
5. **身份透传**：
   - 从 claims 中提取 `uid`、`username`
   - 写入请求头 `X-User-Id`、`X-Username`
   - 下游服务通过这两个头获取用户身份

---

### Q9：帖子的权限控制是怎么实现的？

**回答要点：**

权限控制分为三层：

1. **网关层（Gateway JwtRelayFilter）**：
   - 区分公开路径（列表、详情）和受保护路径（发帖、评论、点赞）
   - 受保护路径必须携带有效 JWT + 活跃 session

2. **前端路由层（Vue Router beforeEach）**：
   - `/login`、`/posts`、`/posts/:id` 为公共页面
   - 其他页面检查 localStorage 中是否有 token，无则跳转登录

3. **业务服务层（forum-service）**：
   - 发帖：需要 `X-User-Id` 头（网关注入），记录作者
   - 编辑/删除帖子：校验当前用户 ID == 帖子 author_id
   - 删除评论：允许评论作者或帖子作者删除
   - 点赞：数据库唯一约束 `(post_id, user_id)` 防止重复

---

### Q10：搜索功能是如何实现的？为什么没用 Elasticsearch？

**回答要点：**

**当前实现**：
- search-service 通过 OpenFeign 调用 forum-service 获取帖子列表
- 在内存中对标题和内容进行关键词过滤
- 支持分区过滤 + 分页

**未使用 Elasticsearch 的理由**：
1. 本科毕设范围控制，避免过度工程化
2. 当前数据规模（高校论坛）不大，内存过滤可满足需求
3. ES 引入会增加部署复杂度（需要额外的 JVM 进程、索引管理、数据同步）
4. 项目设计原则：「能用单服务落地，不额外拆服务」

**未来优化方向**：
- 数据量增大后可引入 Elasticsearch
- 实现全文搜索、高亮、权重排序
- 使用 Canal 或 MQ 做数据库到 ES 的增量同步

---

### Q11：服务间通信是怎么实现的？

**回答要点：**

本系统服务间通信使用 **HTTP + OpenFeign** 方式：

1. **search-service → forum-service**：通过 `@FeignClient(name="forum-service")` 声明式调用帖子列表接口
2. **负载均衡**：Feign 集成 Spring Cloud LoadBalancer，通过 Nacos 服务发现自动路由
3. **降级策略**：当前版本未实现 Sentinel/Hystrix 熔断降级（毕设范围取舍）

**为什么不用消息队列（MQ）**：
- 当前服务间调用是同步的（搜索需要即时返回结果）
- 通知场景可以用 MQ 解耦，但当前规模下直接调用已足够
- 未来可引入 RocketMQ/Kafka 做异步通知和事件驱动

## 三、测试与质量保障类

### Q12：测试策略是怎么设计的？测试数据可靠吗？

**回答要点：**

论文第五章详细描述了测试方案，采用全链路黑盒测试方法：

**测试层次**：

| 层次 | 方式 | 工具 | 覆盖内容 |
|------|------|------|----------|
| 功能测试 | 自动化脚本 | Python (run_system_regression.py) | 11场景·22接口 |
| 缓存基准 | 冷/热对比 | Python (benchmark_forum_cache.py) | 冷/热请求延迟、P95、提升比 |

**测试数据可靠性**：
1. **可复现**：所有测试脚本仅使用Python标准库，一键执行 `generate-test-reports.sh`
2. **报告落盘**：每次执行生成 Markdown 报告，含时间戳和具体数值
3. **多次验证**：缓存基准默认做 20 次热请求取均值和 P95，减少偶然误差

**异常场景覆盖（场景6-9）**：
- 无 Token 写操作 → 预期 code=4010
- 非作者删帖 → 预期 code=4031
- 并发重复点赞 → 预期返回码集合为 [0, 4091]
- 登出后旧 Token → 预期 code=4010

### Q13：缓存基准测试的数据怎么解读？80% 的提升可信吗？

**回答要点：**

**测试方法**：
1. 对帖子列表接口 `/api/forum/posts?page=0&size=10` 先做 1 次冷请求（清除缓存后首次查库）
2. 再做 30 次连续热请求（命中 Redis 缓存）
3. 计算提升比例：`(冷 - 热均值) / 冷 × 100%`

**实际数据**（两次独立测试）：

| 指标 | 测试1 | 测试2 |
|------|-------|-------|
| 冷请求 | 31.45ms | 22.48ms |
| 热均值 | 4.85ms | 4.14ms |
| P95 | 6.35ms | 4.73ms |
| 提升 | 84.59% | 81.57% |

**可信度分析**：
- 冷请求包含数据库查询 + ORM 映射 + 序列化 + 网络传输
- 热请求省去了数据库查询环节，直接从 Redis 读取序列化结果
- 80%+ 的提升符合「数据库 I/O 是主要延迟瓶颈」的预期
- P95 < 7ms 表明缓存命中后延迟非常稳定

## 四、工程实践与部署类

### Q14：一键启停脚本是怎么设计的？

**回答要点：**

项目提供完整的运维脚本链：

1. **基础设施启动** (`start-infra-auto.sh`)：
   - 有 Docker → 自动 `docker compose up -d` 启动 MySQL + Redis + Nacos
   - 无 Docker → 使用本机 MariaDB/Redis + 提示 Nacos 状态

2. **后端服务启动** (`start-backend-all.sh`)：
   - 先构建 common-lib（`mvn -pl common-lib -am install`）
   - 按依赖顺序启动 6 个服务：auth → forum → notification → media → search → gateway（最后）
   - 每个服务启动后自动 `/actuator/health` 验活
   - 支持三种模式：`full`（需 Nacos）、`direct`（无 Nacos，静态路由）、`auto`（自动判断）

3. **验活检查** (`verify-backend-all.sh`)：
   - 检查所有 actuator 健康端点
   - 验证网关业务路由 `/api/forum/sections`

4. **测试报告** (`generate-test-reports.sh`)：
   - 自动选择 full/direct 模式
   - 生成回归报告 + 缓存基准报告到 `docs/`

---

### Q15：如果某个微服务挂了会怎么样？有容错机制吗？

**回答要点：**

**当前实现**：
- 各服务独立进程，单服务宕机不影响其他服务运行
- 网关对不可用服务返回 503 错误
- 前端有错误提示和重试机制

**未实现的容错手段**（毕设范围取舍）：
- 未引入熔断器（Sentinel/Resilience4j）：当下游服务不可用时没有快速失败
- 未做服务多实例部署：单实例宕机即不可用
- 未配置重试策略（Gateway Retry Filter）

**设计原则**：
> 先保证端到端可运行，再做增强。本项目优先验证架构可行性和功能完整性，高可用是未来方向。

---

### Q16：为什么没有引入 CI/CD？

**回答要点：**

这是一个主动的范围取舍决策：

1. **本科毕设重点**：验证系统设计与实现能力，CI/CD 不是核心评审点
2. **替代方案**：通过本地可复现的脚本闭环（启停 + 验活 + 回归 + 报告）替代自动化流水线
3. **工程完整度**：脚本产物等价于 CI 的构建产物，区别仅在于触发方式（手动 vs 提交触发）
4. **部署环境**：毕设通常在单机演示，不需要多环境 CI/CD 流水线

如果要引入，可以用 GitHub Actions：
- push → 构建 → 单元测试 → 打包 → 部署 → 回归测试 → 报告上传

## 五、安全相关类

### Q17：系统有哪些安全措施？

**回答要点：**

| 安全维度 | 实现方式 |
|----------|----------|
| 认证 | JWT 双令牌 + Redis 会话校验 |
| 密码存储 | BCrypt 单向散列（不可逆） |
| 会话管理 | Redis 集中管理，登出即失效 |
| 防暴力破解 | 登录失败 N 次后锁定（IP + 用户名维度） |
| 权限控制 | 网关路径白名单 + 业务层作者校验 |
| 跨域 | Gateway 全局 CORS 配置（限定开发域名） |
| 接口规范 | 写操作必须携带有效 Token |
| 数据完整性 | 数据库唯一约束（点赞去重、用户名唯一） |

**未覆盖的安全领域**（超出毕设范围）：
- SQL 注入防护：JPA 参数化查询已内置防护
- XSS 防护：前端框架默认转义（Vue 的 `{{ }}` 自动转义 HTML）
- CSRF：API 为无状态 JWT 鉴权，已关闭 CSRF（RESTful API 场景下合理）
- WAF / 限流 / 反爬虫：未引入

---

### Q18：为什么关闭了 CSRF 防护？

**回答要点：**

CSRF（跨站请求伪造）防护主要针对基于 Cookie/Session 的认证方式。攻击者利用浏览器自动携带 Cookie 的特性，诱导用户访问恶意页面发起伪造请求。

本系统使用 **JWT Bearer Token** 认证：
1. Token 存储在 localStorage，不会被浏览器自动携带
2. 每个请求需要手动在 `Authorization` 头中携带 Token
3. 攻击者无法通过伪造页面获取到用户的 JWT

因此，关闭 CSRF 是合理的。Spring Security 官方文档也建议对无状态 RESTful API 关闭 CSRF。

---

### Q19：登录失败锁定机制是怎么实现的？

**回答要点：**

1. Redis Key：`auth:login:fail:{username}:{ip}`
2. 每次登录失败 → 计数器 +1
3. 计数器达到阈值（默认 5 次）→ 锁定（默认 600 秒）
4. 锁定期间拒绝该用户从该 IP 的登录请求
5. 登录成功 → 清除计数器
6. Redis Key 自带 TTL → 自动过期解锁

**安全考量**：
- 按 `username + IP` 双维度锁定，防止同一 IP 批量尝试不同账号
- 不按纯 IP 锁定，避免共享 IP 环境（如校园网）下误伤

## 六、项目特色与创新类

### Q20：你认为这个项目最大的创新点是什么？

**回答要点：**

三个核心创新点：

1. **完整的微服务工程闭环**：
   - 不仅实现了功能，还构建了从部署到测试到报告的完整链路
   - 一键启停 → 健康验活 → 自动化回归 → 缓存基准 → 报告产物
   - 这种「可复现的工程验证体系」在同类毕设中较少见

2. **PC + 移动双端协同**：
   - 两端共享同一套后端 API，统一鉴权逻辑
   - 不同 localStorage 前缀支持同域运行不冲突
   - 401 自动刷新 + 登出同步等体验一致性设计

3. **安全机制的实用性**：
   - JWT + Redis 会话混合方案，兼顾无状态性和登出可控
   - 登录失败锁定、并发点赞去重等都是实际场景需求
   - 不是为了技术展示而引入，每个安全措施都对应具体威胁

## 七、项目管理与开发过程类

### Q22：开发过程中遇到的最大困难是什么？如何解决的？

**回答要点：**

**困难 1：缓存与数据库一致性**
- 问题：写操作后列表缓存可能返回旧数据
- 解决：`@CacheEvict(allEntries=true)` 在写操作时清除整个列表缓存 + 短 TTL 兜底

**困难 2：双端认证状态同步**
- 问题：多个并发请求同时收到 401 时，如何避免重复刷新 Token
- 解决：Axios 拦截器中使用 `isRefreshing` 标志位 + pending Promise 队列，只执行一次 refresh，其他请求等待后重试

**困难 3：微服务启动顺序依赖**
- 问题：网关先于业务服务启动时，Nacos 中无可用实例，路由报错
- 解决：启动脚本中固定顺序（gateway 最后启动），每个服务启动后验活才继续

**困难 4：Gateway 中 Redis 的响应式与阻塞式冲突**
- 问题：Gateway 基于 WebFlux（响应式），不能使用普通 RedisTemplate
- 解决：使用 `spring-boot-starter-data-redis-reactive` 的 ReactiveRedisTemplate

### Q25：如何保证微服务之间的数据一致性？

**回答要点：**

**当前策略**：
- 各服务共享同一个 MySQL 数据库（`forum_system`），通过 JPA 事务保证单服务内一致性
- 服务间不存在跨服务写事务场景（设计时刻意避免）

**如果需要跨服务事务**（未来场景）：
1. **最终一致性**：通过 MQ 消息补偿（如发帖成功后异步通知 notification-service）
2. **Saga 模式**：每个服务有本地事务，失败时执行补偿事务
3. **分布式事务框架**：Seata（与 Spring Cloud Alibaba 集成），但复杂度高

**为什么当前不需要**：
- 论坛场景下，用户操作天然是单服务完成的（发帖在 forum-service，登录在 auth-service）
- search-service 只读不写，不涉及事务
- 通知场景即使丢失也不影响核心功能（非关键路径）

### Q26：你的论文中提到"可复现测试"，这和软件工程中的概念有什么关系？

**回答要点：**

可复现测试对应软件工程中**回归测试（Regression Testing）**和**持续集成（CI）**的核心理念：

1. **确定性输入**：测试脚本注册新用户、发布特定内容、执行固定操作序列，每次运行输入一致
2. **可验证输出**：断言 HTTP 状态码和业务响应码，通过/失败有明确判定标准
3. **环境可重建**：Docker Compose 一键拉起基础设施，启动脚本重建服务
4. **报告可对比**：每次运行生成含时间戳的 Markdown 报告，可纵向对比历史数据

这与 DevOps 中「代码即基础设施」「测试即文档」的思想一致，只是触发方式从 CI 流水线改为手动脚本执行（适应毕设交付场景）。

## 九、数据库设计类

### Q27：为什么所有服务共享一个数据库？微服务不应该每个服务独立数据库吗？

**回答要点：**

理论上微服务推荐 Database-per-Service 模式，但在本项目中做了务实的取舍：

1. **毕设规模**：6 个服务共用一个 MySQL 实例，管理简单，资源占用低
2. **数据关联**：帖子和评论需要关联用户信息，独立数据库需要额外的同步机制
3. **运维成本**：多个数据库实例的备份、迁移、权限管理在单人毕设中不划算

**逻辑隔离**：
- 虽然共享数据库，但各服务只访问自己负责的表
- auth 只操作 `user_account`
- forum 只操作 `forum_post`、`post_comment`、`post_like`、`forum_section`
- notification 只操作 `notification_message`

**如果需要独立**：
- 可以在 Nacos 配置中为每个服务指定不同的数据源
- 使用事件驱动（MQ）代替直接数据关联

---

### Q28：帖子点赞的去重是怎么实现的？

**回答要点：**

**实现方式**：数据库唯一约束 + 业务层异常捕获

1. `post_like` 表有 `(post_id, user_id)` 联合唯一约束
2. 用户点赞时尝试 INSERT
3. 如果违反唯一约束 → 捕获 `DataIntegrityViolationException` → 返回业务错误码 4091（已点赞）
4. 同时更新 `forum_post.like_count` 字段

**为什么不用 Redis 做去重**：
- 数据库唯一约束是最可靠的去重手段，不依赖缓存可用性
- 性能可接受（点赞不是高频操作，不是秒杀场景）
- 减少一致性维护成本

**并发测试验证**：
- 回归脚本模拟两个线程同时对同一帖子点赞
- 预期结果：1 个成功（code=0），1 个业务失败（code=4091）
- 实际测试：通过

## 十、补充高频问题

### Q29：Gateway 的 CORS 是怎么配置的？

网关在 `application.yml` 中配置了全局 CORS：
- 允许的来源：`http://localhost:5173`（PC Vite）、`http://localhost:5174`（移动端 Vite）及 `127.0.0.1` 对应地址
- 允许的方法：GET、POST、PUT、DELETE、OPTIONS
- `allowCredentials: true`（允许携带认证信息）

生产环境需要将来源改为实际域名，或使用 Nginx 统一处理 CORS。
