第三章 系统需求分析

## 3.1 系统概述

本文设计的微服务论坛系统面向高校场景，为在校师生提供帖子发布与浏览、分区讨论、评论互动、信息检索和通知触达等在线交流功能。系统的用户分为两种状态：未登录的游客可以浏览帖子列表、公告摘要、搜索结果和媒体推荐等公开内容；完成注册并登录的用户则可以进一步使用发帖、评论、点赞和查看个人通知等需要身份认证的功能。系统不区分学生、教师等角色，所有注册用户享有相同的操作权限，帖子和评论的编辑删除权限通过判断当前用户是否为内容作者来控制。分区和公告等基础数据通过系统启动时的种子数据初始化机制完成配置。

系统采用前后端分离的 B/S 架构，用户通过浏览器访问 PC 端或移动端页面，所有请求经由统一网关路由至后端各微服务处理。后端按业务职责拆分为认证服务、论坛服务、通知服务、搜索服务和媒体服务五个独立模块，各模块通过 Nacos 注册中心进行服务发现，彼此之间以 RESTful 接口通信，数据持久化由 MySQL 完成，热点数据缓存和会话状态由 Redis 管理。系统的整体业务流程为：用户注册并登录后获取访问令牌，携带令牌即可访问论坛的全部功能；网关层根据请求路径和方法判断是否需要认证，GET 类的公开浏览接口允许匿名访问，而发帖、评论、点赞等写操作接口则要求携带有效令牌。系统总体架构如图 3-1 所示。

## 3.2 功能性需求分析

根据系统概述中对用户状态和业务流程的描述，本节按照用户与系统交互的自然顺序，依次分析认证、论坛核心业务、辅助服务和网关路由四个方面的功能性需求。

用户认证是系统所有受保护功能的前置条件。游客访问注册接口提交用户名、昵称和密码后成为注册用户。注册用户通过登录接口提交用户名和密码，系统验证凭据后签发访问令牌和刷新令牌，二者均以 JWT 格式返回。访问令牌用于后续请求的身份校验，有效期较短；刷新令牌用于在访问令牌过期后获取新的令牌对，有效期较长。用户主动登出时，系统在 Redis 中删除对应的会话记录，使已签发的令牌立即失效。为防范暴力破解，系统对同一账户的连续登录失败次数进行计数，达到阈值后临时锁定账户。用户认证模块的用例如图 3-2 所示。

（图 3-2 用户认证模块用例图）

论坛核心业务是系统的主体功能，涵盖帖子管理、评论互动和内容浏览三个方面。在帖子管理方面，注册用户可以在指定分区下发布帖子，填写标题和正文内容；帖子发布后，作者可以对自己的帖子进行编辑或删除，其他用户无此权限。在评论互动方面，注册用户可以对任意帖子发表评论，也可以对已有评论进行回复，形成二级评论结构；评论的编辑和删除同样仅限于评论作者本人。注册用户还可以对帖子进行点赞操作，系统对同一用户对同一帖子的重复点赞进行去重处理。在内容浏览方面，游客和注册用户均可浏览帖子列表，支持按分区筛选和分页查看；点击帖子可查看详情页，包括帖子正文、所属分区、评论列表和点赞数量。系统通过种子数据在启动时初始化若干预设分区，所有用户均可按分区浏览帖子。论坛核心业务模块的用例如图 3-3 所示。

（图 3-3 论坛核心业务模块用例图）

通知、搜索和媒体推荐三个辅助服务为论坛主流程提供补充功能。通知服务管理系统公告和面向特定用户的消息，前端首页展示通知摘要（包含最新公告和当前用户的未读消息计数），注册用户可查看个人通知列表并将指定通知标记为已读。搜索服务接收用户输入的关键词，通过服务间调用获取帖子列表，在内存中按标题、内容摘要和作者名称进行过滤匹配，返回符合条件的帖子集合，支持分区和分页参数。媒体服务在论坛首页提供音乐和书籍推荐内容，数据以静态方式存储在服务内存中，供前端页面展示。辅助服务模块的用例如图 3-4 所示。

（图 3-4 辅助服务模块用例图）

网关作为系统的统一入口，负责请求路由和访问控制。所有来自前端的 HTTP 请求首先到达网关，网关根据预设的路由规则将请求转发至对应的后端服务。在访问控制方面，网关将接口划分为公开接口和受保护接口两类：认证相关接口、帖子列表和详情的 GET 请求、分区列表、通知首页摘要、媒体首页和推荐内容、搜索接口等均允许匿名访问；发帖、编辑、删除、评论、点赞、查看个人通知和标记已读等写操作接口则要求请求头中携带有效的访问令牌。网关在转发受保护请求前完成令牌签名校验和 Redis 会话状态检查，校验通过后将用户 ID 和用户名通过请求头透传至下游服务。网关同时配置了跨域资源共享规则，允许 PC 端和移动端从各自的开发端口访问后端接口。
