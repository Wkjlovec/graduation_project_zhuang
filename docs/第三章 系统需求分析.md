第三章 系统需求分析

## 3.1 系统概述

本文设计的微服务论坛系统面向高校场景，为在校师生提供帖子发布与浏览、分区讨论、评论互动、信息检索和通知触达等在线交流功能。系统的用户分为两种状态：未登录的游客可以浏览帖子列表、公告摘要、搜索结果和媒体推荐等公开内容；完成注册并登录的用户则可以进一步使用发帖、评论、点赞和查看个人通知等需要身份认证的功能。系统不区分学生、教师等角色，所有注册用户享有相同的操作权限，帖子和评论的编辑删除权限通过判断当前用户是否为内容作者来控制。分区和公告等基础数据通过系统启动时的种子数据初始化机制完成配置。

系统采用前后端分离的 B/S 架构，用户通过浏览器访问 PC 端或移动端页面，所有请求经由统一网关路由至后端各微服务处理。后端按业务职责拆分为认证服务、论坛服务、通知服务、搜索服务和媒体服务五个独立模块，各模块通过 Nacos 注册中心进行服务发现，彼此之间以 RESTful 接口通信，数据持久化由 MySQL 完成，热点数据缓存和会话状态由 Redis 管理。系统的整体业务流程为：用户注册并登录后获取访问令牌，携带令牌即可访问论坛的全部功能；网关层根据请求路径和方法判断是否需要认证，GET 类的公开浏览接口允许匿名访问，而发帖、评论、点赞等写操作接口则要求携带有效令牌。系统总体架构如图 3-1 所示。

## 3.2 功能性需求分析

（1）用户认证

用户认证是系统所有受保护功能的前置条件。游客访问注册接口提交用户名、昵称和密码后成为注册用户。注册用户通过登录接口提交用户名和密码，系统验证凭据后签发访问令牌和刷新令牌，二者均以 JWT 格式返回。访问令牌用于后续请求的身份校验，有效期较短；刷新令牌用于在访问令牌过期后获取新的令牌对，有效期较长。用户主动登出时，系统在 Redis 中删除对应的会话记录，使已签发的令牌立即失效。登录操作包含（include）凭据校验环节，校验失败时扩展（extend）触发失败计数逻辑，连续失败达到阈值后临时锁定账户。用户认证模块的用例如图 3-2 所示。

（图 3-2 用户认证模块用例图）

（2）帖子管理

帖子管理是论坛系统的核心功能，用例按参与者分为两组。游客可浏览帖子列表和查看帖子详情，列表浏览包含（include）分页查询，并可扩展（extend）按分区筛选；点击帖子进入详情页，展示帖子正文、所属分区、评论列表和点赞数量。注册用户在拥有游客全部浏览能力的基础上，还可发布帖子，发布操作包含（include）身份认证校验；帖子发布后，编辑和删除操作作为发布的扩展（extend），仅在当前用户为帖子作者时可用。帖子管理模块的用例如图 3-3 所示。

（图 3-3 帖子管理模块用例图）

（3）评论互动

评论功能支持二级结构，用例同样按参与者分为两组。游客可在帖子详情页浏览评论列表，但不能发表或操作评论。注册用户可对帖子发表一级评论，也可对已有评论进行回复形成二级评论，发表评论和回复评论均包含（include）身份认证校验；评论发表后，编辑和删除操作作为扩展（extend），仅在当前用户为评论作者时可用。评论互动模块的用例如图 3-4 所示。

（图 3-4 评论互动模块用例图）

（4）点赞

注册用户可对帖子执行点赞操作，点赞包含（include）身份认证校验。系统对同一用户对同一帖子的重复点赞进行去重处理，确保每个用户对每篇帖子最多产生一次有效点赞。点赞数量在帖子详情页面向所有用户展示。

（5）通知服务

通知服务管理系统公告和面向特定用户的消息。前端首页展示通知摘要，包含最新公告和当前用户的未读消息计数，该接口允许匿名访问。注册用户可查看个人通知列表，查看操作包含（include）身份认证校验；用户可将指定通知标记为已读，标记操作作为查看的扩展（extend）。通知服务模块的用例如图 3-5 所示。

（图 3-5 通知服务模块用例图）

（6）搜索服务

搜索服务为用户提供了在论坛海量帖子中快速定位目标内容的能力。当用户在搜索框中输入关键词并提交后，搜索服务通过 OpenFeign 调用论坛服务获取帖子列表，随后在内存中按标题、内容摘要和作者名称三个维度对帖子进行过滤匹配，将命中的帖子集合返回给前端展示。搜索结果支持按分区进一步缩小范围，也支持分页参数控制单次返回的数据量，便于用户在大量结果中逐页浏览。搜索接口作为公开接口，游客和注册用户均可匿名访问，无须携带访问令牌。搜索服务模块的用例如图 3-6 所示。

（图 3-6 搜索服务模块用例图）

（7）媒体推荐服务

媒体推荐服务为论坛首页提供了学习之外的轻量化内容补充，目前包含音乐推荐和书籍推荐两个板块。推荐数据以静态方式预置在服务内存中，系统启动后即可向前端提供数据，不依赖外部数据库或第三方接口。前端首页通过媒体首页接口获取推荐内容的概要信息，用户也可通过推荐列表接口按类别查看完整的音乐或书籍列表。两个接口均为公开接口，游客和注册用户均可匿名访问。媒体推荐服务模块的用例如图 3-7 所示。

用例名：媒体推荐浏览。参与者：游客和注册用户。基本事件流：用户访问论坛首页，系统自动加载媒体推荐摘要。用户点击推荐板块可查看完整推荐列表，支持按音乐或书籍类别筛选。浏览推荐列表可扩展（extend）按类别筛选。

（图 3-7 媒体推荐服务模块用例图）

（8）网关与访问控制

网关是系统面向外部的唯一入口，承担着请求路由和安全屏障的双重职责。所有来自 PC 端和移动端的 HTTP 请求首先到达网关，网关根据预设的路由规则将请求转发至对应的后端服务。在访问控制方面，网关维护了一份公开接口白名单，包括认证相关接口、帖子列表和详情的 GET 请求、分区列表、通知首页摘要、媒体首页和推荐内容以及搜索接口，这些接口允许匿名访问。白名单之外的所有接口均为受保护接口，要求请求头中携带有效的访问令牌。网关在转发受保护请求前依次执行两项校验：首先验证 JWT 令牌的数字签名和声明字段，确认令牌未被篡改且类型正确；随后查询 Redis 中的会话记录，确认该会话未被主动注销。两项校验均通过后，网关将用户 ID 和用户名写入请求头透传至下游服务，使下游服务无须重复解析令牌。网关同时配置了跨域资源共享规则，允许 PC 端（端口 5173）和移动端（端口 5174）从各自的开发地址访问后端接口。网关与访问控制模块的用例如图 3-8 所示。

用例名：请求路由与鉴权。参与者：游客和注册用户。基本事件流：前端发起 HTTP 请求，网关匹配路由规则转发至目标服务。对受保护接口，网关执行令牌校验，校验包含（include）JWT 签名验证和 Redis 会话检查两个环节，校验通过后透传用户身份信息。对公开接口，网关直接转发请求，不执行认证逻辑。

## 3.3 非功能性需求分析

（1）性能需求

论坛系统的帖子列表和帖子详情是用户访问最频繁的两个页面，如果每次请求都穿透到数据库查询，在用户并发增长时数据库压力将迅速上升，直接影响页面加载速度。为此，系统对帖子列表设置了 2 分钟过期的 Redis 缓存，对帖子详情设置了 5 分钟过期的缓存，其余缓存项统一采用 10 分钟的默认过期时间。缓存命中时接口应在毫秒级完成响应，缓存未命中时响应时间应控制在 500 毫秒以内。系统配套了缓存基准测试脚本，能够在冷启动和热缓存两种条件下分别采集接口响应时间并计算改善幅度，作为性能达标与否的量化判据。

（2）安全需求

系统不使用传统的 Cookie 机制保存用户会话，而是采用 JWT 令牌方案，降低被跨站请求伪造攻击劫持的风险。访问令牌有效期为 86400 秒（24 小时），刷新令牌有效期为 604800 秒（7 天），二者均通过不低于 32 字符的密钥以 HMAC-SHA 算法签名，防止令牌被伪造或篡改。为防范暴力破解，同一账户连续登录失败 5 次后将被锁定 600 秒（10 分钟），失败计数和锁定状态存储在 Redis 中，利用原子操作保证并发条件下计数的准确性。网关在转发受保护请求前依次校验令牌签名和 Redis 会话状态，双重验证通过后方将请求放行至下游服务。跨域访问方面，网关仅允许 PC 端（localhost:5173）和移动端（localhost:5174）两个已注册来源发起请求，其余来源将被拒绝。

（3）可扩展性需求

系统在设计时应充分考虑未来业务规模增长和功能模块新增的可能性。后端六个服务各自运行在独立进程中，通过 Nacos 注册中心完成服务发现，新增业务模块只需开发独立的 Spring Boot 应用并注册至 Nacos，再由网关配置一条新的路由规则即可接入，无须修改已有服务的任何代码。各服务的数据库地址、Redis 地址、Nacos 地址、JWT 密钥和监听端口等参数全部通过环境变量注入，在开发、测试和部署三个阶段可以使用不同的配置值而不改动源码。运维脚本还支持在有 Nacos 的完整模式和无 Nacos 的直连模式之间自动切换，使系统在网络条件受限的环境中也能正常运行。

（4）可维护性需求

可维护性直接关系到系统上线后的运营成本和问题定位效率。基础设施方面，MySQL 8.4、Redis 7.4 和 Nacos 2.3.2 三个组件通过 Docker Compose 编排文件统一管理，镜像版本、端口映射、数据卷和健康检查规则在文件中集中声明，一条命令即可在全新环境中拉起全部依赖服务，避免了手工安装配置带来的人为差错。后端服务方面，Bash 脚本按照 common-lib → auth → forum → notification → media → search → gateway 的顺序依次启动各服务并逐一探测健康端点，停止时按照反向顺序有序关闭，确保网关最先关闭以阻止新请求进入。测试方面，Python 回归测试脚本覆盖从注册、登录、发帖、评论、搜索到通知的完整业务链路，以及权限校验、并发点赞去重和令牌过期拦截等异常路径，执行结束后自动输出 Markdown 格式的测试报告，方便快速判断本次变更是否引入了回归问题。

（5）兼容性需求

用户界面应当风格统一、布局清晰，使用户在不同设备上均能快速理解页面元素。PC 端基于 Element Plus 组件库构建，适配 Chrome、Firefox 和 Edge 等主流桌面浏览器，开发端口为 5173。移动端基于 Vant 组件库构建，针对触屏操作和小屏幕布局进行了专门适配，开发端口为 5174。两端通过同一网关的 8080 端口访问后端服务，在用户操作出现异常时应提供明确的错误提示信息而非空白页面或未捕获的异常堆栈。后端服务基于 Java 17 和 Spring Boot 3 构建，可在 Linux 和 macOS 环境下运行；基础设施通过 Docker 容器化封装，消除了对宿主机操作系统版本和软件安装路径的依赖。
